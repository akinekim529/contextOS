# 2.7 Observability Layer

> **Position in the stack.** ContextOS is replay-grade middleware between applications and any LLM backend. Observability is not a bolt-on dashboard concern here — it is a **first-class subsystem** whose data structures are the *same* structures the [Context Replay Debugger](#2-7-7-replay-the-flagship-contract) consumes. Every span we emit is a candidate input to byte-exact replay; every decision record is a node in the causal graph that answers *"why this answer?"*. The observability layer therefore has two non-negotiable obligations that pull in opposite directions, and this section is largely about how we reconcile them:
>
> 1. **Cheap, lossy, ambient telemetry** for live operations (latency SLOs, throughput, error rates) — must never add meaningful latency to the hot path and must be sampled aggressively.
> 2. **Exact, lossless, durable provenance** for replay and billing — must survive crashes, must be cryptographically attributable to a tenant, and must be *complete* for any request we promise to replay.
>
> The central design move (resolution **C12**) is to refuse to put these on the same write path. **Traces fail open and are sampled; cost records fail closed and are durable.** Everything below follows from that split.

---

## 2.7.1 Design tenets and rejected postures

| Tenet | What we do | Rejected alternative | Why it fails for ContextOS |
|---|---|---|---|
| Two-class telemetry | Best-effort traces (fail-open, sampled) vs. billing-grade cost records (fail-closed, durable outbox) — **C12** | One unified "events" pipeline at the same durability | Either you slow the hot path to billing-grade durability for *everything* (kills the < 250 ms p95 overhead budget) or you make billing lossy (revenue leak + audit failure). No single durability class is correct for both. |
| Replay is structural, not log-scraping | One canonical `ReplayResult` schema (**C7**); deterministic stages are *reconstructed from recorded decision records*, not parsed from log lines | Reconstruct context decisions by grepping structured logs | Logs are sampled and lossy by tenet 1. Replay needs 100% completeness for promised requests. Log-derived replay is non-deterministic and silently wrong. |
| OTel as wire format, not as truth store | Emit OpenTelemetry spans for interop; the **authoritative** provenance lives in the replay bundle (content-addressed, per-tenant-encrypted) | Treat the OTel collector / vendor backend as system of record | Vendor backends drop spans under load, apply their own sampling, and are not per-tenant-encrypted. You cannot crypto-shred (**C11**) a row in a third-party SaaS. |
| Tenant is a partition, not a tag | `tenant_id` is a non-null partition key on every span, decision record, and cost row; RLS-enforced | `tenant_id` as a searchable attribute only | A tag is a leak waiting to happen. Cross-tenant leakage target is **0**; observability data is tenant data. |
| Cost is a span attribute *and* a ledger row | Inference/embedding/compression cost is attributed per span **and** written to the fail-closed cost outbox feeding the budget ledger | Cost lives only in spans | Spans are sampled. A sampled-out expensive request would vanish from billing. Cost must escape the sampling decision. |
| Spans READ agent traces, never drive them | Agent-trace spans are **READ-ONLY correlation** (scope-boundary invariant) | Let the tracer re-execute or reschedule a step to "fill a gap" | That makes ContextOS an agent framework. Out of scope, hard line. |

## 2.7.2 Span taxonomy — one span per pipeline stage

The pipeline ordering invariant is the span tree. We emit **exactly one span per pipeline stage**, parented to a single root request span, in this fixed order:

```
auth/tenant → cache lookup → retrieve candidates → ACL/redaction
            → compression → assembly/packing → routing
            → adapter → stream → async write-back
```

Compression **always** appears after ACL/redaction in the tree; a trace where `ctxos.compression` precedes `ctxos.acl_redaction` is a CI-failing invariant violation, asserted by the trace-shape linter in §2.7.9.

### Span names and the stage they own

| Span name (`ctxos.*`) | Pipeline stage | p95 budget (Section 9 table) | Emits decision record? |
|---|---|---|---|
| `ctxos.request` (root) | whole request | < 250 ms control overhead | yes (top-level verdict) |
| `ctxos.edge.auth` | auth/tenant resolve + `SET LOCAL` RLS | 5 ms | no (audit log instead) |
| `ctxos.cache.lookup` | exact-hash + semantic-ANN cache | 10 ms | **yes** (cache verdict) |
| `ctxos.retrieve` | embed-reuse + pgvector ANN \|\| BM25 + RRF + rescore | 40 ms (≤ 100 ms retrieval SLO) | **yes** (candidate set + raw scores) |
| `ctxos.acl_redaction` | namespace hard-filter + ACL + redaction | (within retrieve/assembly budget) | **yes** (what was dropped + why) |
| `ctxos.compression` | NLI-guarded compression | (within assembly budget) | **yes** (blocks compressed + retention) |
| `ctxos.assembly` | score ≤ 512 + MMR + knapsack + edge-place | **50 ms** (the assembly SLO) | **yes** (final ranking + packing) |
| `ctxos.routing` | difficulty + utility + breaker | 5 ms | **yes** (model chosen + why) |
| `ctxos.adapter` | dispatch + first-token handshake (ContextOS side) | 8 ms | partial (backend request fingerprint) |
| `ctxos.stream` | token relay to client | n/a (duration = TTFT→last token) | no (terminal-event record) |
| `ctxos.writeback.enqueue` | async write-back enqueue | 2 ms | no (enqueue ack only) |

Child spans are permitted *within* a stage span but never restructure the stage tree. For example `ctxos.retrieve` parents `ctxos.retrieve.embed`, `ctxos.retrieve.ann`, `ctxos.retrieve.bm25`, `ctxos.retrieve.rrf` — useful for the latency breakdown (§2.7.6) but invisible to the stage-ordering invariant.

> **Rejected: span-per-function auto-instrumentation.** OpenTelemetry's auto-instrumentors will happily emit a span per ORM call and per HTTP hop. We reject that as the *primary* taxonomy: it produces thousands of spans of no semantic value, blows the per-request byte budget for the replay bundle, and buries the seven decisions that actually matter. We hand-author the stage spans; auto-instrumentation, where enabled, lands as *child* spans under the owning stage and is dropped from the replay manifest (it is non-deterministic I/O, not a ContextOS decision).

### Span attributes (the stable subset)

Every stage span carries this attribute floor (full schema deferred to **Data Models, Section 4** — `04-data-models.md`; reproduced here for the load-bearing fields):

```
# OTel resource attributes (set once per process)
service.name              = "contextos-gateway"
service.version           = <semver>
ctxos.node_id             = <ULID>      # gateway pod identity

# Span attributes (per stage span)
ctxos.tenant_id           = <ULID>      # NON-NULL partition key, always present
ctxos.request_id          = <ULID>      # == trace_id surrogate, ULID for time-sortability
ctxos.namespace           = <project/agent/user>   # within-tenant hard filter (C2)
ctxos.stage               = <enum: edge|cache|retrieve|acl|compress|assembly|route|adapter|stream|writeback>
ctxos.p95_budget_ms       = <int>       # the budget this stage is held to
ctxos.duration_ms         = <float>     # measured
ctxos.budget_breach       = <bool>      # duration_ms > p95_budget_ms
ctxos.sampling_decision   = <enum: keep|drop|force_keep>
ctxos.cost_usd            = <decimal>   # 0 for stages with no model/embed cost
ctxos.decision_record_id  = <ULID|null> # FK into decision_records, null if stage emits none
```

`ctxos.tenant_id` being non-null is enforced by a span processor that **drops to a dead-letter queue** any span missing it — a span without a tenant cannot be safely stored under RLS, so we refuse it rather than leak it into a shared bucket.

## 2.7.3 "Why this answer?" decision records

A trace tells you *how long* things took. A **decision record** tells you *why ContextOS made the choice it made*. These are the rows that answer the support ticket "why did the model get *that* context and not the better chunk I expected?" — and they are the deterministic inputs to replay.

Decision records are written for the five stages that make a *choice*: cache, retrieve+ACL, compression, assembly, and routing. They are **durable and complete for any request flagged replay-eligible** (see §2.7.5), independent of trace sampling. They live in Postgres, partitioned by `tenant_id`, under FORCE RLS.

### 2.7.3.1 Cache verdict record

```json
{
  "record_id": "01J9...",                 // ULID
  "tenant_id": "01J8...",
  "request_id": "01J9...",
  "stage": "cache",
  "ts": "2026-06-28T14:03:11.412Z",       // RFC-3339 UTC
  "exact": {
    "fingerprint": "blake3:9f2c...",       // coarse signature, see below
    "hit": false,
    "lookup_ms": 0.6                        // < 1 ms p99 (Redis exact tier)
  },
  "semantic": {
    "query_embedding_bucket": "lsh:0x3a91",
    "model_id": "gpt-4o-mini-2024",
    "system_prompt_version": "sp_v17",
    "stable_fact_set_version": "facts_v42",
    "candidate_similarity": 0.83,
    "threshold": 0.92,
    "hit": false,
    "lookup_ms": 11.4                       // 8–15 ms p95 (incl. embed on miss)
  },
  "non_cacheable_reason": null,             // e.g. "memory_private_grounded"
  "verdict": "MISS"                          // HIT_EXACT | HIT_SEMANTIC | MISS | NON_CACHEABLE
}
```

The **coarse fingerprint** (resolution **C6**) is exactly:

```
fingerprint = blake3(
    normalized_query_embedding_bucket ||   # LSH bucket, NOT raw vector
    model_id ||
    system_prompt_version ||
    stable_fact_set_version
)
```

Responses grounded in **memory-private** data are flagged `non_cacheable_reason = "memory_private_grounded"` and the record's `verdict = NON_CACHEABLE`; we record *why* so the cache-correctness eval harness (a roadmap item) can audit false-negatives. The sub-1 ms claim is recorded only on the `exact` block; the `semantic` block honestly carries 8–15 ms.

> **Rejected: hashing the raw query string.** A raw-string key gives a near-0% hit ratio on paraphrases and is brittle to whitespace. We reject it in favor of the embedding-bucket coarse signature, which is what lets us honestly project the **25–45% hit ratio** rather than a fantasy number.

### 2.7.3.2 Retrieval + ACL decision record

This record encodes resolution **C1**: the Memory Engine returns candidates with **raw per-modality scores only**; the Context Assembler is the sole final-ranking authority. The retrieval record therefore stores *raw* scores; final weights live in the assembly record.

```json
{
  "record_id": "01J9...",
  "stage": "retrieve",
  "candidate_count_pre_acl": 480,
  "candidate_count_post_acl": 461,         // 19 dropped by ACL/namespace/redaction
  "hard_cap": 512,                          // scope invariant: never exceeds
  "fusion": "RRF",                          // reciprocal rank fusion
  "candidates": [
    {
      "doc_id": "01J7...",
      "namespace": "proj_alpha",
      "raw_scores": {                        // RAW per-modality, C1
        "ann_cosine": 0.71,
        "bm25": 14.2,
        "recency_decay": 0.88                // orthogonal to lost-in-the-middle (C1)
      },
      "rrf_rank": 3,
      "acl_verdict": "ALLOW"
    }
  ],
  "acl_drops": [
    { "doc_id": "01J6...", "reason": "namespace_mismatch", "policy": "C2_hard_filter" },
    { "doc_id": "01J5...", "reason": "redacted_pii_class_ssn" }
  ],
  "recall_mode": "full",                    // "bm25_only" if embed failed (≤12% recall loss)
  "retrieval_ms": 39.7                       // < 100 ms SLO, ~40 ms hot path
}
```

Namespace is a **hard filter, fail-closed** (resolution **C2**): a candidate with a missing or ambiguous namespace is dropped with `reason: "namespace_ambiguous_deny"`. The record proves the deny happened — auditors do not take fail-closed on faith.

### 2.7.3.3 Compression decision record

Per the invariant, compression runs **after** ACL/redaction — so by construction the compressor never sees data the ACL dropped. The record proves the ordering and the fact-retention guard.

```json
{
  "stage": "compress",
  "blocks": [
    {
      "block_id": "01J4...",
      "tokens_in": 1840,
      "tokens_out": 612,
      "ratio": 3.01,                         // within 2–4x target
      "nli_fact_retention": 0.991,           // ≥ 0.98 guard
      "nli_guard_verdict": "PASS"
    }
  ],
  "blocks_skipped_short": 4,                  // below min-length, passed through verbatim
  "total_prompt_token_reduction_pct": 21.0,  // within 10–25% band
  "ran_after_acl": true                       // ordering invariant assertion
}
```

A block failing the NLI guard (`nli_fact_retention < 0.98`) is recorded with `nli_guard_verdict: "REJECT_FALLBACK_VERBATIM"` and the *uncompressed* block is packed instead. We never trade fact retention for tokens silently.

### 2.7.3.4 Assembly (final-ranking) decision record

This is the heart of "why this answer." It is the **sole** final-ranking + budget-packing authority (**C1**), uses **one** weight vocabulary, and packs against the tokenizer of the **already-chosen-or-estimated** model (**C3**).

```json
{
  "stage": "assembly",
  "weight_vocabulary_version": "wv_v9",
  "final_weights": {                          // ONE vocabulary, C1
    "ann_cosine": 0.45,
    "bm25": 0.25,
    "recency_decay": 0.10,
    "mmr_lambda": 0.20
  },
  "tokenizer_basis": {
    "source": "post_route",                   // "post_route" | "conservative_estimate" (C3)
    "model_id": "gpt-4o-mini-2024",
    "tokenizer": "o200k_base",
    "hard_reserve_tokens": 1024,
    "estimate_margin_pct": null               // populated only if source == conservative_estimate
  },
  "packing": {
    "budget_tokens": 16000,
    "packed_tokens": 14976,
    "knapsack_algorithm": "greedy_density_then_swap",
    "items_packed": 22,
    "items_dropped_budget": 7,
    "edge_placement": "lost_in_the_middle_aware"  // high-value items at head/tail
  },
  "selected": [
    { "doc_id": "01J7...", "final_score": 0.812, "position": 0, "tokens": 410 },
    { "doc_id": "01J3...", "final_score": 0.770, "position": 21, "tokens": 388 }
  ],
  "repack_event": null,                        // populated if C3 post-route re-validation re-packed
  "assembly_ms": 47.9                           // < 50 ms p95
}
```

If routing could not be known before packing and we packed against a conservative estimate (**C3**), post-route re-validation either confirms the pack or triggers a `repack_event` — and if even the re-pack overflows the real tokenizer's hard reserve, the request **fails closed with HTTP 413**, recorded as `repack_event.outcome: "fail_closed_413"`. The record shows the whole story: estimate → route → re-validate → outcome.

### 2.7.3.5 Routing decision record

Encodes **C9** (fail-closed hard filters), **C10** (RBAC `route` action is the single allowlist+residency authority), and the GPU-telemetry-reader invariant.

```json
{
  "stage": "route",
  "rbac": {
    "principal": "tenant:01J8.../svc_account_7",
    "resource": "model",
    "action": "route",                          // single authority, C10
    "check_result": "ALLOW",
    "allowed_backends_derived": ["gpt-4o-mini", "bge-rerank-local", "llama3-70b-eu"]
  },
  "hard_filters": {                              // STATIC policy, fail-closed (C9)
    "allowlist": "PASS",
    "residency": "PASS_eu_only",
    "capability": "PASS_streaming",
    "budget": "PASS_under_ledger_cap"
  },
  "optimization_signals": {                      // fail-OPEN to static ranking (C9)
    "health_store_available": true,
    "latency_p95_ms": { "gpt-4o-mini": 980, "llama3-70b-eu": 2100 },
    "queue_depth": { "gpt-4o-mini": 3 },
    "gpu_utilization_read": { "llama3-70b-eu": 0.62 },  // READER only, never schedules
    "quality_prior": { "gpt-4o-mini": 0.81 }
  },
  "difficulty_score": 0.22,                      // easy query → downgrade-eligible
  "chosen_backend": "gpt-4o-mini",
  "chosen_reason": "easy_query_downgrade + within_residency + lowest_utility_cost",
  "safe_default_pool_satisfies_hard_filters": true,  // C9
  "routing_ms": 4.6
}
```

Two things this record proves and that reviewers will check:

- When `health_store_available: false`, the record must show `optimization_signals` degraded to static ranking **but every hard filter still PASS on static policy** — residency is *never* bypassed (**C9**). The safe-default pool itself satisfies all hard filters.
- `gpu_utilization_read` is in the *signals* block, never in an action block. ContextOS reads GPU telemetry to inform routing; it never schedules a GPU. A record that implied scheduling would be a scope-boundary violation caught by the linter.

## 2.7.4 OpenTelemetry compatibility

We are **OTel-native on the wire and OTel-skeptical as a store**.

```
┌────────────────────────────────────────────────────────────────┐
│ ContextOS gateway (Python 3.11 asyncio + uvloop)                │
│  ┌──────────────┐   stage spans    ┌─────────────────────────┐  │
│  │ Tracer (SDK) │ ───────────────► │ TenantSpanProcessor     │  │
│  └──────────────┘                  │  - drop if tenant_id null│  │
│                                    │  - apply C12 sampler     │  │
│                                    │  - tee cost → outbox     │  │
│                                    └───────────┬─────────────┘  │
│                                                │ OTLP/gRPC       │
└────────────────────────────────────────────────┼───────────────┘
                                                 ▼
                                    ┌─────────────────────────┐
                                    │ OTel Collector (sidecar)│  ← interop boundary
                                    └───────┬─────────────────┘
                            ┌───────────────┼──────────────────┐
                            ▼               ▼                  ▼
                     Tempo/Jaeger      Prometheus         (vendor APM)
                     (traces, lossy)   (metrics)          (optional)

   AUTHORITATIVE provenance (NOT via collector):
   decision_records (Postgres, RLS) + replay bundles (content-addressed, per-tenant-encrypted)
```

**What is OTel-standard:**

- **Trace context propagation** via W3C `traceparent`/`tracestate`. An inbound agent trace's `traceparent` is honored so ContextOS spans correlate with the caller's agent-trace spans — but those agent spans are **READ-ONLY correlation** (scope invariant); we never act on them.
- **Spans** exported over **OTLP/gRPC** to a collector sidecar. The collector is the only place vendor APMs plug in.
- **Metrics** via OTel metrics → Prometheus (the §2.7.6 latency histograms and §2.7.8 token counters).
- **Semantic conventions**: we use `gen_ai.*` conventions where they exist (`gen_ai.request.model`, `gen_ai.usage.input_tokens`, `gen_ai.usage.output_tokens`) and add the `ctxos.*` namespace for our domain attributes.

**What is deliberately NOT delegated to OTel:**

- **Decision records** — these are structured rows in Postgres, not span attributes, because they must be complete (unsampled), queryable under RLS, and crypto-shreddable. An OTel attribute on a sampled-out span is not a record of fact.
- **Cost** — emitted as a span attribute *for convenience* (`gen_ai.usage.*`, `ctxos.cost_usd`) but the **system of record** is the fail-closed cost outbox (§2.7.5), not the collector.

> **Rejected: vendor APM (Datadog/Honeycomb) as system of record.** They are excellent for live ops and we export to them. As the *truth store* they fail three ContextOS requirements: (1) their sampling is opaque and would silently break replay completeness; (2) they are not per-tenant-encrypted, so an RTBF crypto-shred (**C11**) cannot reach them; (3) egress of full context payloads to a third party is a multi-tenant data-governance non-starter. They get spans; they never get the replay bundle.

> **Rejected: OpenTracing.** Deprecated and merged into OTel. No reason to target a dead API.

## 2.7.5 The two write paths (resolution C12)

This is the load-bearing split. Traces and cost records do **not** share a durability class or a sampler.

### Path A — best-effort traces (fail-open, sampled)

```python
class C12Sampler(Sampler):
    """
    Tail-based, fail-open. Keeps a small ambient fraction; force-keeps
    anything that matters for debugging or that is expensive.
    """
    AMBIENT_RATE = 0.05          # 5% baseline (configurable 1–10% per tenant)

    def should_sample(self, ctx: SpanContext, span: ReadableSpan) -> SamplingResult:
        # FORCE-KEEP rules (override ambient drop, never dropped):
        if span.status.is_error():
            return RecordAndSample(reason="force_keep_error")
        if span.attributes.get("ctxos.cost_usd", 0) > 0.05:        # cost > $0.05/req
            return RecordAndSample(reason="force_keep_expensive")
        if span.attributes.get("ctxos.budget_breach"):
            return RecordAndSample(reason="force_keep_budget_breach")
        if span.attributes.get("ctxos.replay_eligible"):
            return RecordAndSample(reason="force_keep_replay")

        # AMBIENT tail sampling:
        if deterministic_hash(ctx.trace_id) < self.AMBIENT_RATE:
            return RecordAndSample(reason="ambient_tail")
        return Drop(reason="ambient_drop")

    # FAIL-OPEN: if the exporter/collector is down, we DROP the trace and
    # serve the request normally. A tracing outage must never 5xx a user.
```

Properties:

- **Sampled**: tail 1–10% ambient (default 5%), set per tenant.
- **Force-keep** for: errors, `cost_usd > $0.05/req`, budget breaches, and replay-eligible requests. The expensive-request force-keep is what stops the long tail of costly requests from being invisible.
- **Fail-open**: a collector outage degrades to *no traces*, never to request failure. Tracing is ambient; it is not on the request's critical path for success.

### Path B — billing-grade cost records (fail-closed, durable outbox)

```python
# Transactional outbox: cost is committed in the SAME Postgres transaction
# as the request's terminal state. No cost row → no commit → fail closed.

async def commit_cost(conn, req: Request, usage: Usage) -> None:
    async with conn.transaction():                 # atomic with terminal write
        await conn.execute(
            """INSERT INTO cost_outbox
               (record_id, tenant_id, request_id, model_id,
                input_tokens, output_tokens, embed_tokens,
                cost_usd, billing_status, created_at)
               VALUES ($1,$2,$3,$4,$5,$6,$7,$8,'PENDING', now())""",
            ulid(), req.tenant_id, req.request_id, usage.model_id,
            usage.input_tokens, usage.output_tokens, usage.embed_tokens,
            usage.cost_usd,
        )
    # A relay worker drains cost_outbox → billing ledger with at-least-once
    # delivery and idempotency on record_id. If the relay is down, rows
    # accumulate durably; nothing is lost. If the INSERT fails, the request's
    # terminal commit fails → fail CLOSED.
```

Properties:

- **Fail-closed**: if the cost row cannot be durably written, the terminal commit does not happen. We do not serve unmetered inference into the void.
- **Durable outbox**: survives crashes; a relay worker forwards to the billing ledger with idempotent at-least-once semantics.
- **Escapes sampling entirely**: cost never depends on the trace sampler. The two paths are independent by construction.
- **Memory consolidation cost is in here too.** The async, rate-limited consolidation batch job is **cost-tracked** — its inference cost lands in `cost_outbox` against the owning tenant and enters the budget ledger. It is a batch job, not an agent loop.

### Replay-eligibility flag

A request is `replay_eligible` if (a) the tenant has replay retention enabled, or (b) it tripped a force-keep rule (error/expensive/breach). For replay-eligible requests we persist the **complete** decision-record set and the **replay manifest** (§2.7.5.1) regardless of trace sampling. This is the bridge: ambient traces are lossy, but the things you'd actually want to replay are never lost.

### Client-abort accounting (resolution C8)

The **terminal-event source decides** write-back and cost:

| Terminal event | Write-back | Cost attribution |
|---|---|---|
| Server `finish_reason` reached | **Commit** write-back | Full cost (input + output tokens actually generated) |
| Client TCP close *before* server terminal | **Discard** write-back | **Partial cost**: input tokens + output tokens streamed *up to the abort point*, attributed to tenant; flagged `terminal: "client_abort"` |

Partial-cost attribution is explicit: we bill the tokens the backend actually produced before the client vanished (the backend charged us for them), tagged `client_abort` so it is auditable and distinguishable from a clean completion. The `ctxos.stream` span carries `ctxos.terminal_event ∈ {server_finish, client_abort}` and the cost outbox row carries the same. This is cited identically in the API spec and the adapter spec — one rule, three documents.

## 2.7.5.1 Replay manifests

The replay manifest is the **index** into a replay bundle. The bundle is content-addressed and per-tenant-encrypted (the flagship Replay Debugger's storage); the manifest is the table of contents that lets `replay()` reconstruct deterministic stages without decrypting the whole world.

```json
{
  "manifest_version": "rm_v3",
  "request_id": "01J9...",
  "tenant_id": "01J8...",
  "created_at": "2026-06-28T14:03:11.380Z",
  "contextos_version": "1.4.2",
  "weight_vocabulary_version": "wv_v9",
  "deterministic_inputs": {                      // everything needed to re-derive decisions
    "normalized_query_hash": "blake3:...",
    "namespace": "proj_alpha",
    "rbac_policy_snapshot_id": "01J2...",        // STATIC policy snapshot (C9/C10)
    "stable_fact_set_version": "facts_v42",
    "system_prompt_version": "sp_v17",
    "tokenizer": "o200k_base"
  },
  "decision_record_ids": {                        // FKs to the C1/C6/.../C9 records
    "cache":    "01J9a...",
    "retrieve": "01J9b...",
    "compress": "01J9c...",
    "assembly": "01J9d...",
    "route":    "01J9e..."
  },
  "candidate_corpus_ref": {                       // content-addressed, NOT re-fetched live
    "type": "cas",
    "digest": "blake3:7c1d...",                   // the ≤512 candidates frozen at request time
    "dek_id": "dek:tenant_01J8...:2026Q2"         // per-tenant DEK (crypto-shred scope, C11)
  },
  "backend_request": {
    "model_id": "gpt-4o-mini-2024",
    "packed_prompt_digest": "blake3:...",         // the exact bytes sent to the backend
    "recorded_output_ref": "cas:blake3:...",      // CAS locator, present iff output recorded
    "recorded_output_digest": "blake3:..."        // BLAKE3 of the recorded output bytes;
                                                  //   re-verified by verify_digest() on replay
  },
  "integrity": {
    "manifest_digest": "blake3:...",
    "signature": "ed25519:..."                    // tamper-evidence
  }
}
```

Key properties:

- **Frozen candidate corpus.** The ≤ 512 candidates are content-addressed and stored in the bundle. Replay does **not** re-query the live retrieval store — the corpus may have changed. Determinism requires freezing inputs.
- **Static policy snapshot.** RBAC/residency decisions replay against the *snapshot* used at request time, not against current policy (resolution **C9/C10**). Otherwise a later policy change would make replay diverge for the wrong reason.
- **Crypto-shred aware (C11).** The candidate corpus and recorded output are encrypted under a per-subject/per-tenant DEK identified by `dek_id`. RTBF destroys the DEK; the manifest then points at undecryptable ciphertext — the request is provably unrecoverable, including its **embeddings** (embeddings are within crypto-shred scope; vector payload/id encrypted under the per-subject DEK). The manifest itself is scrubbed of plaintext by design.

## 2.7.6 Latency breakdown per stage

The §2.7.2 stage spans are the per-request latency breakdown. We do **not** restate the budget numbers here as new facts — **Section 9 owns the one latency table**; this layer's job is to *measure against it* and surface deltas.

For each stage span we record an OTel histogram `ctxos.stage.duration_ms{stage, tenant_id, model_id}` and compare against the stage's `ctxos.p95_budget_ms`. The authoritative budgets we hold each stage to (referenced, not redefined):

| Stage span | p95 budget (Section 9) | In 50 ms assembly? | In 250 ms overhead? |
|---|---|---|---|
| `ctxos.edge.auth` | 5 ms | no | yes |
| `ctxos.cache.lookup` | 10 ms (exact < 1 ms; semantic miss +6 ms embed +4 ms ANN) | no | yes |
| `ctxos.retrieve` | 40 ms (counts toward < 100 ms retrieval SLO) | no | yes |
| `ctxos.assembly` | **50 ms** | **YES** | yes |
| `ctxos.routing` | 5 ms | no | yes |
| `ctxos.adapter` | 8 ms | no | yes |
| `ctxos.writeback.enqueue` | 2 ms | no | yes |

Within `ctxos.retrieve`, child spans expose the internal split so a regression is localizable: pgvector HNSW ANN probe **p95 = 18 ms** (single source of truth for vector-query latency at launch scale ≤ 5M vectors/tenant; Qdrant cutover holds ≤ 25 ms beyond that), BM25 ~12 ms (parallel), query embedding ~6 ms p95 (in-process CPU BAAI/bge-small-en-v1.5, 384-dim), RRF + rescore ~6 ms.

```
ctxos.retrieve            [████████████████████░] 39.7 ms
├─ retrieve.embed         [███░]                   6.1 ms   (reused downstream)
├─ retrieve.ann (pgvec)   [█████████░]            18.0 ms   ┐ run in parallel
├─ retrieve.bm25          [██████░]               12.0 ms   ┘
└─ retrieve.rrf_rescore   [███░]                   5.9 ms
```

**Critical-path semantics, not a sum.** Cache, retrieval, and assembly *overlap*; the < 250 ms control overhead is the **critical-path p95**, not the naive addition of stage budgets. The root `ctxos.request` span's duration is the critical path; the budget-breach detector evaluates it against 250 ms, and per-stage breaches against each stage budget. Model inference is **excluded** from every ContextOS budget — the `ctxos.adapter`/`ctxos.stream` spans measure ContextOS-side handshake and relay, not the backend's think time.

The fail-open retrieval degradation is observable: if query embedding fails, `ctxos.retrieve` carries `recall_mode: "bm25_only"` and `ctxos.retrieve.embed` is absent; the latency drops but the retrieval record's `recall_mode` flags the bounded **≤ 12% recall loss** so the degradation is never silent.

## 2.7.7 Replay — the flagship contract

> **C7: there is exactly ONE `ReplayResult` schema**, shared across the API surface, the observability layer, and the killer-feature debugger. Deterministic stages = all ContextOS decisions. `backend.invoke` is non-deterministic. `output_equal` is asserted **only** for recorded-output replay. `live_backend=True` yields a **diff**, not byte-equality.

### The schema

```python
from dataclasses import dataclass
from enum import Enum

class ReplayMode(Enum):
    RECORDED_OUTPUT = "recorded_output"   # backend output replayed from bundle → output_equal assertable
    LIVE_BACKEND    = "live_backend"      # re-invoke real backend → diff only, never byte-equal

class StageVerdict(Enum):
    DETERMINISTIC_MATCH    = "deterministic_match"     # decision re-derived identically
    DETERMINISTIC_MISMATCH = "deterministic_mismatch"  # ContextOS bug or input drift → investigate
    NON_DETERMINISTIC      = "non_deterministic"       # backend.invoke only

@dataclass(frozen=True)
class StageReplay:
    stage: str                      # edge|cache|retrieve|acl|compress|assembly|route|adapter
    verdict: StageVerdict
    recorded_decision_id: str       # ULID of original decision record
    replayed_decision: dict         # freshly re-derived decision
    diff: dict | None               # populated iff mismatch

@dataclass(frozen=True)
class ReplayResult:
    request_id: str                 # ULID of the original request
    tenant_id: str
    mode: ReplayMode
    contextos_version_original: str
    contextos_version_replay: str
    # DETERMINISTIC stages: all ContextOS decisions, must re-derive identically
    stages: list[StageReplay]
    all_deterministic_stages_match: bool
    # NON-DETERMINISTIC backend:
    backend_invoked: bool
    # output_equal is ONLY meaningful for RECORDED_OUTPUT mode:
    output_equal: bool | None       # None when mode == LIVE_BACKEND
    output_diff: dict | None        # textual/structural diff when mode == LIVE_BACKEND
    notes: str
```

### `replay(trace_id, live_backend) -> ReplayResult`

```python
async def replay(trace_id: str, live_backend: bool = False) -> ReplayResult:
    manifest = await load_manifest(trace_id)          # §2.7.5.1
    _assert_tenant_rls(manifest.tenant_id)            # replay runs under the tenant's RLS

    # 1) Reconstruct DETERMINISTIC stages from frozen inputs + static policy snapshot.
    #    Candidates come from the FROZEN content-addressed corpus, NOT the live store.
    corpus  = await load_cas(manifest.candidate_corpus_ref)   # decrypt under tenant DEK
    policy  = await load_policy_snapshot(manifest.rbac_policy_snapshot_id)  # C9/C10 static
    stages: list[StageReplay] = []

    for stage in ("cache", "retrieve", "acl", "compress", "assembly", "route"):
        original = await load_decision_record(manifest.decision_record_ids.get(stage))
        replayed = await rederive_decision(stage, manifest, corpus, policy)
        if deep_equal(original, replayed):
            stages.append(StageReplay(stage, StageVerdict.DETERMINISTIC_MATCH,
                                      original.id, replayed, None))
        else:
            stages.append(StageReplay(stage, StageVerdict.DETERMINISTIC_MISMATCH,
                                      original.id, replayed,
                                      structural_diff(original, replayed)))

    all_match = all(s.verdict == StageVerdict.DETERMINISTIC_MATCH for s in stages)

    # 2) The BACKEND is non-deterministic — handled per mode (C7).
    if not live_backend:
        # RECORDED_OUTPUT: replay the bundle's recorded output. output_equal IS assertable
        # as a REAL integrity check, not a tautology: recompute BLAKE3 over the recorded
        # bytes pulled from the CAS bundle and assert it equals the digest the signed
        # manifest committed to. A mismatch means the bundle was corrupted or tampered
        # after recording (bit-rot, partial write, malicious edit) and replay must NOT
        # claim integrity. verify_digest also re-checks the ed25519 signature over
        # manifest.integrity so the committed digest is itself trusted.
        recorded_bytes = await load_cas_bytes(manifest.backend_request.recorded_output_ref)
        output_equal = verify_digest(
            recorded_bytes,                                       # bytes actually on disk now
            manifest.backend_request.recorded_output_digest,     # blake3:... committed in manifest
            signature=manifest.integrity.signature,              # ed25519 over manifest_digest
            manifest_digest=manifest.integrity.manifest_digest,
        )
        return ReplayResult(
            request_id=trace_id, tenant_id=manifest.tenant_id,
            mode=ReplayMode.RECORDED_OUTPUT,
            contextos_version_original=manifest.contextos_version,
            contextos_version_replay=CURRENT_VERSION,
            stages=stages, all_deterministic_stages_match=all_match,
            backend_invoked=False,
            # REAL check: recomputed BLAKE3 of recorded bytes == manifest digest under
            # the ed25519 signature. False => detected bundle corruption/tampering.
            output_equal=output_equal,
            output_diff=None,
            notes=("Deterministic stages re-derived; recorded output verified by "
                   "BLAKE3 digest match against the signed manifest (bundle integrity)."),
        )

    # LIVE_BACKEND: re-invoke the real backend with the SAME packed prompt.
    # Output WILL differ (temperature, model drift). We return a DIFF, never byte-equality.
    packed = await load_cas(manifest.backend_request.packed_prompt_digest)
    live_output = await backend_invoke(manifest.backend_request.model_id, packed)
    recorded   = await load_cas(manifest.backend_request.recorded_output_ref)
    return ReplayResult(
        request_id=trace_id, tenant_id=manifest.tenant_id,
        mode=ReplayMode.LIVE_BACKEND,
        contextos_version_original=manifest.contextos_version,
        contextos_version_replay=CURRENT_VERSION,
        stages=stages, all_deterministic_stages_match=all_match,
        backend_invoked=True,
        output_equal=None,                          # NEVER asserted for live backend (C7)
        output_diff=text_diff(recorded, live_output),
        notes="Deterministic stages re-derived; backend re-invoked live → diff only.",
    )
```

### Bundle-integrity verification (`verify_digest`)

`output_equal` in `RECORDED_OUTPUT` mode is a **real cryptographic integrity check**, not a self-comparison. The recorded output is content-addressed; `backend_request.recorded_output_digest` (a `blake3:…` string) is committed inside the manifest, and the whole manifest is in turn covered by an **ed25519** signature over `manifest_digest`. Verification is a two-step chain: (1) confirm the signed manifest is authentic and unaltered, then (2) confirm the bytes on disk still hash to the digest the manifest promised.

```python
import hmac                         # constant-time compare_digest
import blake3                       # BLAKE3, not SHA-256: see rejected-alternative note below
from nacl.signing import VerifyKey  # ed25519 (libsodium / PyNaCl)

def verify_digest(
    recorded_bytes: bytes,          # output bytes actually fetched from the CAS bundle
    committed_digest: str,          # "blake3:<hex>" recorded in manifest.backend_request
    *,
    signature: str,                 # "ed25519:<hex>" over manifest_digest
    manifest_digest: str,           # "blake3:<hex>" of the canonical manifest body
) -> bool:
    """
    True  iff (a) the manifest's ed25519 signature over manifest_digest verifies under
              the tenant-scoped signing key, AND
              (b) BLAKE3(recorded_bytes) == committed_digest.
    False iff the recomputed hash diverges from the committed digest -> the bundle was
              corrupted or tampered with after recording (bit-rot, truncated/partial
              write, or a malicious edit). We surface integrity failure as output_equal
              == False rather than raising, so the ReplayResult records the corruption
              instead of crashing the debugger.
    Raises  only if the SIGNATURE itself fails — an unauthentic manifest is not a
              replayable artifact at all.
    """
    alg_sig, sig_hex = signature.split(":", 1)
    alg_md,  md_hex  = manifest_digest.split(":", 1)
    assert alg_sig == "ed25519" and alg_md == "blake3"
    # (a) signature chain: the committed_digest is only trustworthy if the manifest is.
    VerifyKey(_tenant_signing_pubkey()).verify(
        bytes.fromhex(md_hex), bytes.fromhex(sig_hex)
    )                                                   # raises BadSignatureError if unauthentic
    # (b) recompute over the bytes on disk NOW and compare to the committed digest.
    alg_d, want_hex = committed_digest.split(":", 1)
    assert alg_d == "blake3"
    recomputed_hex = blake3.blake3(recorded_bytes).hexdigest()
    return hmac.compare_digest(recomputed_hex, want_hex)   # constant-time compare
```

`load_cas_bytes()` returns the raw decrypted bytes (under the tenant DEK) so the hash is computed over exactly what was stored — distinct from `load_cas()`, which deserializes into an object for re-derivation. Worked example: a one-bit flip in a stored 4 KB output yields `recomputed_hex = "a91f…"` against a committed `"7c1d…"`, so `verify_digest` returns `False`, the gate in §2.7.9 ("Replay-determinism") fails, and the bundle is quarantined.

> **Rejected: SHA-256 for the content digest.** SHA-256 is correct but ~3–5× slower than BLAKE3 on the multi-MB recorded-output blobs we re-hash on every `RECORDED_OUTPUT` replay and CI determinism run; BLAKE3's tree hashing also parallelizes across cores. We already use `blake3:` for the coarse cache fingerprint (**C6**) and the candidate-corpus digest, so adopting SHA-256 here would mean two hash families in one bundle for no benefit. ed25519 is kept for *signatures* (asymmetric, tamper-evidence by a key the gateway holds), where a hash alone cannot prove authorship.

### What the contract guarantees (and refuses to)

| Aspect | `RECORDED_OUTPUT` (live_backend=False) | `LIVE_BACKEND` (live_backend=True) |
|---|---|---|
| ContextOS decision stages | Re-derived; **must** match (`DETERMINISTIC_MATCH`) or it's a bug | Same — re-derived and must match |
| `backend.invoke` | Replayed from bundle (no real call) | Re-invoked live (real call, real cost) |
| `output_equal` | **Asserted** (bundle integrity) | **None** — never asserted |
| Result on output difference | n/a (deterministic) | Returns `output_diff`, not a failure |
| Use case | Audit, regression CI, "prove the context was X" | "Would today's backend answer differently?" A/B, drift analysis |

This is the crisp line resolution **C7** forces: a `deterministic_mismatch` on any ContextOS stage is a **ContextOS defect** (or input drift the manifest will pinpoint) and is alarmed; a backend output difference under `LIVE_BACKEND` is **expected** and is *data*, not an error. We never let the non-determinism of the model leak into a claim of broken determinism in our own decisions — and we never pretend a live backend call can be byte-equal to a recorded one.

A `LIVE_BACKEND` replay charges real inference; its cost flows through the **fail-closed cost outbox** (§2.7.5) tagged `origin: "replay"`, attributed to the tenant initiating the replay, so debugger usage is itself metered and enters the budget ledger.

## 2.7.8 Token-usage tracking (per request / tenant / model)

Token usage is tracked at three granularities off the **same** events, but the *durability class* differs by purpose: live dashboards read sampled spans; billing reads the fail-closed outbox.

```sql
-- Authoritative usage rollup (materialized from cost_outbox, NOT from spans).
-- Partitioned by tenant_id; FORCE RLS.
CREATE TABLE token_usage_rollup (
    tenant_id        text     NOT NULL,           -- partition key
    window_start     timestamptz NOT NULL,        -- RFC-3339 UTC, hourly bucket
    model_id         text     NOT NULL,
    request_count    bigint   NOT NULL DEFAULT 0,
    input_tokens     bigint   NOT NULL DEFAULT 0,
    output_tokens    bigint   NOT NULL DEFAULT 0,
    embed_tokens     bigint   NOT NULL DEFAULT 0,  -- self-hosted BGE embedding tokens
    compress_tokens_saved bigint NOT NULL DEFAULT 0,
    cache_hits       bigint   NOT NULL DEFAULT 0,
    cache_misses     bigint   NOT NULL DEFAULT 0,
    routing_downgrades bigint NOT NULL DEFAULT 0,
    cost_usd         numeric(12,6) NOT NULL DEFAULT 0,
    PRIMARY KEY (tenant_id, window_start, model_id)
) PARTITION BY LIST (tenant_id);
```

OTel counters (for live ops, sampled-tolerant since they're rate gauges, not billing):

```
ctxos.tokens.input{tenant_id, model_id}        # counter
ctxos.tokens.output{tenant_id, model_id}       # counter
ctxos.tokens.embed{tenant_id}                  # counter (BGE, 384-dim)
ctxos.cost_usd{tenant_id, model_id, origin}    # counter (origin: live|consolidation|replay)
ctxos.cache.hit_ratio{tenant_id}               # gauge, target 25–45%
ctxos.savings.pct{tenant_id, source}           # gauge, source: cache|routing|compression
```

The savings counters let us substantiate the **40–65% combined token-cost savings** claim with live data rather than a slide:

| Savings source | Contribution band | Counter |
|---|---|---|
| Semantic + exact caching | 15–30% | `ctxos.savings.pct{source="cache"}` |
| Model-routing downgrade on easy queries | 20–40% of model spend | `ctxos.savings.pct{source="routing"}` |
| Compression (prompt-token reduction) | 10–25% | `ctxos.savings.pct{source="compression"}` |

> **Billing reads the rollup, never the spans.** A dashboard may show "this hour's input tokens" from sampled spans (cheap, approximate). An invoice is computed from `token_usage_rollup`, which is materialized only from the fail-closed `cost_outbox`. The two can differ by the sampling factor — by design — and only the rollup is authoritative. This is **C12** at the reporting layer: approximate where cheap, exact where money is involved.

## 2.7.9 CI gates and invariants enforced by observability

Observability is also a **test oracle**. These hard gates run in CI against recorded traces and manifests:

| Gate | Asserts | Failure = |
|---|---|---|
| Stage-order linter | Span tree follows the pipeline invariant; `compression` strictly after `acl_redaction` | Pipeline-ordering bug |
| Tenant-completeness | Every span/decision/cost row has non-null `tenant_id` under RLS | Cross-tenant leak risk |
| Cross-tenant probe | ≥ 10,000 hostile second-tenant property probes find **0** leakage across spans, decision records, and replay bundles | Isolation breach (blocks merge) |
| Cost-completeness | Every `finish_reason`-terminal request has a `cost_outbox` row; client-abort requests have a partial-cost row tagged `client_abort` | Revenue leak / C8 violation |
| Replay-determinism | For a corpus of recorded requests, `RECORDED_OUTPUT` replay yields `all_deterministic_stages_match == True` and `output_equal == True` | ContextOS decision non-determinism (a defect) |
| Scope-boundary linter | No decision record implies index-building, GPU scheduling, agent-step re-execution, or write to agent-trace spans | Scope drift |
| Crypto-shred reachability | After DEK destruction, replay of the subject's request returns undecryptable corpus (incl. embeddings) | RTBF (C11) failure |
