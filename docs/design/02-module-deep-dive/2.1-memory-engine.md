# 2.1 Memory Engine

The Memory Engine is ContextOS's durable, multi-tier recollection subsystem. It owns *what the system remembers about a tenant* and the *retrieval of pre-ranked candidates* — and nothing more. It is **not** a vector database, **not** a retrieval-as-a-service product, and **not** the final-ranking authority. Per the scope-boundary invariants, the Memory Engine **never builds or owns an index** beyond the co-located pgvector HNSW structure it queries through a `VectorStore` adapter; it scores and fuses *over a bounded, pre-retrieved candidate set (≤ 512)* and hands raw signals upward.

Two hard contracts govern everything below:

- **C1 — Retrieval ownership.** The Memory Engine returns `MemoryCandidate` objects carrying **raw per-modality scores only** (vector distance, BM25 score, recency factor, importance). It does **not** compute a single blended final rank, does **not** apply lost-in-the-middle edge placement, and does **not** perform budget knapsack. The **Context Assembler is the sole final-ranking and budget-packing authority** (see §2.2). The Memory Engine's only *ordering* responsibility is internal: producing a deterministic, deduplicated top-k candidate list. Memory-decay recency is a **per-candidate signal**, orthogonal to the Assembler's positional ordering.
- **C2 — Namespace isolation.** Every retrieval and write is scoped by `(tenant_id, namespace)` where `namespace ∈ {project, agent, user}`. The within-tenant namespace is a **HARD filter, evaluated at the repository boundary alongside `tenant_id`, fail-closed**. A missing or ambiguous namespace is a **deny** (`NamespaceAmbiguousError`), never a wildcard. Cross-org shared namespaces exist only behind an explicit `RBACPolicy` `route`/`read` grant; absent that grant, the shared scope is invisible.

## 2.1.1 The Four Memory Tiers

ContextOS models memory as four functional tiers mapped onto **two physical stores** (canonical): working/short-term in **Redis (TTL)**, and long-term/episodic/semantic in **Postgres 16 + pgvector**. We reject a single-store design (Postgres-only) because conversational working memory has a sub-millisecond read SLO and a natural TTL that Redis serves far better than a partitioned relational table; we reject a memory-specific store (e.g. a bespoke embedded KV like RocksDB per pod) because it fragments the multi-tenant RLS and crypto-shred guarantees that Postgres gives us for free.

| Tier | Lifetime | Physical store | Eviction | Typical content | Embedded? |
|---|---|---|---|---|---|
| **Working** | Current turn / request window | Redis (TTL 30 min, sliding) | TTL + LRU | In-flight scratch, last-N turns, tool outputs | No (raw) |
| **Short-term** | Current session | Redis (TTL 24 h) | TTL | Session dialogue, recent facts | Lazy (on consolidation trigger) |
| **Long-term / episodic** | Indefinite, decayed | Postgres 16 + pgvector (HNSW) | Decay-pruned (§2.1.5) | Dated events: "user deployed v2 on 2026-06-01" | Yes (384-dim) |
| **Semantic** | Indefinite, decayed slower | Postgres 16 + pgvector (HNSW) | Decay-pruned, longer half-life | Distilled facts: "user's prod region is eu-west-1" | Yes (384-dim) |

**Why two Redis tiers rather than one:** working memory is *write-hot, read-hot, throwaway*; short-term is *read-warm and a consolidation candidate*. Collapsing them forces a single TTL that is either too short for session continuity or too long for working-set memory pressure. They are distinct Redis keyspaces, both `tenant_id`-namespaced (`t:{tenant_id}:n:{namespace}:wm:{session_id}` and `:stm:{session_id}`).

**Why episodic and semantic share pgvector but differ in decay:** they are the *same table* (`memory`) discriminated by a `tier` enum, so hybrid retrieval queries both in one ANN sweep. What differs is the **decay half-life** (§2.1.5): episodic memories are time-bound and should fade; semantic facts are intended to be durable and decay an order of magnitude slower. We reject separate tables because RRF fusion and the candidate cap operate cleanest over one candidate pool, and a `WHERE tier = ANY($scope_tiers)` predicate is index-friendly under the partial HNSW indexes we maintain.

## 2.1.2 The `Memory` Object

```python
from __future__ import annotations
from dataclasses import dataclass, field
from enum import Enum
from typing import Optional
from datetime import datetime  # all timestamps RFC-3339 UTC

class MemoryTier(str, Enum):
    WORKING   = "working"
    SHORT_TERM = "short_term"
    EPISODIC  = "episodic"
    SEMANTIC  = "semantic"

class MemoryModality(str, Enum):
    TEXT = "text"          # default; embeddable + tsvector-indexable
    EVENT = "event"        # structured episodic event
    FACT = "fact"          # distilled semantic assertion

@dataclass(frozen=True, slots=True)
class MemoryScope:
    tenant_id: str                  # non-null partition key on EVERY row/key (ULID)
    namespace_kind: str             # "project" | "agent" | "user"  (C2 hard filter)
    namespace_id: str               # ULID of the project/agent/user
    # Resolution is fail-closed: missing/ambiguous -> NamespaceAmbiguousError (deny)

@dataclass(slots=True)
class Memory:
    # ---- identity ----
    id: str                         # ULID, time-sortable
    scope: MemoryScope              # (tenant_id, namespace_kind, namespace_id)
    tier: MemoryTier
    modality: MemoryModality = MemoryModality.TEXT

    # ---- content ----
    content: str = ""               # canonical text (raw, pre-compression)
    structured: dict = field(default_factory=dict)  # event/fact payload
    source_ref: Optional[str] = None  # ULID of originating turn/trace span (READ-ONLY correlation)

    # ---- retrieval signals (raw; the Assembler blends, not us — C1) ----
    embedding: Optional[bytes] = None  # 384-dim BAAI/bge-small-en-v1.5, ENCRYPTED under per-subject DEK (C11)
    tsv_present: bool = False          # has a Postgres tsvector row for BM25

    # ---- decay & ranking inputs ----
    importance: float = 0.5         # [0,1]; consolidation may raise; pin -> 1.0
    pinned: bool = False            # exempt from decay pruning
    created_at: datetime = None     # RFC-3339 UTC
    last_accessed_at: datetime = None  # updated on retrieval (recency)
    access_count: int = 0

    # ---- lifecycle ----
    consolidated_from: list[str] = field(default_factory=list)  # source memory ULIDs
    tombstoned: bool = False        # RTBF marker; GC sweep crypto-shreds (C11)
    version: int = 1
```

**Design notes.**

- **`importance ∈ [0,1]`, default 0.5.** This is an explicit, persisted scalar, not a derived heuristic at query time. The consolidation job (§2.1.6) is the only writer that *raises* importance (a fact referenced across N sessions earns importance); explicit user pins set `pinned=True` and `importance=1.0`. We reject computing importance purely from `access_count` at read time because it makes retrieval non-deterministic w.r.t. concurrent reads and pollutes the recency signal.
- **`embedding` is encrypted at rest (C11).** The 384-dim vector payload *and* the memory `id` reference in the vector path are encrypted under the per-subject Data Encryption Key. RTBF crypto-shred of the DEK renders the vector unrecoverable; embeddings are explicitly **within** crypto-shred scope.
- **`source_ref` is READ-ONLY correlation.** It links a memory to the agent-trace span or turn that produced it for replay/debug. The Memory Engine never re-executes or schedules from this reference (scope-boundary invariant).

## 2.1.3 Write Policy

The write path is **two-phase by design**: a *synchronous* write to working memory (Redis) on the request hot path, and an *asynchronous* consolidation into durable pgvector memory off the hot path via Redis Streams. This split is the only way to keep the write-back enqueue at **2 ms p95** (Section 9 latency table) while still eventually producing embeddable, searchable long-term memory.

```
write(memory):
   phase 1 (SYNC, on hot path, <=2ms): validate scope (C2) -> Redis SET (working/short-term, TTL)
   phase 2 (ASYNC, Redis Streams): consolidation consumer -> embed -> RLS INSERT into pgvector
```

Why this ordering matters: **the embedding step (~6 ms p95 CPU) must never block the request's async write-back enqueue (2 ms p95).** Embedding on the hot path would blow the write-back budget threefold. So `write()` returns after the Redis SET and the stream `XADD`; the embed + durable insert happen on the consumer.

### Synchronous path — `write()`

```python
async def write(self, memory: Memory) -> str:
    """Synchronous working-memory write + async consolidation enqueue.
    Returns the memory ULID. Hot-path budget: <= 2 ms p95 (Section 9)."""
    # C2: fail-closed namespace resolution at the repository boundary
    scope = self._resolve_scope_or_deny(memory.scope)   # raises NamespaceAmbiguousError

    memory.id = memory.id or ulid()
    memory.created_at = memory.created_at or utc_now()
    memory.last_accessed_at = memory.created_at

    # Phase 1: synchronous Redis write (working OR short-term tier)
    key = self._redis_key(scope, memory.tier, memory.source_ref)
    ttl = WORKING_TTL_S if memory.tier is MemoryTier.WORKING else SHORT_TERM_TTL_S
    await self.redis.set(key, self._serialize(memory), ex=ttl)

    # Phase 2: enqueue durable consolidation (does NOT block on embed/insert)
    await self.redis.xadd(
        self._consolidation_stream(scope.tenant_id),
        {"memory_id": memory.id, "scope": scope.encode(), "op": "ingest"},
    )
    return memory.id
```

**Redis Streams, not Kafka, not RabbitMQ, not Celery.** We already run Redis for the exact-hash cache and working memory; reusing it as the async plane avoids a second piece of stateful infrastructure. Redis Streams gives us consumer groups, at-least-once delivery, and replayable offsets — and that stream **doubles as the replay log** the Context Replay Debugger consumes. We reject Kafka as operationally heavy for the per-tenant volumes at launch scale; we reject Celery/RabbitMQ because they are opaque task queues, not an append-only, offset-addressable log we can replay byte-for-byte. The consumer is a **custom asyncio consumer** (not Celery workers) so we keep one runtime and one backpressure model.

**Idempotency.** Consolidation is keyed on `memory_id`; the durable `INSERT` is `ON CONFLICT (tenant_id, id) DO NOTHING`. At-least-once delivery therefore yields exactly-once durable state.

### Asynchronous consolidation consumer (ingest)

```python
async def _consume_ingest(self, entry: StreamEntry) -> None:
    scope = MemoryScope.decode(entry["scope"])
    mem = await self._load_from_redis(scope, entry["memory_id"])
    if mem is None:
        return  # TTL expired before consolidation -> intentionally dropped (working memory is throwaway)

    # Embedding cost is COST-TRACKED and enters the budget ledger (scope-boundary invariant)
    vec = await self.embeddings.embed(mem.content)        # ~6 ms p95, 384-dim
    ct = self.crypto.encrypt_under_dek(scope, vec)        # C11: encrypted vector payload

    async with self.repo.tenant_session(scope.tenant_id):  # SET LOCAL app.tenant_id -> RLS
        await self.repo.upsert_memory(scope, mem, encrypted_embedding=ct)  # ON CONFLICT DO NOTHING
        await self.repo.upsert_tsvector(scope, mem)        # to_tsvector('english', content)
    self.cost_ledger.record_embedding(scope.tenant_id, tokens=count_tokens(mem.content))
```

## 2.1.4 Hybrid Retrieval

Retrieval is **dense + sparse, fused by Reciprocal Rank Fusion (RRF)**, then rescored with recency and importance, then deduplicated with MMR. The whole subsystem SLO is **< 100 ms p95** (hot path typically ~40 ms), per the canonical facts and the Section 9 budget line (`embed reused; pgvector ANN 18 ms || BM25 12 ms; RRF + rescore 6 ms`).

```
retrieve(query, scope, k):
   0. C2 namespace hard filter (fail-closed)                              [repo boundary]
   1. embed query  -> 384-dim                          ~6 ms (REUSED from cache-miss embed if present)
   2a. pgvector HNSW ANN  (top N_dense)                 18 ms p95  ──┐
   2b. Postgres tsvector BM25 (top N_sparse)            12 ms p95  ──┤ run concurrently (asyncio.gather)
   3. RRF fuse (k=60)                                                │
   4. recency + importance rescore                       6 ms ───────┘  (steps 3-5 share this budget)
   5. MMR dedup (lambda=0.7, cosdup>=0.95) -> top k
   6. return list[MemoryCandidate]  (RAW per-modality scores ONLY — C1)
```

`N_dense = N_sparse = 200` candidate ceilings; the **fused, post-MMR result is hard-capped at ≤ 512** before it ever leaves the engine (scope-boundary invariant: ContextOS never exceeds the 512 candidate cap into assembly). In practice `k` requested by the Assembler is far smaller (typically 12–48); the 512 cap is the absolute ceiling.

### The `retrieve()` signature and `MemoryCandidate`

```python
@dataclass(frozen=True, slots=True)
class MemoryCandidate:
    """RAW per-modality scores ONLY. The Context Assembler (§2.2) computes the
    single final rank and does budget packing. The Memory Engine NEVER blends
    these into one number for downstream ranking (C1)."""
    memory_id: str
    tier: MemoryTier
    content: str                 # raw, pre-compression (compression runs AFTER ACL/redaction, §2.3)
    # --- raw per-modality signals (un-blended) ---
    vector_score: Optional[float]   # cosine similarity in [0,1]; None if sparse-only hit
    bm25_score: Optional[float]     # Postgres ts_rank_cd; None if dense-only hit
    rrf_score: float                # fusion rank score (internal ordering only)
    recency_factor: float           # exp decay in (0,1], see §2.1.5
    importance: float               # [0,1]
    rank_in_dense: Optional[int]    # 1-based; None if not in dense list
    rank_in_sparse: Optional[int]   # 1-based; None if not in sparse list
    age_seconds: float
    source_ref: Optional[str]       # READ-ONLY correlation

async def retrieve(
    self, query: str, scope: MemoryScope, k: int
) -> list[MemoryCandidate]:
    """Hybrid dense+sparse retrieval over the (tenant_id, namespace) scope.
    SLO: < 100 ms p95 (typ. ~40 ms). Returns RAW candidates (C1).
    k is the caller's desired candidate count; hard ceiling is min(k, 512)."""
    scope = self._resolve_scope_or_deny(scope)          # C2 fail-closed
    k = min(k, MAX_CANDIDATES)                            # 512 ceiling

    qvec = await self.embeddings.embed(query)            # ~6 ms; reused if cache miss already embedded
    async with self.repo.tenant_session(scope.tenant_id):  # RLS SET LOCAL
        dense_task = self.vstore.ann(scope, qvec, top=N_DENSE)        # pgvector HNSW, 18 ms p95
        sparse_task = self.repo.bm25(scope, query, top=N_SPARSE)      # tsvector, 12 ms p95
        dense, sparse = await asyncio.gather(dense_task, sparse_task) # concurrent
    fused = self._rrf_fuse(dense, sparse, k_rrf=RRF_K)   # k=60
    rescored = self._rescore(fused)                      # recency + importance (raw signals attached)
    deduped = self._mmr(rescored, lam=MMR_LAMBDA, dup=COS_DUP)        # 0.7 / 0.95
    await self._touch_recency(scope, [c.memory_id for c in deduped]) # async, off critical path
    return deduped[:k]
```

### pgvector HNSW — the dense path

The dense leg is a **pgvector HNSW** ANN query. **Canonical latency: p95 = 18 ms** at launch scale (≤ 5M vectors/tenant). This is the single source of truth for vector-query latency; no other figure is used. Beyond 5M vectors/tenant, the `VectorStore` adapter cuts over to **Qdrant** holding ≤ 25 ms p95.

We reject IVFFlat (pgvector's other index) because its recall/latency degrades on high-churn tenant data and requires periodic `REINDEX`/list re-tuning; HNSW gives stable p95 under incremental inserts, which matches our async consolidation write pattern. We reject a standalone vector DB *at launch* (Qdrant/Weaviate/Milvus) because co-locating vectors in Postgres gives us `tenant_id` partitioning, `FORCE ROW LEVEL SECURITY`, transactional consistency with the relational facts, and crypto-shred under one DEK regime — for free. Qdrant remains the documented **escape hatch** at scale, behind the same adapter, not a rewrite.

HNSW tuning at launch scale: `m = 16`, `ef_construction = 64`, query-time `ef_search = 64` (yields the 18 ms p95 against recall ≥ 0.95). Partial HNSW indexes are maintained per `tier` so the `WHERE tier = ANY(...)` predicate stays index-only.

### Postgres `tsvector` BM25 — the sparse path

The sparse leg is full-text `ts_rank_cd` over a maintained `tsvector` column (GIN-indexed), giving lexical/keyword recall the dense path misses (exact identifiers, rare tokens, error codes). **Canonical latency: 12 ms p95.** We reject Elasticsearch/OpenSearch for the sparse leg because it is a third stateful system to operate, shard, and secure per-tenant when Postgres FTS already covers our recall needs at launch scale; it also breaks the single-store crypto-shred and RLS story. Postgres `tsvector` lives in the same row, same RLS policy, same partition.

### Reciprocal Rank Fusion (k = 60)

We fuse the two ranked lists with **RRF**, not a learned linear combination of raw scores, because dense cosine and BM25 `ts_rank_cd` are **not on a comparable scale** and their distributions shift per query — normalizing them into a weighted sum is brittle and tenant-specific. RRF depends only on **rank position**, is parameter-light, and is robust to scale mismatch.

```python
RRF_K = 60  # canonical RRF constant

def _rrf_fuse(self, dense, sparse, k_rrf=RRF_K):
    scores: dict[str, float] = {}
    raw: dict[str, MemoryCandidate] = {}
    for rank, hit in enumerate(dense, start=1):
        scores[hit.id] = scores.get(hit.id, 0.0) + 1.0 / (k_rrf + rank)
        raw[hit.id] = _attach(raw.get(hit.id, hit), rank_in_dense=rank, vector_score=hit.cos)
    for rank, hit in enumerate(sparse, start=1):
        scores[hit.id] = scores.get(hit.id, 0.0) + 1.0 / (k_rrf + rank)
        raw[hit.id] = _attach(raw.get(hit.id, hit), rank_in_sparse=rank, bm25_score=hit.bm25)
    ordered = sorted(scores.items(), key=lambda kv: kv[1], reverse=True)
    return [raw[mid]._replace_rrf(s) for mid, s in ordered]
```

`k = 60` is the standard RRF constant (Cormack et al.); it damps the contribution of low-ranked hits so a single list cannot dominate. We reject `k = 1` (over-weights the top hit, brittle) and very large `k` (flattens the ranking into near-uniform, discarding signal). **The RRF score is an internal ordering signal only** — it is reported on the candidate (`rrf_score`) for transparency/replay but is **not** the final blended rank; the Assembler reranks (C1).

## 2.1.5 Recency + Importance Rescore, and MMR Dedup

After fusion, each candidate carries an internal ordering that *also* reflects how recent and how important the memory is. **These are computed as explicit, raw factors attached to the candidate** — they are *not* folded into a single number the Assembler must trust blindly (C1). The Assembler may weight them per its own one weight vocabulary; the Memory Engine's internal ordering exists only to pick which ≤ 512 candidates survive.

### Recency factor (exponential half-life decay)

```
recency_factor(age) = 0.5 ** (age_seconds / half_life_seconds)
```

Half-lives differ **per tier** because episodic events should fade while semantic facts should persist:

| Tier | Recency half-life | Rationale |
|---|---|---|
| Working | 30 min | Matches Redis TTL; near-1.0 within a turn |
| Short-term | 12 h | Half a session-day |
| **Episodic** | **7 days** | Dated events lose relevance within a week |
| **Semantic** | **90 days** | Distilled facts are durable; decay an order of magnitude slower |

`half_life(episodic) = 7 d`, `half_life(semantic) = 90 d`. The ~13× ratio is the formal expression of "episodic fades, semantic persists." A 7-day-old episodic memory has `recency_factor = 0.5`; a 7-day-old semantic fact has `recency_factor = 0.5**(7/90) ≈ 0.947`.

### Internal ordering score (engine-internal ONLY)

For *internal candidate selection* (deciding which survive into the ≤ 512 set), the engine combines RRF, recency, and importance with **fixed, documented weights**:

```python
# ENGINE-INTERNAL selection score only. NOT the final rank (C1).
W_RRF, W_RECENCY, W_IMPORTANCE = 0.60, 0.25, 0.15   # sum to 1.0

def _selection_score(c: MemoryCandidate) -> float:
    return (W_RRF * normalize(c.rrf_score)
            + W_RECENCY * c.recency_factor
            + W_IMPORTANCE * c.importance)
```

These weights (`0.60 / 0.25 / 0.15`) are the **single weight vocabulary** for the Memory Engine's internal ordering. They prioritize relevance (RRF) while letting recency break ties and importance rescue durable facts. **Pinned memories (`importance = 1.0`) are exempt from being dropped at the 512 boundary** as long as they appear in either retrieval list.

### MMR dedup — λ = 0.7, cosine duplicate threshold = 0.95

Maximal Marginal Relevance removes near-duplicate memories (the same fact restated across sessions) so the candidate set is diverse before it reaches the Assembler.

```python
MMR_LAMBDA = 0.70   # relevance vs. diversity tradeoff
COS_DUP    = 0.95   # >= this cosine -> treated as duplicate, collapsed

def _mmr(self, cands, lam=MMR_LAMBDA, dup=COS_DUP):
    selected: list[MemoryCandidate] = []
    pool = sorted(cands, key=self._selection_score, reverse=True)
    while pool and len(selected) < MAX_CANDIDATES:
        best, best_val = None, -1e9
        for c in pool:
            rel = self._selection_score(c)
            max_sim = max((cos(c, s) for s in selected), default=0.0)
            if max_sim >= dup:          # hard duplicate collapse
                continue
            val = lam * rel - (1 - lam) * max_sim
            if val > best_val:
                best, best_val = c, val
        if best is None:
            break
        selected.append(best); pool.remove(best)
    return selected
```

`λ = 0.70` weights relevance over diversity (we are deduping, not exploring). `cosdup ≥ 0.95` is a **hard collapse**: two memories that similar are the same fact and one is discarded outright (keeping the higher-importance / more-recent one). We reject pure top-k-by-score (no diversity → the Assembler receives five paraphrases of one fact, wasting its token budget) and we reject λ ≤ 0.5 (over-diversifies, drops genuinely relevant near-neighbors). The cross-encoder reranker is **opt-in, out-of-band only** and is never inserted into this hot path (scope-boundary invariant).

## 2.1.6 Decay, Forgetting, and Cross-Session Consolidation

Memory that only grows is memory that rots: retrieval quality and pgvector latency both degrade as dead facts accumulate. The Memory Engine therefore runs a **decay/forgetting pass** and a **consolidation/summarization pass** as **async, rate-limited, COST-TRACKED batch jobs** — explicitly **not** an agent loop (scope-boundary invariant). Their LLM/embedding inference cost **enters the budget ledger** like any other model spend.

### Decay & forgetting

A periodic GC sweep (per tenant, off the hot path) computes a **retention score** and prunes or down-weights:

```
retention(m) = importance(m) * recency_factor(age(m), half_life[m.tier])

if m.pinned:                      keep, no change
elif retention(m) < FORGET_FLOOR:  hard-delete (tombstone + crypto-shred, C11)
elif retention(m) < DOWNWEIGHT:    keep but multiply importance by 0.9 (gentle decay)
else:                              keep
```

| Constant | Value | Meaning |
|---|---|---|
| `FORGET_FLOOR` | 0.02 | Below this, memory is hard-deleted (tombstone + GC crypto-shred) |
| `DOWNWEIGHT` | 0.10 | Below this, importance is multiplicatively decayed by 0.9 per sweep |
| Sweep cadence | every 6 h per tenant | Rate-limited; bounded rows/sweep |

Hard-delete uses the **RTBF mechanism (C11): tombstone + idempotent GC sweep**, and because the embedding is encrypted under the per-subject DEK, deletion crypto-shreds the vector payload — embeddings are within crypto-shred scope. We reject `DELETE`-only forgetting because the vector index and the encrypted-at-rest backups would otherwise retain recoverable embeddings; tombstone + DEK-shred guarantees true forgetting.

### Cross-session consolidation / summarization

Consolidation distills many low-importance, redundant episodic memories across sessions into fewer, higher-importance semantic facts. It is a **batch job**, rate-limited, cost-tracked:

```python
async def _consolidate_session_cluster(self, scope, cluster: list[Memory]) -> None:
    """Distill a cluster of redundant episodic memories into semantic fact(s).
    Rate-limited, cost-tracked batch — NOT an agent loop. Inference cost -> ledger."""
    if self.rate_limiter.exceeded(scope.tenant_id, "consolidation"):
        return  # bounded: deferred to next window, never unbounded fan-out

    # Summarize via the routed backend; this is COST-TRACKED model spend (budget ledger)
    summary, cost = await self.summarizer.summarize(  # opt-in, bounded prompt
        [m.content for m in cluster], max_facts=8)
    self.cost_ledger.record_consolidation(scope.tenant_id, cost)  # enters budget ledger

    fact = Memory(
        id=ulid(), scope=scope, tier=MemoryTier.SEMANTIC,
        modality=MemoryModality.FACT, content=summary,
        importance=min(1.0, max(m.importance for m in cluster) + 0.2),  # distilled => more important
        consolidated_from=[m.id for m in cluster],
        created_at=utc_now(), last_accessed_at=utc_now(),
    )
    await self.write(fact)                       # re-enters async embed + insert
    for m in cluster:                            # down-weight sources (do not delete; preserve provenance)
        await self.repo.set_importance(scope, m.id, m.importance * 0.5)
```

**Triggering.** Consolidation candidates are clusters of short-term/episodic memories with mutual cosine ≥ 0.90 within a session, accumulated over a 24 h window, processed at most **once per 6 h per tenant** (the same rate-limit window as decay). The rate limiter makes the job's cost bounded and predictable; a runaway consolidation cannot fan out into an agent-like loop. **Source memories are down-weighted, not deleted**, preserving provenance (`consolidated_from`) for the Context Replay Debugger.

We reject an LLM-driven "memory agent" that decides what to remember in a loop: it would (a) violate the scope-boundary invariant against agent loops, (b) make cost unbounded, and (c) make memory non-deterministic and thus unreplayable. Batch consolidation with a fixed rate limit and a ledgered cost is deterministic at the schedule level and fully auditable.

## 2.1.7 Failure Modes

The governing principle: **availability failures fail OPEN (serve degraded, without memory); isolation/correctness failures fail CLOSED (deny).** Memory is an enhancement, not a correctness gate — except for the namespace boundary, which is a security boundary and must deny on ambiguity.

| Failure | Posture | Behavior | Recall impact |
|---|---|---|---|
| **Working/short-term Redis down** | **Fail-OPEN** | `retrieve()` skips Redis tiers; serve from pgvector long-term only; request proceeds without working memory | Loses in-session continuity only |
| **pgvector / Postgres ANN down** | **Fail-OPEN** | Skip dense leg; serve **BM25-only** from a read replica if available; if all Postgres down, return empty candidate set and proceed | Bounded recall loss ≤ 12% (BM25-only) |
| **Embedding provider down** | **Fail-OPEN** | **BM25-only retrieval** (no query vector); request proceeds | Bounded recall loss ≤ 12% |
| **Ambiguous / missing namespace (C2)** | **Fail-CLOSED** | Raise `NamespaceAmbiguousError` → deny; **never** wildcard across namespaces | N/A — request rejected |
| **Cross-namespace / cross-tenant scope mismatch** | **Fail-CLOSED** | Repository boundary rejects; RLS `SET LOCAL` mismatch aborts txn | N/A — denied (0 leakage) |
| **Consolidation backlog (Streams lag)** | **Fail-OPEN, degrade** | New memories searchable only via working/short-term Redis until consumer catches up; durable insert eventually-consistent | Temporary: long-term not yet indexed |
| **Rate-limit window exceeded (consolidation/decay)** | **Defer** | Job deferred to next 6 h window; no unbounded fan-out | None to retrieval |
| **DEK unavailable for a subject (post-RTBF)** | **Fail-CLOSED** | Encrypted embedding undecryptable → candidate omitted (crypto-shred is correct, intended forgetting) | Intended |

**Why BM25-only is the universal degraded mode.** When either the embedder or pgvector ANN is unavailable, the sparse lexical leg over Postgres `tsvector` still functions and bounds recall loss to **≤ 12%** (canonical). This is the same fail-open path the cache subsystem uses on embedding failure, keeping the degraded behavior consistent across ContextOS. The retrieval SLO (< 100 ms p95) is *maintained or beaten* in this mode because the 18 ms dense leg is skipped.

**Why the namespace boundary is the lone fail-closed exception.** Serving without memory is a quality regression; serving the *wrong tenant's or wrong namespace's* memory is a security incident. C2 makes the namespace a hard filter at the repository boundary with `tenant_id`, fail-closed, so cross-tenant leakage stays at the canonical **0** (enforced by Postgres `FORCE ROW LEVEL SECURITY` + app RBAC firewall, gated in CI by ≥ 10,000 hostile second-tenant probes).
