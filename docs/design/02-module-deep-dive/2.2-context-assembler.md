# 2.2 Context Router / Assembler

The Context Assembler is the **sole final-ranking and token-budget authority** in ContextOS (resolution **C1**). Every other subsystem — the Memory Engine, the semantic cache, the retrieval adapters — hands it *candidates with raw, per-modality scores*. Nothing else in the system decides what survives into the prompt, in what order, or under what token reserve. This single-owner design is deliberate: ranking and packing are the two places where "lost-in-the-middle" failures, silent budget overflows, and cross-modality score-laundering bugs hide, and we refuse to spread that authority across services where it would drift out of sync.

The Assembler sits at the **assembly/packing** position in the pipeline invariant:

```
auth/tenant -> cache lookup -> retrieve candidates -> ACL/redaction
            -> compression -> [ASSEMBLY/PACKING] -> routing -> adapter
            -> stream -> async write-back
```

It receives candidates that have **already** passed ACL/redaction and **already** been compressed (compression ALWAYS runs after ACL/redaction — a candidate's token count is final by the time the Assembler sees it). It runs **before** the adapter dispatch but has a tokenizer dependency on routing that we resolve explicitly below (C3).

**Hard performance contract:** context assembly — score over `<=512` pre-retrieved candidates, MMR diversification, budget knapsack, and edge placement — completes in **< 50 ms p95**. This budget *excludes* retrieval I/O and model inference; it is pure in-process CPU work. It is line 4 of the authoritative latency table (Section 9):

| Stage | p95 | counts toward `<50ms` assembly | counts toward `<250ms` overhead |
| --- | --- | --- | --- |
| Context assembly (score `<=512`, MMR, knapsack, edge-place) | 50 ms | **YES** | yes |

---

## 2.2.1 Scope boundaries the Assembler must not cross

The Assembler is a CPU-bound in-process scorer. It is **not** a retrieval engine, **not** an index, and **not** a reranking service. The invariants (C13) constrain it precisely:

- It scores and packs **only over pre-retrieved candidates**, capped at **`<=512`**. It NEVER issues a vector query, NEVER builds or owns an index, and NEVER widens the candidate set. If fewer than the cap arrive, it works with what it has; if more arrive, the Memory Engine truncated upstream by raw fused score before handing off.
- Any **cross-encoder reranker is opt-in and out-of-band** — it is a separate pluggable stage that, when enabled, *rewrites the `vector` signal score* before the Assembler runs. The Assembler itself does no neural inference; it does linear blending and greedy combinatorics only. This keeps the `<50ms` budget honest.
- It is **stateless** per request. All weights and budgets come from the request's resolved `TenantPolicy`; it holds no mutable cross-request state.

The `<=512` cap is the single most load-bearing number for the latency budget. At 512 candidates with 384-dim embeddings, the full score+MMR+knapsack pass is a few hundred microseconds of float math plus memory traffic — comfortably inside 50 ms even on a `~0.7 vCPU`-per-1k-req/s hot path. We rejected an *unbounded* candidate set (the obvious "just rank everything" design) precisely because it turns a bounded `O(n^2)` MMR into a latency landmine and quietly makes ContextOS an index owner.

## 2.2.2 Blended relevance scoring — ONE weight vocabulary

Per **C1**, the Memory Engine returns each candidate with **raw per-modality scores only** (a cosine similarity, a BM25 score, a recency age, an importance label, a source tag). It does *not* pre-blend them. The Assembler owns the **single weight vocabulary** that fuses these into one comparable relevance scalar. There is exactly one such vocabulary in the entire system; no other component may invent a second.

### Signals and canonical weights

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class RankWeights:
    """The ONE weight vocabulary (C1). Lives in TenantPolicy; these are defaults."""
    vector:     float = 0.40   # dense semantic similarity (cosine, normalized)
    lexical:    float = 0.20   # BM25 / sparse exact-term overlap
    recency:    float = 0.15   # memory-decay freshness (orthogonal to placement)
    importance: float = 0.15   # write-time / consolidation-assigned salience
    source:     float = 0.10   # provenance trust prior (pinned > curated > scraped)
    # Invariant: weights sum to 1.0 (validated at policy load; fail-closed otherwise).

    def validate(self) -> None:
        total = self.vector + self.lexical + self.recency + self.importance + self.source
        if abs(total - 1.0) > 1e-6:
            raise PolicyError(f"RankWeights must sum to 1.0, got {total:.6f}")
```

| Signal | Source | Raw form (from Memory Engine) | Normalization in Assembler | Default weight |
| --- | --- | --- | --- | --- |
| `vector` | pgvector HNSW ANN cosine (or opt-in cross-encoder rewrite) | cosine in `[-1, 1]` | `(cos + 1) / 2` -> `[0, 1]` | **0.40** |
| `lexical` | BM25 (Postgres FTS / in-process) | unbounded `>= 0` | min-max over the candidate set | **0.20** |
| `recency` | memory-decay age | RFC-3339 UTC age in seconds | `exp(-age / half_life)` | **0.15** |
| `importance` | write/consolidation salience | ordinal `{0..3}` | `level / 3` | **0.10–0.15** |
| `source` | provenance tag | categorical | fixed prior table (below) | **0.10** |

**Why these weights.** Vector dominates (0.40) because dense semantic match is the strongest single predictor of usefulness on mixed workloads, but we cap it below 0.5 so that a confident-but-off-topic embedding cannot bulldoze an exact lexical hit. Lexical at 0.20 rescues acronyms, IDs (ULIDs!), and rare tokens that BGE embeddings smear. Recency and importance at 0.15 each let fresh, salient memories surface without letting a five-minute-old throwaway outrank a curated fact. Source at 0.10 is a tie-breaker prior, never a primary driver. These are tenant-overridable but the **vocabulary is fixed** — a tenant may re-weight, never add a sixth axis.

**Rejected alternative — learned/LTR fusion (e.g., LambdaMART, a learned reranker as the primary scorer).** A learned-to-rank model would likely beat a linear blend on NDCG. We reject it as the *default* path for three reasons: (1) it requires per-tenant labeled relevance judgments we do not have at launch; (2) it is opaque, which directly breaks the flagship **Replay Debugger** — a linear blend with named weights replays byte-exactly and is human-auditable, a gradient-boosted ensemble is a black box in the replay bundle; (3) inference cost pushes us toward the `<50ms` wall. LTR is available *only* as the opt-in out-of-band cross-encoder that rewrites the `vector` signal — never as the fusion authority.

**Rejected alternative — Reciprocal Rank Fusion (RRF) inside the Assembler.** RRF is excellent and we *do* use it — but **upstream, in the Memory Engine**, to fuse pgvector-ANN and BM25 candidate *lists* into the single ranked candidate set (`RRF+rescore 6ms` in the retrieval budget). Using RRF *again* in the Assembler would double-fuse and discard the raw per-modality scores C1 requires us to preserve. The boundary is sharp: **RRF fuses lists for retrieval; the weight vocabulary fuses scores for final ranking.** Memory-decay recency (a retrieval-side freshness signal) is **orthogonal** to the Assembler's lost-in-the-middle *placement* (C1) — recency affects *whether* a candidate ranks; placement affects *where* a surviving candidate sits in the prompt.

### Source provenance prior

```python
SOURCE_PRIOR: dict[str, float] = {
    "pinned":     1.00,   # tenant-pinned canonical fact
    "curated":    0.85,   # human-reviewed knowledge base
    "tool":       0.70,   # tool / API call result this turn
    "memory":     0.60,   # consolidated long-term memory
    "scraped":    0.40,   # ingested web / doc, unreviewed
}  # unknown source -> 0.30 (fail-low, never fail-high)
```

### Blended score

```python
def blended_score(c: Candidate, w: RankWeights, norm: NormContext) -> float:
    s_vec  = (c.cosine + 1.0) / 2.0
    s_lex  = norm.minmax_lexical(c.bm25)
    s_rec  = math.exp(-c.age_seconds / norm.recency_half_life_s)   # decay, not placement
    s_imp  = c.importance_level / 3.0
    s_src  = SOURCE_PRIOR.get(c.source, 0.30)
    return (w.vector*s_vec + w.lexical*s_lex + w.recency*s_rec
            + w.importance*s_imp + w.source*s_src)   # in [0, 1]
```

## 2.2.3 MMR diversity

Raw relevance ranking produces redundancy: five candidates that all paraphrase the same fact crowd out coverage. The Assembler applies **Maximal Marginal Relevance** over the top-relevance survivors to trade relevance against diversity.

```
MMR(c) = lambda * rel(c) - (1 - lambda) * max_{s in Selected} sim(c, s)
```

- **`lambda = 0.7`** (default; tenant-tunable in `[0.5, 0.9]`). At 0.7, relevance dominates but a candidate that is near-duplicate (`sim ~ 1.0`) to something already selected is penalized hard enough to be skipped in favor of complementary coverage.
- `rel(c)` is the **blended score** from 2.2.2 (one vocabulary — MMR consumes the blended scalar, it does not re-fuse).
- `sim(c, s)` is **cosine over the already-available 384-dim BGE embeddings** — zero extra inference. We deliberately reuse the retrieval embeddings rather than recompute, keeping MMR inside the latency budget.
- MMR runs over at most the top `K = min(128, len(candidates))` by blended score, not the full 512. Beyond ~128 the marginal diversity gain is noise and the `O(K * |Selected|)` cost is wasted. This bound is what keeps MMR's worst case at a few hundred microseconds.

**Why `lambda = 0.7` and not lower.** A document-search system might use 0.5 for breadth. ContextOS feeds an LLM under a tight token budget, where a slightly-redundant-but-highly-relevant chunk beats a diverse-but-tangential one — the model can dedupe, it cannot conjure missing facts. We bias toward relevance.

**Rejected alternative — clustering-based diversity (e.g., k-means over candidate embeddings, pick cluster representatives).** Clustering gives cleaner diversity but (1) needs a `k` we cannot principle a priori per query, (2) is non-deterministic under standard k-means init, which **breaks byte-exact replay**, and (3) costs more than greedy MMR. MMR is deterministic given fixed input order (we sort candidates by `(blended_score, ulid)` before the greedy loop, so ties break on ULID — fully reproducible in the replay bundle).

## 2.2.4 Hierarchical token-budget allocation

The Assembler partitions the model's input context window into **hierarchical budgets** with **hard reserves** that can never be evicted, then knapsack-packs the remainder.

### Budget tiers and reserves

```python
@dataclass
class TokenBudget:
    window_tokens: int            # model context window (from router-selected model, C3)
    output_reserve: int           # tokens reserved for the completion (hard)
    system_reserve: int           # system prompt — HARD, never evicted
    latest_user_reserve: int      # latest user turn — HARD, never evicted

    # Soft, knapsack-allocated pools (fractions of the remaining budget):
    memory_frac:   float = 0.30   # consolidated/episodic memory
    docs_frac:     float = 0.45   # retrieved documents (the candidates)
    history_frac:  float = 0.25   # prior conversation turns
    # memory_frac + docs_frac + history_frac == 1.0 over the SOFT remainder

    def soft_remainder(self) -> int:
        used = self.output_reserve + self.system_reserve + self.latest_user_reserve
        return self.window_tokens - used
```

**The two hard reserves are inviolable (the core fail-closed rule):**

1. **System prompt** — reserved before anything else. The model's behavior contract is not negotiable against retrieved chunks.
2. **Latest user turn** — the actual question. A system that drops or truncates the user's current message to fit more retrieved context is broken; we reserve it whole.

If `system_reserve + latest_user_reserve + output_reserve > window_tokens`, the request **cannot be satisfied** and we **fail closed with HTTP 413** *before* spending a single token on retrieval packing. We never silently truncate a hard reserve.

```python
class ContextOverflow(Exception):
    """Maps to HTTP 413 Payload Too Large. Fail-closed."""
    http_status = 413

def check_hard_reserves(b: TokenBudget) -> None:
    floor = b.output_reserve + b.system_reserve + b.latest_user_reserve
    if floor > b.window_tokens:
        raise ContextOverflow(
            f"hard reserves ({floor} tok) exceed window ({b.window_tokens} tok); "
            f"reduce system prompt or route to a larger-context model"
        )
    if b.soft_remainder() <= 0:
        raise ContextOverflow("no budget left for memory/docs/history after reserves")
```

The soft pools (memory 30% / docs 45% / history 25% of the *remainder*) are starting allocations. They are **elastic with borrowing**: if `history` underfills, its slack flows to `docs` and `memory` proportionally (and vice-versa), so we never strand budget. The knapsack (2.2.6) does this borrowing as a single combined pack against the full soft remainder, with per-pool *caps* rather than rigid partitions — this is strictly better than rigid partitions, which waste budget when one pool is sparse.

**Rejected alternative — flat single-budget packing (no tiers).** Simpler, but it lets a flood of high-scoring retrieved docs starve conversation history, destroying multi-turn coherence — or vice-versa. The tiered floors guarantee each *kind* of context gets a fair minimum while still allowing borrowing. **Rejected alternative — fixed rigid partitions (no borrowing).** Wastes the most precious resource (context tokens) whenever a pool is under-supplied. Elastic caps dominate both.

## 2.2.5 Tokenizer truth (C3) — model chosen before final packing

Token counts are **tokenizer-specific**. Packing against the wrong tokenizer either overflows the model (rejected request, wasted round-trip) or under-fills (wasted budget, worse answers). The pipeline invariant places routing *after* assembly, which appears to create a chicken-and-egg problem. We resolve it explicitly (**C3**):

> **The router selects the model BEFORE final packing**, so the Assembler enforces the hard reserve with the *correct* tokenizer. If the model is not yet knowable, the Assembler packs against a **conservative max-tokenization estimate plus a documented margin**, then **re-validates post-route** — re-packing or failing closed (413) if the real tokenizer disagrees.

Mechanically, routing has two touch-points around assembly:

1. **Pre-pack model hint (the normal path).** Difficulty/utility routing (Section on Routing) produces a *candidate model* cheaply from the query *before* the heavy packing work. The Assembler loads that model's exact tokenizer and packs precisely. This is the path for `>95%` of requests; the `Model routing` 5 ms stage's difficulty signal is computed early.
2. **Conservative-estimate fallback.** If the model genuinely cannot be known pre-pack (e.g., routing depends on the *assembled* token count, a rare cycle), the Assembler packs against the **maximum bytes-per-token-inverse** across the allowlisted model pool — i.e., it assumes the *most token-greedy* tokenizer — and applies a **+8% safety margin** (the single canonical C3 margin used by the compressor, router, and API surfaces):

```python
CONSERVATIVE_MARGIN = 0.075   # documented; tuned so re-pack rate stays < 1%

def estimate_tokens_conservative(text: str, pool: list[ModelSpec]) -> int:
    # worst-case (highest token count) tokenizer over the candidate pool
    worst = max(m.tokens_per_char for m in pool)
    return math.ceil(len(text) * worst * (1.0 + CONSERVATIVE_MARGIN))
```

After routing resolves the real model, the Assembler runs a **post-route re-validation**:

```python
def revalidate_post_route(packed: AssembledContext, model: ModelSpec) -> AssembledContext:
    real = model.tokenizer.count(packed.render())
    if real <= packed.budget.window_tokens - packed.budget.output_reserve:
        return packed                      # conservative estimate held; ship it
    # estimate was too optimistic -> re-pack against the real tokenizer
    repacked = pack(packed.sources, packed.budget, tokenizer=model.tokenizer)
    if repacked is None:                   # cannot fit even the hard reserves
        raise ContextOverflow("post-route re-validation: hard reserve overflow")
    return repacked
```

Because the fallback estimate is **always conservative (over-counts)**, re-validation can only ever find we have *more* room than we packed — never less for the soft pools — and the hard-reserve check already passed against the worst case. The re-pack path therefore fires `<1%` of the time and only to *reclaim* slack, keeping us inside `<50ms` p95.

**Rejected alternative — a universal "average" token estimator (e.g., chars/4).** Convenient and wrong in the unsafe direction: it under-counts for code-heavy or CJK content and produces 413s *from the backend* after we've already spent the retrieval and network budget. Conservative-worst-case + re-validate fails closed *locally and early*, which is the only acceptable failure mode for a budget authority. **Rejected alternative — always pack to the smallest-window model in the pool.** Safe but throws away the entire point of routing to large-context models; wastes budget on every request that routes large.

## 2.2.6 Edge-loading placement — fighting lost-in-the-middle

LLMs attend most reliably to the **head and tail** of their context and degrade in the middle ("lost-in-the-middle"). After ranking and packing decide *which* candidates survive, the Assembler decides *where* each sits. **Placement is orthogonal to recency** (C1): recency already influenced *survival* via the blend; placement is purely about attention geometry.

**Edge-loading rule:** sort survivors by final relevance, then place the **highest-relevance items at the head and tail**, weakest in the middle — a "valley" layout.

```python
def edge_load(survivors: list[Packed]) -> list[Packed]:
    """Highest relevance at both edges, weakest buried in the middle."""
    ranked = sorted(survivors, key=lambda p: (-p.score, p.ulid))  # deterministic
    head, tail = [], []
    for i, item in enumerate(ranked):
        (head if i % 2 == 0 else tail).append(item)  # 0->head, 1->tail, 2->head...
    return head + list(reversed(tail))   # strongest at both ends, weakest in middle
```

Fixed structural elements bracket this valley: the **system prompt is always first**, the **latest user turn is always last** (immediately before generation, maximal recency-of-attention). Conversation history and retrieved docs fill the valley with the edge-loaded ordering.

```
[ SYSTEM PROMPT (hard) ]
[ strongest doc/mem ] [ 3rd ] [ 5th ] ... [ weakest ] ... [ 6th ] [ 4th ] [ 2nd strongest ]
[ recent conversation history ]
[ LATEST USER TURN (hard) ]
```

**Rejected alternative — pure descending relevance (rank order, strongest-first only).** This is the naive default and it dumps every weak item into the tail-middle region where the model ignores them *and* leaves the literal tail (highest attention) occupied by the weakest chunk. Edge-loading deliberately spends both high-attention edges on high-relevance content. **Rejected alternative — relevance-agnostic chronological order.** Coherent for pure chat, but for retrieval-augmented prompts it buries the single most relevant fact in the middle — the exact failure mode we are engineering against.

## 2.2.7 Public interface

```python
async def assemble(
    budget: TokenBudget,
    sources: ContextSources,
    weights: RankWeights,
    tokenizer: Tokenizer | None,   # None -> conservative estimate path (C3)
    mmr_lambda: float = 0.70,
) -> AssembledContext:
    """
    SOLE final-ranking + budget authority (C1).
    Pure CPU, stateless, < 50 ms p95 over <= 512 candidates.
    Raises ContextOverflow (-> HTTP 413) on hard-reserve overflow (fail-closed).
    """
```

```python
@dataclass(frozen=True)
class ContextSources:
    system_prompt: str                 # -> hard reserve
    latest_user_turn: str              # -> hard reserve
    candidates: list[Candidate]        # <= 512, ALREADY ACL-filtered + compressed
    history: list[Turn]                # prior conversation, newest-last

@dataclass(frozen=True)
class AssembledContext:
    messages: list[Message]            # final ordered prompt (edge-loaded)
    used_tokens: int
    budget: TokenBudget
    decisions: AssemblyDecisionLog     # per-candidate: score breakdown, MMR, place, evict
    sources: ContextSources            # retained for post-route re-pack (C3)
    def render(self) -> str: ...
```

`AssemblyDecisionLog` records, per candidate, the five raw signals, the blended score, the MMR marginal, the placement index, and the keep/evict verdict. This log is the Assembler's contribution to the **content-addressed, per-tenant-encrypted replay bundle** — it is what makes every assembly decision byte-exactly replayable. Because every step (sort, MMR, knapsack, edge-load) is deterministic given identical inputs and ULID tie-breaks, replaying `assemble()` on the recorded inputs reproduces `messages` byte-for-byte (consistent with the deterministic-stage guarantee in the replay contract, C7).

## 2.2.8 Ranking + knapsack pseudocode

```python
def assemble_impl(budget, sources, weights, tokenizer, mmr_lambda):
    # 0. FAIL-CLOSED: hard reserves must fit before any work (C3, 413).
    check_hard_reserves(budget)
    tok = tokenizer or ConservativeTokenizer(budget.model_pool, CONSERVATIVE_MARGIN)

    # 1. BLEND: one weight vocabulary over raw per-modality scores (C1).
    norm = NormContext.from_candidates(sources.candidates)
    scored = [(c, blended_score(c, weights, norm)) for c in sources.candidates]
    # deterministic order: blended score desc, then ULID asc (replay-stable)
    scored.sort(key=lambda cs: (-cs[1], cs[0].ulid))

    # 2. MMR: diversify the top-K survivors (K <= 128) against selected set.
    pool = scored[:min(128, len(scored))]
    selected: list[ScoredCandidate] = []
    while pool:
        best, best_mmr = None, -math.inf
        for c, rel in pool:
            redundancy = max((cosine(c.embedding, s.embedding) for s in selected),
                             default=0.0)
            mmr = mmr_lambda * rel - (1 - mmr_lambda) * redundancy
            if mmr > best_mmr:
                best, best_mmr = (c, rel), mmr
        selected.append(ScoredCandidate(*best, mmr=best_mmr))
        pool.remove(best)

    # 3. KNAPSACK: greedy density pack into the elastic soft remainder.
    #    Per-pool caps (memory/docs/history) with borrowing; NOT rigid partitions.
    remainder = budget.soft_remainder()
    caps = {"memory":  int(remainder * budget.memory_frac),
            "docs":    int(remainder * budget.docs_frac),
            "history": int(remainder * budget.history_frac)}
    used = {"memory": 0, "docs": 0, "history": 0}
    total_used, packed = 0, []

    #    Order by VALUE DENSITY = score / token_cost (0/1-knapsack greedy approx).
    #    Optimal 0/1 knapsack is O(n*W) DP — overkill and slower than the SLO
    #    allows; greedy-by-density is within a bounded factor of optimal for
    #    homogeneous chunk sizes and stays well inside < 50 ms.
    items = [ScoredItem(sc, tok.count(sc.text), pool_of(sc)) for sc in selected]
    items += [history_item(t, tok) for t in sources.history]
    items.sort(key=lambda it: it.score / max(it.tokens, 1), reverse=True)

    for it in items:
        if total_used + it.tokens > remainder:
            continue                                 # global budget hit -> skip
        pool = it.pool
        # borrowing: a pool may exceed its cap if global remainder allows AND
        # no contended pool is currently starved below 50% of its own cap.
        if used[pool] + it.tokens > caps[pool] and _pool_is_contended(used, caps):
            continue
        packed.append(it); used[pool] += it.tokens; total_used += it.tokens

    # 4. EDGE-LOAD: strongest at head and tail; system first, user last.
    body = edge_load(packed)
    messages = (
        [Message("system", sources.system_prompt)]
        + [it.to_message() for it in body]
        + [Message("user", sources.latest_user_turn)]
    )
    return AssembledContext(messages, total_used + budget.system_reserve
                            + budget.latest_user_reserve, budget,
                            AssemblyDecisionLog.build(scored, selected, packed, body),
                            sources)
```

**Why greedy-by-density and not exact 0/1-knapsack DP.** Exact DP is `O(n * W)` in tokens `W` (tens of thousands) — that alone can blow the 50 ms budget at p95 under load, and it buys little: our chunks are size-homogeneous after compression, where greedy-by-value-density is provably within a small bounded factor of optimal. We rejected DP for latency. We also rejected **pure score-ordering (ignore token cost)** — it over-pays for one giant high-score chunk and starves three nearly-as-good small ones; density ordering fixes exactly that. The greedy pass plus borrowing-aware caps is deterministic and replay-stable.

## 2.2.9 Latency accounting — staying under 50 ms p95

| Sub-step | Work over `<=512` candidates | Approx p95 budget |
| --- | --- | --- |
| Blend + normalize + sort | 512 linear scores, one sort | ~3 ms |
| MMR (K`<=128`, cosine over 384-dim) | `O(K * |Selected|)` cosine | ~8 ms |
| Tokenize survivors + knapsack | token count + greedy pass | ~12 ms |
| Edge-load + render messages | sort + interleave | ~2 ms |
| Headroom (GC, contention, post-route re-pack `<1%`) | — | ~25 ms |
| **Total context assembly** | | **< 50 ms p95** |

This stage contributes its full 50 ms to the **`<250ms` total ContextOS control overhead** (Section 9), and is the *only* line that also counts toward the `<50ms` assembly SLO. It overlaps with retrieval and cache stages on the wire; the `<250ms` figure is the **critical-path p95, not a naive sum**.

## 2.2.10 PROVISIONAL Rust ContextAssembler (C14)

The Assembler is the single most CPU-intensive hot-path component (`<=512`-candidate scoring, `O(K^2)`-ish MMR, knapsack, all per request). It is therefore the **prime candidate** for the conditional Rust/PyO3 hot-path kernel gated by **ADR-0001**. We mark the Rust interface **PROVISIONAL** and define the gate precisely so the decision is data-driven, not aspirational.

```rust
// PROVISIONAL — gated by ADR-0001 benchmark. Subject to change. Not yet built.
// Mirrors the Python assemble() contract exactly; PyO3-exposed.
pub fn assemble(
    budget: &TokenBudget,
    sources: &ContextSources,   // <= 512 candidates, 384-dim f32 embeddings
    weights: &RankWeights,
    mmr_lambda: f32,
) -> Result<AssembledContext, ContextOverflow>;
```

### The benchmark gate (C14)

| Gate parameter | Definition |
| --- | --- |
| **Workload** | Candidate-count distribution matching production: p50 = 180, p95 = 512 candidates; **384-dim** BGE embeddings; mixed doc/memory/history sources. |
| **Target throughput** | **8,000 req/s sustained per 8-vCPU node** of assembly-only work (matching the gateway throughput target; canonical gate load per ADR-0001 / NFR §). |
| **Trigger threshold** | Built **only if** the **pure-Python** `assemble()` p95 is **`>=40 ms`** under that load. **`35–40 ms` is a watch band** — do *not* port; optimize Python (object pooling, slotted dataclasses, vectorized scoring) first. **`<=35 ms` holds** Python as canonical. (At `>=40 ms` the stage consumes `>=80%` of the 50 ms SLO, leaving `<10 ms` for GC/contention/re-pack.) |
| **Headroom requirement** | Rust must restore **`>=15 ms` p95 headroom** *and* not regress the deterministic replay guarantee (bit-identical `messages` output vs. the Python reference, validated by a cross-implementation differential test). |
| **Owner** | The Hot-Path Kernel owner (per ADR-0001). |

Until the gate fires, the **Python asyncio + uvloop** implementation is canonical and ships. **Rejected alternative — build Rust speculatively now.** Premature: it doubles the maintenance surface (two implementations to keep replay-identical) before we have evidence Python misses the budget. The `<=512` cap was chosen specifically so that *pure Python has a real chance* of holding `<50ms`; we let the benchmark decide. **Rejected alternative — Cython / Numba instead of Rust.** Numba's JIT warm-up and Cython's weaker memory-safety story lose to PyO3's clean FFI boundary and deterministic, no-warm-up performance — and PyO3 is already the locked choice for the hot-path kernel.

## 2.2.11 Cross-section contract summary

- **C1:** Memory Engine returns raw per-modality scores; this Assembler is the sole final-ranking + budget authority with one weight vocabulary; recency (decay) is orthogonal to edge placement.
- **C3:** Router supplies the model (and thus tokenizer) *before* final packing; absent that, pack conservative-worst-case **+8% margin** and re-validate post-route, re-pack or 413.
- **C7:** `assemble()` is a fully deterministic replay stage; `AssemblyDecisionLog` feeds the replay bundle; identical inputs reproduce `messages` byte-for-byte.
- **C13/C14:** scope-boundary invariants (no index, `<=512` cap, opt-in out-of-band cross-encoder) hold; Rust kernel PROVISIONAL behind the ADR-0001 benchmark gate above.
