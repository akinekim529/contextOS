# 2.6 Model Router

The Model Router is the component that decides **which backend model serves a
given request**. It sits at the `routing` stage of the pipeline ordering
invariant:

```
auth/tenant -> cache lookup -> retrieve candidates -> ACL/redaction
            -> compression -> assembly/packing -> ROUTING -> adapter
            -> stream -> async write-back
```

Routing runs **after** assembly/packing in the data flow but, per **C3**, the
router *selects the model before final token packing* whenever the model choice
is knowable from request metadata, so the correct tokenizer enforces the hard
context reserve. The two facts are not in tension: the router is invoked as a
pure function over request metadata (difficulty signal, tenant policy, declared
max output, residency tag) that does **not** require the fully packed prompt;
the packer then uses the selected model's tokenizer. When the model is *not*
knowable in advance (e.g. difficulty is content-derived and needs the assembled
context), we pack against a conservative max-tokenization estimate plus a
documented +8% margin and re-validate post-route — re-packing or failing closed
with `413` if the real tokenizer overflows the hard reserve. This section owns
the routing decision; Section 9 owns the latency table, which budgets routing at
**5 ms p95** (`difficulty + utility + breaker`).

The router is a **telemetry reader and policy enforcer**, never a scheduler. Per
the scope-boundary invariants it does **not** own GPUs, does **not** schedule
inference, and does **not** re-execute work. GPU-aware routing consumes a
read-only health/telemetry feed; it can *prefer* or *avoid* a backend pool, but
it never places a workload on a specific device. ContextOS is not an inference
engine.

---

## 1. Responsibilities and non-responsibilities

| The router DOES | The router DOES NOT |
|---|---|
| Resolve the **hard-filtered** candidate set via `check(principal, model, 'route')` | Maintain a second allowlist/residency store (C10) |
| Estimate query **difficulty** (two-stage) | Run the model or own a tokenizer |
| Score survivors by **utility = w_q·quality − w_c·cost − w_l·latency** | Schedule GPUs or place workloads on devices |
| Order a **utility-ranked fallback chain** | Retry indefinitely or hide cost from the ledger |
| Trip/clear **circuit breakers** per backend | Decide *whether* a request is allowed (that is RBAC upstream) |
| **P2C** load-balance across replicas of the chosen pool | Build or own a vector index |
| Emit a `RoutePolicy` decision into the replay bundle | Block on the optimization health store (C9) |

## 2. The decision contract

```python
# All IDs are ULID; all timestamps RFC-3339 UTC.

@dataclass(frozen=True)
class RouteRequest:
    tenant_id: str                 # non-null partition key, on every object
    principal: Principal           # for check(action='route')
    namespace: Namespace           # project/agent/user; HARD filter (C2)
    residency_tag: str             # e.g. "eu", "us"; NEVER bypassed (C9)
    declared_max_output_tokens: int
    required_capabilities: frozenset[str]   # e.g. {"json_mode","vision","fn_call"}
    difficulty_hint: Optional[float]        # caller override in [0,1]; advisory
    assembled_context: Optional[AssembledContext]  # present iff model not pre-knowable
    request_id: str                # ULID, correlation into replay bundle

@dataclass(frozen=True)
class BackendChoice:
    backend_id: str                # concrete pool, e.g. "vllm-llama3-8b-eu"
    model_id: str                  # canonical model identity for tokenizer + cache key
    replica: ReplicaAddr           # P2C-selected endpoint within the pool
    fallback_chain: tuple[str, ...]  # utility-ordered backend_ids to try on failure
    difficulty: float              # [0,1], recorded
    utility: float                 # winning utility score, recorded
    decided_on: Literal["optimized", "static_default"]  # C9 posture flag
    reserve_tokenizer: str         # tokenizer the packer must use (C3)
    decision_ts: str               # RFC-3339 UTC
```

`decided_on` is load-bearing for replay and for SLO accounting: it records
whether the choice came from live optimization signals or from the **static
safe-default ranking** because the health store was unavailable. A reviewer
reading the replay bundle can therefore tell a *cost-optimal* route from a
*degraded-but-correct* route.

## 3. Hard filters first — fail CLOSED (C9 + C10)

The single most important rule of this router: **hard-policy filters evaluate on
static policy and fail closed, independent of the health/optimization store.**
The optimization layer (latency, queue depth, observed quality) is allowed to be
stale, partitioned, or entirely down — and the router still produces a *correct,
compliant* answer, just a possibly suboptimal one.

Hard filters, in order:

1. **Route authorization (C10).** The router calls
   `check(principal, resource=model, action='route')` as the *single authority*
   for both the model allowlist and residency. There is no second policy store;
   `RoutePolicy.allowed_backends` is **derived** from this call. The `Action`
   enum is `{read, write, delete, admin, route, cache_read}`.
2. **Namespace (C2).** Within-tenant namespace (project/agent/user) is a hard
   filter evaluated against `tenant_id`, fail-closed. Missing or ambiguous
   namespace ⇒ **deny**. (Routing never *reads* memory rows, but the namespace
   constrains which model-policy rows apply — e.g. a project pinned to an
   on-prem pool.)
3. **Residency.** The candidate's `residency_tag` must equal the request's.
   **Residency is never bypassed**, including by the safe-default pool.
4. **Capability.** Candidate must advertise every `required_capabilities` flag
   (JSON mode, vision, function-calling, ≥ the declared context window).
5. **Budget.** Static per-tenant budget ledger gate: if the *cheapest* compliant
   candidate's projected cost exceeds remaining budget, fail closed with `402`.
   This evaluates on the ledger's static balance, not on live optimization data.

These five filters require **only** the principal, the static policy snapshot,
and the request metadata. None of them touch the health store. This is the C9
contract made concrete.

```python
def hard_filter(req: RouteRequest, policy: PolicySnapshot) -> list[Candidate]:
    # check(action='route') is the SOLE allowlist+residency authority (C10)
    allowed = policy.check(req.principal, resource="model", action="route")
    if not allowed.granted:
        raise RouteDenied(403, allowed.reason)

    if req.namespace.is_missing_or_ambiguous():   # C2 fail-closed
        raise RouteDenied(403, "namespace missing/ambiguous")

    survivors = []
    for c in allowed.backends:                     # derived from check(), no 2nd store
        if c.residency_tag != req.residency_tag:   # C9: residency NEVER bypassed
            continue
        if not req.required_capabilities <= c.capabilities:
            continue
        if c.min_context_window < req.required_context_tokens:
            continue
        survivors.append(c)

    if not survivors:
        raise RouteDenied(409, "no compliant backend for policy+residency+capability")

    # Static budget gate evaluates on ledger balance, not health store
    if cheapest_projected_cost(survivors, req) > policy.remaining_budget(req.tenant_id):
        raise RouteDenied(402, "tenant budget exhausted for compliant pool")
    return survivors
```

### The safe-default pool also satisfies every hard filter

Per C9 the safe-default fallback pool is **not** an escape hatch from policy. It
is pre-computed per `(tenant_id, residency_tag, capability-class)` as the
*statically cheapest compliant pool* and is itself subject to filters 1–5. There
is no code path in which selecting the default bypasses residency, allowlist, or
capability. If even the default pool is empty after filtering, the router fails
closed (`409`) rather than reaching for a non-compliant backend.

> **Rejected alternative — "degrade to any healthy backend."** A common router
> design, on total health-store failure, falls back to *whatever endpoint
> answers a ping*. We reject this outright: it can route EU-resident traffic to
> a US pool during an outage — the exact moment compliance matters most. Our
> static safe-default pool is the only sanctioned degraded path, and it is
> filter-clean by construction.

## 4. Two-stage difficulty / complexity estimation

The router downgrades **easy** queries to cheap models. This is where 20–40% of
model spend is saved (see §8). Misclassifying a hard query as easy is the
expensive failure mode (bad answer, possible re-ask), so the estimator is
**conservative**: when unsure, it rates *harder*, which routes *up*.

### Stage 1 — cheap heuristic (always on, ships first)

A pure-CPU feature extractor running in well under 1 ms. It produces a
difficulty score in `[0, 1]`:

| Signal | Direction | Rationale |
|---|---|---|
| Prompt token length (post-assembly) | ↑ longer ⇒ harder | Long contexts need stronger long-context models |
| Reasoning/markers (`step by step`, `prove`, `derive`, `why`) | ↑ harder | Explicit reasoning requests |
| Code-fence / structured-output request | ↑ harder | Codegen + strict JSON stress small models |
| Output token budget (`declared_max_output_tokens`) | ↑ harder | Long generations amplify small-model drift |
| Question vs. lookup ratio (`?` density, imperative verbs) | ↓ lookup ⇒ easier | Pure retrieval/formatting is cheap |
| Caller `difficulty_hint` | clamp toward hint | Advisory, never below heuristic floor |

```python
def heuristic_difficulty(req: RouteRequest, packed: PackedPrompt) -> float:
    d  = 0.0
    d += clamp01(packed.prompt_tokens / 8000) * 0.35
    d += 0.20 if REASONING_RE.search(packed.text) else 0.0
    d += 0.15 if packed.requests_structured_output else 0.0
    d += clamp01(req.declared_max_output_tokens / 2000) * 0.20
    d -= 0.10 * lookup_ratio(packed.text)          # pure retrieval is easy
    d  = clamp01(d)
    # Caller hint can only RAISE difficulty (conservative), never lower the floor
    if req.difficulty_hint is not None:
        d = max(d, req.difficulty_hint)
    return d
```

The heuristic alone is sufficient to ship and already captures the bulk of the
"obvious lookup vs. obvious reasoning" split.

### Stage 2 — optional distilled classifier (roadmap dependency)

A small distilled difficulty classifier (a 2-layer MLP head over the *already
computed* `bge-small-en-v1.5` 384-dim query embedding — the same ~6 ms p95
embedding the retrieval path produced, **reused**, zero extra embedding cost)
refines the heuristic score. It runs in-process on CPU, adds < 1 ms, and is
gated behind a config flag.

```python
def difficulty(req, packed, emb: Optional[np.ndarray]) -> float:
    base = heuristic_difficulty(req, packed)
    if not CLASSIFIER_ENABLED or emb is None:
        return base                       # ships first; conservative default
    p = difficulty_head.predict(emb)      # [0,1], distilled
    # Blend, but never route EASIER than the heuristic says (conservative bias)
    return max(base, 0.5 * base + 0.5 * p)
```

> **Labeling pipeline is a roadmap dependency (ships *after* the heuristic).**
> The classifier needs supervised labels — "did the cheap model produce an
> answer of acceptable quality for this query?" Those labels come from an
> offline harness that replays historical requests through both a cheap and a
> reference model and scores agreement (and ties into the offline cache-
> correctness eval harness, also a roadmap item). Until that pipeline produces a
> validated model, **`CLASSIFIER_ENABLED=False` is the shipped default** and the
> router runs on the heuristic alone. The system is fully functional and
> compliant without Stage 2; Stage 2 only sharpens the cost-savings curve. This
> conservative-default fallback is deliberate: an *unvalidated* classifier that
> routes hard queries down is worse than no classifier.

> **Rejected alternative — LLM-as-difficulty-judge on the hot path.** Calling a
> small LLM to rate difficulty per request adds a full inference round-trip
> (tens to hundreds of ms) to a 5 ms routing budget and would make routing cost
> a function of the thing it is trying to save. Rejected. The distilled head is
> the out-of-band-trained, in-process-served compromise.

> **Rejected alternative — fixed per-route model pinning.** Statically pinning
> "endpoint X always uses GPT-class model" ignores per-query difficulty and
> leaves the 20–40% easy-query savings on the table. We keep difficulty
> *content-derived*.

## 5. The utility function

Over the **hard-filtered survivors only**, the router scores each candidate:

```
utility(c) = w_q · quality_norm(c, difficulty)
           − w_c · cost_norm(c, req)
           − w_l · latency_norm(c)
```

All three terms are normalized to `[0, 1]` so the weights `(w_q, w_c, w_l)` are
dimensionless and tenant-tunable. Defaults: `w_q = 0.55, w_c = 0.30,
w_l = 0.15` (quality-leaning, because a wrong cheap answer is the costliest
outcome).

| Term | Source | Failure posture (C9) |
|---|---|---|
| `quality_norm` | Per-model **quality-for-difficulty** curve: a model's expected quality at this difficulty band, from offline eval + online sampled grading | Optimization signal ⇒ **fail-open** to static per-model prior |
| `cost_norm` | `(prompt_tokens·in_price + max_output·out_price)` from the **static** price table, min-max normalized across survivors | **Static** input ⇒ always available |
| `latency_norm` | Observed p95 from the **read-only** health store | Optimization signal ⇒ **fail-open** to static latency prior |

The critical C9 distinction: `cost_norm` is computed from a **static** price
table and is therefore always available even with the health store down.
`quality_norm` and `latency_norm` are **optimization signals**; if the health
store is unavailable they fall back to *static priors* shipped with the model
registry, and the router sets `decided_on="static_default"`. The decision is
never blocked on health-store availability.

The **difficulty score modulates `quality_norm`**, not the weights: a cheap 8B
model has a high `quality_norm` at difficulty 0.2 (so utility favors it — cheap
*and* good enough) and a low `quality_norm` at difficulty 0.85 (so the
expensive model wins despite its cost penalty). This is the mechanism by which
easy queries route down.

```python
def score(c: Candidate, req: RouteRequest, diff: float, health: HealthView) -> float:
    q = c.quality_for_difficulty(diff)                  # static curve, [0,1]
    cost = c.projected_cost(req)                        # STATIC price table
    lat  = health.p95_or_static_prior(c.backend_id)     # fail-open to prior

    q_n = q                                             # already [0,1]
    c_n = minmax(cost, COST_LO, COST_HI)
    l_n = minmax(lat,  LAT_LO,  LAT_HI)
    return W_Q * q_n - W_C * c_n - W_L * l_n
```

> **Rejected alternative — pure cheapest-first.** Always picking the lowest
> `cost_norm` candidate maximizes the easy-query savings but tanks quality on
> hard queries (and triggers expensive re-asks). The utility blend with a
> quality-leaning default weight is the correct trade.

> **Rejected alternative — bandit/RL online router.** A contextual bandit could
> in principle learn the route. We reject it *as the launch mechanism* because
> it is non-replayable (its decision depends on hidden online state), it
> explores into bad routes on live traffic, and it fights the byte-exact replay
> contract (C7): every routing decision must be deterministically reconstructable
> from the recorded inputs. The deterministic utility function over recorded
> difficulty + static price table + snapshotted health view replays exactly. A
> bandit is a *post-launch optimization* whose decisions, if adopted, must be
> recorded as inputs to preserve replay.

## 6. `route()` — full algorithm

```python
def route(req: RouteRequest, policy: PolicySnapshot,
          health: HealthView, breakers: BreakerRegistry) -> BackendChoice:

    # 1. HARD FILTERS — static policy, fail-CLOSED, independent of health (C9/C10)
    survivors = hard_filter(req, policy)

    # 2. Difficulty (two-stage; classifier optional, heuristic always on)
    diff = difficulty(req, req.packed, req.query_embedding)

    # 3. OPTIMIZATION layer — fail-OPEN to static ranking if health degraded (C9)
    if health.degraded():
        ranked = static_default_ranking(survivors, req, diff)   # priors only
        decided_on = "static_default"
    else:
        ranked = sorted(survivors,
                        key=lambda c: score(c, req, diff, health),
                        reverse=True)
        decided_on = "optimized"

    # 4. Drop OPEN-breaker backends from the *head*; they remain in the tail
    #    so the fallback chain still records them (and can probe on HALF_OPEN).
    primary_pool, chain = split_on_breakers(ranked, breakers)
    if primary_pool is None:
        # Every candidate's breaker is OPEN: use the safe-default pool, which
        # is itself filter-clean (residency never bypassed). If IT is open too,
        # fail closed rather than route non-compliant.
        primary_pool = policy.safe_default_pool(req)            # C9: filter-clean
        if primary_pool is None or breakers.is_open(primary_pool.backend_id):
            raise RouteUnavailable(503, "all compliant backends unavailable")
        decided_on = "static_default"

    # 5. P2C replica selection within the chosen pool
    replica = power_of_two_choices(primary_pool, health)

    return BackendChoice(
        backend_id=primary_pool.backend_id,
        model_id=primary_pool.model_id,
        replica=replica,
        fallback_chain=tuple(c.backend_id for c in chain),
        difficulty=diff,
        utility=primary_pool.utility,
        decided_on=decided_on,
        reserve_tokenizer=primary_pool.tokenizer_id,            # C3: packer uses this
        decision_ts=now_rfc3339_utc(),
    )
```

Step 3 is the explicit C9 fork: optimization signals (`health.degraded()`) cause
a **fail-open to static ranking**, never a denial. Hard filters in step 1
already guaranteed compliance, so the degraded path is *suboptimal-but-correct*.

## 7. Resilience: breakers, fallback chains, P2C, health, failover

### 7.1 Circuit breakers (per backend pool)

Three-state breaker per `backend_id`:

| State | Behavior | Transition |
|---|---|---|
| `CLOSED` | Traffic flows; count failures | `→ OPEN` when failure-rate > 50% over the last 20 requests **or** 5 consecutive failures |
| `OPEN` | Pool removed from primary head; requests skip to next in fallback chain | `→ HALF_OPEN` after `cooldown = 5 s` (jittered ±20%) |
| `HALF_OPEN` | Allow **1** probe request | 1 success ⇒ `CLOSED`; 1 failure ⇒ `OPEN`, cooldown doubles (cap 60 s) |

"Failure" = transport error, `5xx`, timeout exceeding the adapter deadline, or
malformed stream. A `429` is **not** a hard failure — it feeds P2C load
shedding (§7.3) and only trips the breaker if sustained.

> **Rejected alternative — global single breaker.** One breaker across all
> backends would trip the *entire* router when one pool degrades. Per-pool
> breakers isolate blast radius; a sick 70B pool must not stop easy queries from
> reaching the healthy 8B pool.

### 7.2 Utility-ordered fallback chain

The fallback chain is the **utility-ranked tail** from step 3, *minus*
open-breaker pools at the head, *plus* the safe-default pool appended last. On
adapter failure, the dispatcher walks the chain in order. Every chain hop is
recorded in the replay bundle, so a replay shows *why* request R landed on
backend #3 (breaker open on #1, timeout on #2).

```
chain example for an EASY query (difficulty 0.18):
  1. llama3-8b-eu      (utility 0.71, CLOSED)  <- chosen
  2. mistral-7b-eu     (utility 0.66, CLOSED)
  3. llama3-70b-eu     (utility 0.41, CLOSED)  # higher quality, worse utility for easy
  4. safe-default-eu   (utility n/a, filter-clean)  # always last, residency-clean
```

Note the chain **never** leaves the `eu` residency class — every hop is a hard-
filter survivor. Failover does not relax compliance.

**Retry budget.** At most `len(chain)` hops, capped at 3, with a total wall-clock
deadline equal to the request's remaining budget. Each *attempted* hop that
reached the backend and incurred cost is attributed to the budget ledger (C8
partial-cost rules apply to a hop that emitted tokens then failed mid-stream).

### 7.3 P2C (power-of-two-choices) load balancing

Within the chosen pool's replicas, sample **two** replicas at random and send to
the one with lower in-flight load (a composite of outstanding requests + most
recent queue-depth telemetry from the read-only health feed). P2C gives
near-optimal load distribution with O(1) state and no global coordinator.

```python
def power_of_two_choices(pool: Pool, health: HealthView) -> ReplicaAddr:
    a, b = random.sample(pool.replicas, k=min(2, len(pool.replicas)))
    return min(a, b, key=lambda r: health.inflight_load(r))  # lower load wins
```

> **Rejected alternative — round-robin.** Round-robin ignores heterogeneous
> replica load (a replica stuck on a long generation keeps getting traffic).
> **Rejected alternative — least-connections with a global counter.** Requires
> shared mutable state and a coordination round-trip per request, blowing the
> 5 ms routing budget under load. P2C is the standard, coordination-free answer.

### 7.4 Health checks + automatic failover

- **Passive health** (primary signal): every adapter outcome (latency, status,
  stream integrity) updates the in-process health view and feeds breakers. This
  is free — it rides on real traffic.
- **Active health** (liveness floor): a lightweight async probe per pool every
  `2 s` confirms the pool answers a trivial request; a dead pool's breaker is
  forced `OPEN` even with no live traffic flowing to it.
- **Automatic failover** is the composition of the above: breaker `OPEN` →
  fallback-chain advance → P2C on the survivor. No human in the loop, no
  scheduler — the router only *chooses among* backends; it never starts or
  drains them.

The health store is an **optimization** input (C12 best-effort, sampled, fail-
open). Its unavailability degrades us to static ranking (§6 step 3), it never
blocks routing.

## 8. Cost savings (quantified)

The router is the **model-routing-downgrade** lever of the 40–65% combined
token-cost savings target. Per the canonical figures, routing downgrade
contributes **20–40% of model spend on easy queries** (the other levers:
caching 15–30%, compression 10–25% prompt-token reduction — owned by other
sections).

How the 20–40% arises:

| Difficulty band | Share of mixed workload (typical) | Routed to | Relative model $/req |
|---|---|---|---|
| Easy (≤ 0.30) — lookups, formatting, short Q&A | ~45% | small (8B-class) | ~0.15× of flagship |
| Medium (0.30–0.65) | ~35% | mid (mixtral/70B-class) | ~0.45× of flagship |
| Hard (> 0.65) — reasoning, long-context, codegen | ~20% | flagship | 1.0× |

Routing the ~45% easy band from flagship down to an 8B-class model at ~0.15×
cost removes the dominant cost mass on those queries. Even with the medium band
held at mid-tier, the blended model spend lands **20–40% below an
all-flagship baseline** — the realistic range depends on the workload's easy-
band share. Savings are **measured and recorded** per request into the budget
ledger (chosen `model_id` × token counts × static price), so the claim is
auditable in replay, not asserted.

The conservative difficulty bias (§4) is deliberately tuned to keep the *hard*
band from being under-served: it is cheaper to occasionally over-route a medium
query to flagship than to under-route a hard query and pay for a re-ask plus a
bad user experience.

## 9. Replay & observability hooks

Every `route()` decision emits a deterministic record into the per-tenant
content-addressed replay bundle (C7): the **inputs** (difficulty score, the
static price snapshot hash, the health-view snapshot or its absence, the policy
snapshot version) and the **output** (`BackendChoice` incl. `decided_on` and the
full `fallback_chain`). Because all inputs are recorded, the routing *decision*
is byte-exactly reproducible — `route()` is a pure deterministic stage. Only
`backend.invoke` downstream is non-deterministic; the router's choice is not.

Cost records for each attempted hop are **billing-grade** and go through the
fail-closed durable outbox (C12); routing *trace* spans (chain walk, breaker
transitions, P2C pick) are best-effort, tail-sampled (1–10%, force-keep on error
or cost > $0.05/req).

---

## 10. Latency

Routing is budgeted at **5 ms p95** (Section 9, "Model routing — difficulty +
utility + breaker"). Breakdown within that budget: difficulty heuristic
(< 1 ms, embedding reused from retrieval), utility scoring over ≤ a few dozen
survivors (< 1 ms), breaker check + P2C sample (< 1 ms), with headroom. The
optional distilled classifier adds < 1 ms and stays inside budget. The router
adds no I/O on the hot path — policy, prices, and the health view are all
in-process snapshots. This 5 ms sits inside the **< 250 ms p95** total control
overhead and is **not** part of the **< 50 ms p95** assembly budget.

## 11. Summary of applied resolutions

- **C9** — Hard filters (route-auth, namespace, residency, capability, budget)
  evaluate on **static** policy and fail **closed**, independent of the health
  store. Only quality/latency optimization signals fail **open** to static
  ranking (`decided_on="static_default"`). The safe-default pool is itself
  filter-clean; **residency is never bypassed**, including on total degradation.
- **C10** — `check(principal, model, action='route')` is the **single**
  allowlist + residency authority; `RoutePolicy.allowed_backends` derives from
  it. `Action ∈ {read, write, delete, admin, route, cache_read}`.
- **C3** — Router selects the model before final packing; `reserve_tokenizer` is
  handed to the packer; conservative max-tokenization estimate + 8% margin with
  post-route re-validation (re-pack or `413`) when the model is not pre-knowable.
- **C2** — Namespace is a hard, fail-closed filter; missing/ambiguous ⇒ deny.
- **C8** — Per-hop partial-cost attribution to the budget ledger.
- **C12** — Billing-grade cost via durable outbox; routing traces best-effort,
  tail-sampled.
- Scope invariants — router is a GPU/health **telemetry reader**, never a
  scheduler; ≤ 512 candidate cap is a retrieval concern, not a routing one.
