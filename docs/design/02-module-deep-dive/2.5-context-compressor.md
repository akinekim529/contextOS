# 2.5 Context Compressor

The Context Compressor is the stage that turns "we retrieved the right things" into "we can afford to send them." It sits in a fixed slot in the request pipeline and its single job is to shrink the token footprint of individual context blocks while provably preserving the facts those blocks carry. It is **not** a summarizer-for-humans, not a retriever, not a ranker, and not an index. It receives already-selected, already-redacted blocks and returns smaller blocks plus a machine-checkable retention guarantee.

This section defines the three-tier compression strategy, the exact decision rules for which tier runs, the `compress()` contract, the NLI-based fact-retention guardrail and its fail-safe behavior, the cost accounting for the abstractive tier, and the build phasing.

---

## 2.5.1 Position in the pipeline (the invariant)

ContextOS runs a strict, ordered pipeline. The Compressor's position is non-negotiable:

```
auth/tenant
  -> cache lookup
  -> retrieve candidates              (Memory Engine; <= 512 candidates)
  -> ACL / redaction                  (drops/masks anything the principal may not see)
  -> COMPRESSION                       <-- THIS STAGE
  -> assembly / packing               (score + MMR + budget-knapsack)
  -> routing
  -> adapter
  -> stream
  -> async write-back
```

> **PIPELINE INVARIANT: Compression ALWAYS runs AFTER ACL/redaction, and BEFORE assembly/packing.**

Both halves of that invariant are load-bearing.

**Why after ACL/redaction.** Compression rewrites text. An abstractive rewrite of a block that still contains a redacted salary figure, a sibling tenant's customer name, or PII that the redactor would have masked can *re-encode* that secret into a paraphrase that no downstream string-level redactor will catch. By forcing compression strictly downstream of redaction, every byte the compressor ever sees is already cleared for this principal in this tenant/namespace. The compressor therefore inherits the ACL decision; it never re-derives it and never sees a token the principal isn't entitled to. Putting compression first — which a naive "shrink early to do less redaction work" design would do — is **rejected**: it trades a measurable CPU saving for an unbounded data-exfiltration surface, and redaction over compressed/paraphrased text is strictly harder and less reliable than over the original.

**Why before assembly.** The Context Assembler is the *sole* final-ranking and budget-packing authority (resolution C1). Its knapsack only makes good decisions if the "cost" (token count) of each block reflects what will actually be sent. If we compressed *after* packing, the assembler would pack against pre-compression sizes, then compression would change the sizes underneath it, invalidating the knapsack solution and the hard token reserve. So the compressor's output — including the *post-compression* token count under the eventual model's tokenizer — must be final before the assembler runs.

This creates a tokenizer-ordering subtlety addressed in §2.5.6.

## 2.5.2 Scope boundaries (C13) — what the Compressor is forbidden from doing

The Compressor is a hot-path, in-process transform over a bounded candidate set. It explicitly honors the ContextOS scope-boundary invariants:

| Invariant | How the Compressor complies |
| --- | --- |
| **In-process scoring over pre-retrieved candidates only** | The structural and extractive tiers score sentences/spans *within a single already-retrieved block*. They never issue a retrieval, never touch pgvector/Redis for candidate discovery, never build or own an index. Input is `<= 512` candidate blocks, already in memory. |
| **Never builds/owns an index** | Extractive sentence ranking uses an in-RAM TF-IDF / embedding-similarity pass over the *sentences of one block* (tens to low-hundreds of sentences). This is ephemeral, per-call, and discarded. No persistent ANN structure is created. ContextOS is not a vector DB. |
| **Abstractive tier = one bounded LLM call, rate-limited + cost-tracked** | The abstractive tier (§2.5.5) issues a single `route('compress')` LLM call through the same router/adapter as user traffic. It is rate-limited per tenant and its token cost is debited into the **budget ledger** before the request's own model call, identical in spirit to memory-consolidation cost accounting. It is NOT an agent loop, never self-invokes, never retries semantically. |
| **No re-ranking authority** | The compressor does not reorder blocks relative to each other and does not change their assembler scores. It changes block *contents and sizes* only. Final ranking + lost-in-the-middle edge-placement remain the Assembler's job (C1). |

The mental model: the Compressor is a **pure-ish function from block to smaller block** with one controlled, accounted side effect (the abstractive LLM call) and one hard safety property (the NLI guard). It has no opinion about which blocks matter — that's the Assembler — only about how to make a given block cheaper without lying.

## 2.5.3 The three tiers

Compression is tiered because the cost/quality/risk profile of the techniques is wildly different, and most blocks should never reach the expensive, risky tier. Cheapest and safest first; escalate only when the token budget demands it.

```
Tier 0  STRUCTURAL    deterministic, lossless-ish, ~0 risk,   ~0.1-1 ms,  no model
Tier 1  EXTRACTIVE    select original spans, low risk,        ~3-15 ms,   in-proc embed
Tier 2  ABSTRACTIVE   rewrite via LLM, real risk, NLI-gated,  100-400 ms, router LLM call + $$
```

### Tier 0 — Structural (deterministic, near-lossless)

Operates on the *form* of the text, not its meaning. No model, no probabilistic step, fully reversible in effect on facts.

Techniques:

- **Whitespace and markup normalization** — collapse runs of spaces/newlines, strip redundant markdown decoration, normalize bullet glyphs. A chat log dumped with double newlines and `> ` quote prefixes can shed 5-15% of tokens with zero semantic change.
- **Boilerplate / chrome stripping** — remove email signature blocks, legal footers, "Sent from my iPhone", repeated headers, navigation breadcrumbs scraped from web sources. Pattern-driven, allowlist of removable chrome.
- **De-duplication of verbatim repeats** — within a block, fold identical or near-identical (normalized-hash-equal) sentences/paragraphs to a single instance. Common in concatenated tool outputs and forwarded email chains.
- **Structured-data tightening** — re-serialize verbose JSON/HTML tables into compact key:value or pipe-delimited form when the block is recognized as structured. A 40-token pretty-printed JSON object becomes a 12-token flat line; numbers and keys are preserved exactly.
- **Tokenizer-aware formatting** — prefer renderings that the target tokenizer encodes in fewer tokens (e.g., `2026-06-28` over `June 28th, 2026` when both are equivalent and the source is already a date), only when the mapping is provably value-preserving.

Tier 0 is **always run first, unconditionally**, on every block. It is so cheap and so safe that there is no scenario where skipping it helps. It typically yields **1.1x-1.4x** reduction on real-world messy context. Because it is deterministic and fact-preserving by construction (it only deletes provably redundant or non-semantic bytes, or losslessly re-serializes), **Tier 0 does NOT invoke the NLI guard** — there is nothing to verify. The rejected alternative — running everything through extractive/abstractive and skipping cheap structural wins — wastes CPU and, worse, sends *clean* text through risky stages.

### Tier 1 — Extractive (select, never rewrite)

Operates on *salience*. It keeps a subset of the *original* sentences/spans and drops the rest. Crucially, **every surviving token is verbatim from the source** — extractive compression cannot fabricate, because it only deletes.

Algorithm (in-process, over the sentences of one block, `<= ~300` sentences):

1. Split the block into sentences/clauses (deterministic segmenter; structured rows are atomic).
2. Score each sentence's salience as a blend of:
   - **Query relevance** — cosine similarity between the sentence's embedding (reuse the in-process `bge-small-en-v1.5` 384-dim embedder; the query embedding is already computed once upstream at ~6 ms p95 and reused) and the request query.
   - **Lexical centrality** — TF-IDF / LexRank-style centrality *within the block*, capturing sentences that carry the block's core nouns/numbers.
   - **Information markers** — boost sentences containing entities, numbers, dates, units, negations, and named facts (cheap regex/NER heuristics); these are the spans whose loss would most hurt fact retention.
3. Greedily keep top-scoring sentences until the per-block target token budget is met, **preserving original document order** (do not reorder — reordering harms coherence and is the Assembler's lost-in-the-middle concern, not the compressor's; C1 keeps these orthogonal).
4. Stitch kept spans with minimal connective normalization (Tier 0 rules).

Rejected alternatives:

- **Pure first-N / truncation** — trivially cheap but catastrophic for fact retention; the load-bearing number is usually *not* in sentence 1. Rejected as the default; truncation is only the last-resort fallback after a guard failure with no safe smaller form (§2.5.7).
- **Cross-encoder sentence scoring inside the hot path** — higher quality salience, but a cross-encoder is **opt-in, out-of-band only** per the global invariant; we will not put a cross-encoder on the synchronous compression path. The bi-encoder (bge-small) similarity is the launch default.
- **Building a per-block ANN index for centrality** — rejected outright: violates the "never builds/owns an index" invariant and is absurd overkill for a few hundred sentences; a dense in-RAM similarity matrix is `O(n^2)` over `n <= ~300`, which is microseconds.

Extractive typically yields **1.5x-3x** on prose. Because it only deletes original spans, its fact-retention risk is *omission* (a needed fact got dropped), never *fabrication*. The NLI guard (§2.5.6) still runs, but here it is checking "did we drop something the kept text now contradicts or that the block's core claims depend on," and the fail-safe (keep more / keep all) is cheap because the original spans are right there.

### Tier 2 — Abstractive (rewrite via LLM, the only risky tier)

Operates on *expression*. It rewrites a long block into a shorter paraphrase using an LLM. This is the only tier that can achieve **3x-4x+** on dense, redundant prose (meeting transcripts, rambling docs) — and the only tier that can *hallucinate*, *drop*, or *distort* facts. It is therefore the most tightly governed.

Properties:

- **One bounded LLM call per block**, dispatched via `router.route(action='compress')` through the standard adapter. The router picks a cheap, fast model for compression (a downgrade target — compression is an "easy" generation task), consistent with the model-routing strategy.
- **Rate-limited per tenant.** A token-bucket limiter caps abstractive calls/sec/tenant so a flood of huge blocks cannot turn the compressor into an unbounded LLM-spend amplifier or starve the tenant's actual request traffic. On limiter exhaustion the block **falls back to extractive** (Tier 1) rather than blocking.
- **Cost-tracked into the budget ledger.** Before the abstractive call, its *estimated* prompt+completion cost is reserved against the request's budget; after the call, *actual* token usage is reconciled. This inference cost enters the same budget ledger as the request's primary model call and as memory consolidation — there is one cost truth. If reserving the cost would exceed the request budget, the abstractive tier is skipped and the block stays extractive/structural.
- **NLI-gated, mandatorily.** No abstractive output is ever emitted without passing the fact-retention guard (§2.5.6). On guard failure, the abstractive result is discarded and we fall back down the tier ladder (fail-safe; §2.5.7).
- **Prompt-constrained.** The compression prompt instructs: preserve every entity, number, date, quantity, named decision, and negation; do not add information; do not infer; output only compressed text. (The NLI guard does not *trust* this — it *verifies* it. The prompt reduces failure rate; the guard enforces correctness.)

Rejected alternatives:

- **Always-abstractive ("just LLM-compress everything")** — rejected: 100-400 ms and real dollars per block, multiplied across up to 512 candidates, is a latency and cost disaster, and exposes every block to fabrication risk for blocks that Tier 0/1 would have handled safely and freely.
- **Local distilled summarizer model co-located in-process** — tempting for latency, but it adds a second model-serving dependency to the hot-path container, competes with the bge embedder for CPU, and still needs the NLI guard. Rejected at launch in favor of the router LLM call (which also gives per-tenant cost visibility "for free" via the ledger). Revisit only if the abstractive tier's volume justifies a dedicated `compress` model deployment.

## 2.5.4 Tier-selection policy (exactly when to use which)

The compressor does **not** pick a tier per philosophy; it picks the *cheapest tier that hits the per-block target token budget while passing the guard*. Escalation is demand-driven.

The per-block target (`target_tokens`) is supplied by the Assembler-facing budget logic: the Assembler knows the overall hard reserve and apportions a target to each candidate block (proportional to its salience score and current size). The compressor's job is to *meet or beat* that target for this block without violating retention.

```python
def select_and_compress(block, target_tokens, tokenizer, ctx) -> CompressedBlock:
    # Tier 0 is unconditional and free.
    b0 = structural_compress(block, tokenizer)
    if count_tokens(b0.text, tokenizer) <= target_tokens:
        return b0                      # done; cheapest tier sufficed

    # Need more reduction -> extractive.
    b1 = extractive_compress(b0, target_tokens, tokenizer, ctx.query_embedding)
    if (count_tokens(b1.text, tokenizer) <= target_tokens
            and nli_guard(block, b1).passed):     # extractive: only-deletes; guard checks omission
        return b1

    # Still over budget, OR extractive could not hit target without dropping load-bearing facts.
    # Escalate to abstractive ONLY if (a) block is large enough to be worth it,
    # (b) tenant rate budget allows, (c) the $ cost fits the ledger.
    if (count_tokens(b0.text, tokenizer) >= ABSTRACTIVE_MIN_TOKENS      # default 256
            and ctx.rate_limiter.try_acquire(ctx.tenant_id)
            and ctx.budget_ledger.can_reserve(estimate_compress_cost(b0, tokenizer))):
        b2 = abstractive_compress(b0, target_tokens, tokenizer, ctx)    # one router LLM call, cost-tracked
        guard = nli_guard(block, b2)
        if guard.passed and count_tokens(b2.text, tokenizer) <= target_tokens:
            return b2
        # abstractive failed guard or missed target -> fail-safe down the ladder
        ctx.metrics.incr("compress.abstractive.guard_fail" if not guard.passed
                         else "compress.abstractive.missed_target")

    # Fail-safe: return the best SAFE result we have, even if slightly over target.
    # The Assembler's knapsack will account for the real (larger) size and may drop
    # the block entirely rather than ship an unfaithful compression. Correctness > fit.
    return best_safe(b0, b1)
```

Selection rules in prose:

| Situation | Tier used | Rationale |
| --- | --- | --- |
| Block already under target after structural | **Tier 0 only** | Never pay for extractive/abstractive when free structural wins suffice. |
| Block over target, mostly redundant/low-salience prose | **Tier 1 extractive** | Dropping low-salience original sentences hits target with zero fabrication risk. |
| Structured data (JSON/tables/logs) | **Tier 0 (tighten) then stop**, never abstractive | Re-serialization is lossless; abstractive paraphrase of structured data is high-risk and pointless. Structured blocks are **excluded from Tier 2**. |
| Long, dense, redundant natural-language block where extractive can't reach target without dropping load-bearing facts, AND block `>= 256` tokens, AND tenant rate + budget allow | **Tier 2 abstractive (NLI-gated)** | Only here does paraphrase earn its cost/risk; the 3-4x target lives here. |
| Tenant rate limit hit, or abstractive cost would breach request budget | **Stay at Tier 1/0** (best safe) | Cost/latency safety beats fit; the Assembler handles the larger block. |
| Any tier's NLI guard fails | **Fall back to next-safer tier / uncompressed** | Fail-safe (§2.5.7). Never ship an unfaithful block. |

## 2.5.5 The `compress()` contract

```python
from __future__ import annotations
from dataclasses import dataclass
from enum import Enum
from typing import Protocol

class CompressionTier(str, Enum):
    NONE        = "none"          # block returned unchanged (fail-safe / already small)
    STRUCTURAL  = "structural"
    EXTRACTIVE  = "extractive"
    ABSTRACTIVE = "abstractive"

@dataclass(frozen=True)
class CompressedBlock:
    block_id: str                 # ULID of the source candidate block (unchanged)
    tenant_id: str                # non-null; carried through, never crosses tenants
    text: str                     # the compressed (or original, on fail-safe) text
    tier: CompressionTier         # the tier that PRODUCED this text
    original_tokens: int          # tokens of input under `tokenizer`
    final_tokens: int             # tokens of `text` under `tokenizer`  (<= original_tokens)
    ratio: float                  # original_tokens / final_tokens  (>= 1.0)
    fact_retention: float         # NLI-verified retained-fact fraction in [0.0, 1.0]
    guard_passed: bool            # True iff fact_retention >= RETENTION_FLOOR (0.98)
    abstractive_cost_usd: float   # 0.0 unless Tier 2 ran; debited to budget ledger
    replay: "CompressionReplay"   # deterministic record for the Replay Debugger

@dataclass(frozen=True)
class CompressionReplay:
    # Everything needed to byte-exactly reproduce this compression decision (C7).
    tier_attempts: tuple[str, ...]      # e.g. ("structural","extractive","abstractive")
    nli_model_id: str                   # version of the NLI guard model
    nli_claim_scores: tuple[float, ...] # per-claim entailment scores
    abstractive_model_id: str | None    # router-selected compress model, if Tier 2 ran
    abstractive_prompt_hash: str | None # hash of the exact compression prompt
    seed: int | None                    # decoding seed used for the abstractive call
    target_tokens: int
    tokenizer_id: str

class Tokenizer(Protocol):
    def count(self, text: str) -> int: ...

def compress(block: "CandidateBlock",
             target_tokens: int,
             tokenizer: Tokenizer,
             ctx: "CompressionContext") -> CompressedBlock:
    """
    Compress a SINGLE pre-retrieved, post-ACL/redaction block toward `target_tokens`
    measured under `tokenizer`, preserving >= 98% of facts (NLI-guarded).

    GUARANTEES:
      * final_tokens <= original_tokens                 (never grows a block)
      * tier == ABSTRACTIVE  ==>  guard_passed is True   (no unguarded paraphrase ships)
      * guard_passed is False ==>  text is the best SAFE form (often the original);
                                    tier reflects what produced that safe form
      * abstractive_cost_usd is debited to ctx.budget_ledger BEFORE return
      * input is already ACL/redaction-cleared; compress() performs NO access control

    NON-GOALS:
      * does not rank blocks against each other (Assembler owns final ranking, C1)
      * does not retrieve, does not build an index (C13)
      * does not guarantee final_tokens <= target_tokens; correctness wins over fit,
        and the Assembler accounts for the actual final_tokens in its knapsack
    """
```

Key contract points:

- **`tokenizer` is an argument, not a global.** Token counts are tokenizer-relative, and the model (hence tokenizer) is chosen by the router *before* final packing (C3). See §2.5.6.
- **`compress()` never enlarges a block.** Worst case it returns the input (`tier=NONE`, `ratio=1.0`, `guard_passed=True`).
- **`abstractive_cost_usd`** flows into the budget ledger so token-cost-savings accounting is honest: the compression *win* (10-25% prompt-token reduction toward the 40-65% combined savings) is reported net of the compressor's own inference spend.

## 2.5.6 The fact-retention guard (NLI) — how it works

The guard answers one question per compression: **"Does the compressed text still support every fact the original asserted?"** A reduction that drops or distorts facts is worse than no reduction, because a confidently-wrong context poisons the model's answer silently.

### The retention metric

We frame retention as **Natural Language Inference (NLI) entailment of the original's atomic claims by the compressed text**:

1. **Claim extraction.** Decompose the *original* block into a set of atomic factual claims `C = {c_1, ..., c_k}` (entity-relation triples / short factual sentences). This is deterministic and cached per block (it depends only on the original text, so it is computed once and reused across tiers and across cache hits).
2. **Entailment scoring.** For each claim `c_i`, run an NLI model with `premise = compressed_text`, `hypothesis = c_i`, yielding `P(entailment)`. A claim is *retained* iff `P(entailment) >= ENTAIL_THRESHOLD` (default `0.90`) **and** `P(contradiction) < CONTRA_THRESHOLD` (default `0.05`). The contradiction check is what catches *fabrication/distortion*: an abstractive rewrite that flips a number or negation produces a contradiction, not merely a non-entailment.
3. **Retention score.** `fact_retention = |retained claims| / |C|`.
4. **Guard verdict.** `guard_passed = (fact_retention >= RETENTION_FLOOR)`, with **`RETENTION_FLOOR = 0.98`** — the canonical `>= 98%` fact-retention target.

```python
RETENTION_FLOOR  = 0.98
ENTAIL_THRESHOLD = 0.90
CONTRA_THRESHOLD = 0.05

def nli_guard(original_block, compressed) -> GuardResult:
    claims = extract_atomic_claims(original_block)   # cached per original block
    if not claims:
        return GuardResult(passed=True, retention=1.0, scores=())   # nothing to lose
    scores = []
    retained = 0
    for c in claims:
        ent, contra = nli_model.score(premise=compressed.text, hypothesis=c)
        scores.append(ent)
        if ent >= ENTAIL_THRESHOLD and contra < CONTRA_THRESHOLD:
            retained += 1
    retention = retained / len(claims)
    return GuardResult(passed=(retention >= RETENTION_FLOOR),
                       retention=retention, scores=tuple(scores))
```

### The NLI model

- A **self-hosted, small cross-encoder NLI model** (e.g., a DeBERTa-v3-class MNLI/ANLI-tuned checkpoint), CPU-resident alongside the bge embedder, batched over claims. It is a *classifier*, not a generator — no decoding, low and predictable latency, fully deterministic given input.
- **Rejected alternative — "LLM-as-judge" via another router call.** Rejected for the synchronous guard: it doubles the per-block LLM cost and latency we're trying to save, is non-deterministic (breaks replay determinism unless seeded and recorded, which is fragile), and an LLM judge is *itself* prone to hallucinating that a claim is supported. A discriminative NLI classifier is cheaper, deterministic, and purpose-built for entailment. (An LLM-judge cache-correctness/compression-quality harness exists as an *offline roadmap* eval, not on the hot path — consistent with the offline cache-correctness eval harness being a roadmap item.)
- **Rejected alternative — embedding cosine similarity as the "retention" check.** Rejected because cosine similarity rewards topical resemblance, not factual entailment: a paraphrase that says "Q3 revenue rose" when the original said "Q3 revenue *fell* by 12%" is highly cosine-similar and completely wrong. Only directional NLI (with the contradiction check) catches that.

### Why the guard runs on extractive too (not just abstractive)

Extractive *can't fabricate*, but it *can omit a load-bearing fact* (drop the one sentence with the number). Running the same claim-entailment guard catches omission as non-entailment and triggers the fail-safe to keep more spans. Tier 0 structural is exempt (it provably preserves facts by construction; nothing to check) — this exemption keeps the common cheap path free of NLI latency.

### Guard latency budget

The guard adds NLI inference over `k` claims (typically `k` in single digits to low tens for a block). It is sized to fit inside the assembly-stage neighborhood and **does not** widen the authoritative latency table: structural-only blocks skip it entirely; extractive guard cost is small CPU; the abstractive path's dominant cost is the LLM call (100-400 ms), within which the NLI check is a rounding error. Compression as a whole overlaps with retrieval/cache work and is *not* a separate line item that breaks the `< 250 ms p95` control-overhead budget or the `< 50 ms p95` assembly budget (assembly explicitly *excludes* model inference, and the abstractive LLM call — like all model inference — is excluded from ContextOS budgets; only the cheap structural/extractive transforms touch the synchronous assembly-adjacent path).

### Tokenizer correctness (C3 interaction)

`compress()` counts tokens with the **router-selected model's tokenizer**, which is why the router selects the model *before* final packing (C3). Compression sits before assembly but *after* the routing decision is knowable for token-counting purposes:

- **Preferred:** the difficulty/utility router resolves the target model early; the compressor and assembler both count against that model's exact tokenizer, so the hard reserve is enforced precisely.
- **Fallback (model not yet knowable):** the compressor counts against a **conservative max-tokenization estimate plus a documented margin** (default `+8%`). After the route is finalized, the assembler re-validates; if the conservative estimate was wrong enough to breach the hard reserve, the pipeline re-packs or fails closed with a `413`. The compressor's `final_tokens` is always re-counted under the true tokenizer post-route; it never asserts a fit it can't honor.

## 2.5.7 Guard failure = fail-safe (keep uncompressed)

The cardinal rule: **a failed guard never ships a smaller-but-wrong block.** The compressor degrades along the tier ladder toward *more* faithful, *larger* output, and in the limit returns the original.

```
abstractive guard FAIL
    -> discard abstractive text, debit its (already-incurred) cost to the ledger anyway
    -> retry with extractive result (b1) if it passed guard and fits "well enough"
       extractive guard FAIL (or none)
           -> retry with structural-only result (b0)   [structural is fact-safe by construction]
               b0 still over target_tokens
                   -> return b0 as-is (tier=STRUCTURAL): the block is FAITHFUL but large.
                      The Assembler's knapsack now sees the real (larger) final_tokens and
                      MAY drop this block entirely in favor of others. Dropping a block is a
                      faithful outcome; shipping a distorted block is not.
```

Decisions encoded in the fail-safe:

- **Correctness dominates fit.** The compressor will return an over-budget faithful block before it returns an in-budget unfaithful one. The Assembler — the sole budget authority — resolves the over-budget situation by reallocating or dropping, never by accepting falsity.
- **No silent truncation as a "compression."** Hard truncation is only ever the *structural* baseline (b0) when even structural is over target, and it is reported as `tier=STRUCTURAL` with the true (still-over-target) `final_tokens` — never dressed up as a successful compression. We do not chop the tail to hit a number and pretend facts survived.
- **Abstractive cost is still charged on failure.** The LLM call already happened; its dollars are real and are debited to the ledger even though the output was discarded. This makes the cost of *risky* compression visible and discourages over-aggressive abstractive targets — failed abstractive attempts show up as cost with no token savings, which the routing/target-tuning logic can learn from.
- **Observable.** Every guard failure increments `compress.<tier>.guard_fail`, records the per-claim NLI scores in `CompressionReplay`, and is replayable byte-for-byte for the deterministic stages (the abstractive call itself is non-deterministic backend.invoke per C7, recorded with its seed and prompt hash so a recorded-output replay can assert output-equality while a live replay yields a diff).

## 2.5.8 Worked numbers (canonical-fact alignment)

| Property | Value | Source / note |
| --- | --- | --- |
| Reduction target (long blocks) | **2x-4x** token reduction | abstractive reaches the top of this range; extractive the low end |
| Fact retention floor | **>= 98%** | `RETENTION_FLOOR = 0.98`, NLI-guarded |
| Compression contribution to cost savings | **10-25%** prompt-token reduction | part of the 40-65% combined savings (with caching 15-30% + routing-downgrade 20-40%) |
| Candidate cap entering compression | **<= 512** blocks | scope-boundary cap; compressor never sees more |
| Abstractive minimum block size | **256 tokens** (`ABSTRACTIVE_MIN_TOKENS`) | below this, paraphrase isn't worth the cost/risk; stay extractive |
| Query embedder (reused for salience) | `BAAI/bge-small-en-v1.5`, 384-dim, **~6 ms p95** | embedded once upstream, reused; no extra embed cost in extractive |
| Compressor's place in latency budget | overlaps assembly/retrieval; **not** a separate budget line | `< 50 ms` assembly and `< 250 ms` overhead per §9; abstractive LLM call excluded as model inference |

The compression win is reported *net* of `abstractive_cost_usd`, so the 10-25% prompt-token reduction is a real economic saving, not a gross one that hides the compressor's own LLM spend.

This phasing means we get real token savings (structural+extractive: comfortably 1.5x-3x on prose) in Month 1, and the full 2x-4x with fabrication-proof guarantees in Month 3.
