# 2.3 — RBAC Context Firewall

The RBAC Context Firewall is the access-control spine of ContextOS. It is the
component that makes the headline guarantee — **cross-tenant leakage = 0** —
mechanically true rather than aspirational. ContextOS is middleware: every byte
of context it assembles is harvested from multi-tenant stores, semantically
cached, routed across model backends, and streamed back to a caller whose
identity it did not mint. A single missing predicate anywhere on that path is a
data breach. The firewall exists because "be careful in the application code" is
not an isolation model.

This section specifies (a) why we chose **row-level isolation** keyed on
`tenant_id` with Postgres `FORCE ROW LEVEL SECURITY` over schema-level and
database-level isolation; (b) the **two-layer RBAC + ABAC** policy engine, its
deny-overrides combiner, fail-closed posture, and policy schema; (c) the **four
enforcement points** at which `check()` is called on the pipeline; and (d) the
**proof** that no cross-tenant leakage survives — RLS database backstop, the
application firewall, the ≥10,000 hostile-probe property test, and mutation
testing on the fail-closed branches.

This is a defense-in-depth design. The application firewall is the *primary*
control (it fails closed and produces the deny audit record); RLS is the
*backstop* that turns an application bug into a returned-zero-rows non-event
instead of a leak. Neither is trusted alone.

## 2.3.1 — Isolation model: row-level (`tenant_id` + FORCE RLS)

Three isolation granularities are industry-standard for multi-tenant Postgres.
We evaluated all three against ContextOS's actual shape: thousands-to-millions of
small tenants, a hot path that is CPU-bound and latency-critical (< 250 ms p95
control overhead, see [§9 latency budget](#)), pgvector co-located in the same
Postgres, and a flagship replay debugger that must reconstruct *exactly which
rows a tenant could see at decision time*.

| Dimension | **Row-level (CHOSEN)** | Schema-per-tenant (REJECTED) | Database-per-tenant (REJECTED) |
|---|---|---|---|
| Isolation unit | `tenant_id` column + RLS policy | One Postgres schema per tenant | One database (or cluster) per tenant |
| Tenants per node | Millions | ~Hundreds–low thousands (catalog bloat) | Tens–low hundreds |
| Connection pooling | One pool, `SET LOCAL` per txn | Pool fragments per schema / `search_path` churn | Pool per DB — pgbouncer explosion |
| pgvector HNSW index | One shared HNSW index, partitioned by `tenant_id` | N indexes; cold for small tenants, no cross-tenant ANN reuse | N clusters; impossible to co-locate |
| Migrations | One `ALTER`, atomic | N schemas — partial-failure hell | N databases — orchestration nightmare |
| Onboard a tenant | `INSERT` a row | DDL transaction (DDL takes locks) | Provision a database (minutes) |
| Noisy-neighbor blast radius | Query-planner + RLS predicate | Same shared resources anyway | True physical isolation |
| Leak failure mode | **Missing `WHERE` → RLS denies (fail-closed)** | Wrong `search_path` → silent cross-tenant read | N/A within a tenant DB |
| Crypto-shred (RTBF, C11) | Per-subject DEK over row payload; tombstone + GC sweep | Drop schema (coarse; can't shred a sub-tenant subject) | Drop DB (coarsest) |

### Why row-level wins for ContextOS

1. **Scale shape.** ContextOS targets *many* tenants, most of them small. A
   schema-per-tenant model puts every tenant's tables into `pg_class` /
   `pg_attribute`; at tens of thousands of tenants the system catalog itself
   becomes the bottleneck — `pg_dump`, autovacuum, and planner catalog lookups
   degrade. Database-per-tenant multiplies that by the entire WAL/checkpoint/
   background-worker overhead per tenant. Row-level keeps **one** physical schema
   and lets the `tenant_id` partition key + RLS predicate do the work.

2. **The vector store must be shared.** pgvector HNSW is co-located in Postgres
   behind the `VectorStore` adapter. A single HNSW index partitioned by
   `tenant_id` gives us the canonical **pgvector HNSW ANN probe p95 = 18 ms** at
   launch scale (≤ 5M vectors/tenant). Schema/DB-per-tenant would shard the index
   into thousands of cold structures — tiny per-tenant HNSW graphs are *worse*
   (no shared graph connectivity, constant rebuild churn) and they make the
   Qdrant cutover-at-scale escape hatch incoherent. Row-level isolation + a
   `tenant_id` filter pushed into the ANN probe is the only model where the 18 ms
   number holds.

3. **The pooled-connection hot path.** The gateway is stateless and CPU-bound,
   sustaining **5k–10k req/s per node** off one shared connection pool. Row-level
   needs only a per-transaction `SET LOCAL app.tenant_id` — a 5 ms edge step (see
   §9). Schema-per-tenant forces `search_path` mutation per request (pool
   fragmentation, prepared-statement cache thrash); database-per-tenant forces a
   *different connection* per tenant, which collapses pooling entirely.

4. **Fail-closed leak semantics.** This is the decisive property. With RLS, the
   default for a row is *invisible*. A developer who forgets a `WHERE tenant_id`
   clause gets **zero rows**, not someone else's rows. With schema-per-tenant,
   the analogous bug — a stale `search_path` or a hardcoded schema name — returns
   *the wrong tenant's data* silently. Row-level's failure mode is denial;
   schema-level's failure mode is leakage. For a system whose top-line guarantee
   is "leakage = 0," we pick the model whose mistakes deny.

### The schema

`tenant_id` is a **non-null partition key on every row, object, and key** in
ContextOS — Postgres rows, Redis keys, cache namespaces, vector payloads, replay
bundles. IDs are **ULID**; timestamps are **RFC-3339 UTC**.

```sql
-- Every multi-tenant relation follows this template. Partitioned by tenant_id
-- (LIST or HASH) so the planner prunes partitions AND the RLS predicate is a
-- partition-key equality (index-friendly, no seq scan).
CREATE TABLE memory_item (
    id            TEXT      NOT NULL,             -- ULID
    tenant_id     TEXT      NOT NULL,             -- partition key, never null
    namespace     TEXT      NOT NULL,             -- project/agent/user (C2)
    kind          TEXT      NOT NULL,             -- episodic|semantic|...
    content       BYTEA     NOT NULL,             -- encrypted under tenant/subject DEK
    embedding     vector(384),                    -- bge-small-en-v1.5
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    subject_dek_id TEXT,                          -- per-subject DEK ref (C11 crypto-shred)
    PRIMARY KEY (tenant_id, id)
) PARTITION BY LIST (tenant_id);

ALTER TABLE memory_item ENABLE ROW LEVEL SECURITY;
ALTER TABLE memory_item FORCE  ROW LEVEL SECURITY;  -- applies to table OWNER too

-- The session GUC is set per transaction by the edge (SET LOCAL app.tenant_id).
CREATE POLICY tenant_isolation ON memory_item
    USING       (tenant_id = current_setting('app.tenant_id', true))
    WITH CHECK  (tenant_id = current_setting('app.tenant_id', true));
```

`FORCE ROW LEVEL SECURITY` is non-negotiable. Plain `ENABLE` exempts the table
owner; under connection pooling the app role is frequently the owner, which would
silently bypass RLS. `FORCE` closes that hole. The application **never** connects
as a superuser or `BYPASSRLS` role — those roles do not exist in the runtime
credential set.

```sql
-- current_setting('app.tenant_id', true) returns NULL if unset.
-- NULL = NULL is NULL (not TRUE) in SQL three-valued logic => policy denies.
-- Therefore an UNSET tenant context returns ZERO rows. Fail-closed by construction.
```

> **Invariant FC-1 (fail-closed RLS):** if the edge has not executed
> `SET LOCAL app.tenant_id = <ulid>` inside the current transaction, every RLS-
> protected query returns zero rows and every write is rejected by `WITH CHECK`.
> Forgetting to set the tenant context is a *denial*, never a leak.

Setting the GUC uses `set_config(..., is_local => true)` so the value is scoped
to the transaction and cannot bleed across pooled connections:

```python
async def with_tenant(conn: asyncpg.Connection, tenant_id: str):
    # SET LOCAL: reverts at txn end; impossible to leak to the next pool checkout.
    await conn.execute("SELECT set_config('app.tenant_id', $1, true)", tenant_id)
```

## 2.3.2 — The two-layer policy engine: RBAC + ABAC

RLS answers exactly one question: *"is this row in the caller's tenant?"* It does
**not** know whether `principal P` may perform `action route` on `resource model
gpt-4o`, whether `P`'s namespace allows reading project `acme/billing`, or
whether residency policy forbids sending tenant `T`'s data to a US-region
backend. Those are *authorization* decisions and they live in the application
firewall — a **two-layer** engine.

```
            ┌──────────────────────── check(principal, resource, action) ──────────────────────┐
            │                                                                                    │
  Layer 1   │   RBAC: does any ROLE the principal holds grant `action` on `resource.type`?      │
  (coarse)  │          roles -> permissions -> (resource_type, action) grants                   │
            │                                                                                    │
  Layer 2   │   ABAC: do the attribute RULES permit this *specific* resource instance?          │
  (fine)    │          tenant match, namespace match (C2), residency, budget, sensitivity,      │
            │          time-of-day, RBACPolicy shared-org grant, model allowlist (C10)          │
            │                                                                                    │
  Combiner  │   DENY-OVERRIDES: any explicit DENY wins; absence of a PERMIT = DENY (closed)     │
            └────────────────────────────────────────────────────────────────────────────────-─┘
```

**Why two layers, and why this order.** RBAC is cheap, cacheable, and answers the
common case ("a `viewer` can never `delete`") with a hash lookup. ABAC is
expensive (it inspects resource attributes — residency tags, namespace ACLs,
budget ledgers) and is only consulted when RBAC has not already denied. Running
RBAC first is a short-circuit that keeps `check()` inside its **~1 ms** budget on
the hot path; it is *not* a relaxation — a missing Layer-1 grant denies before
Layer 2 is ever consulted.

### Rejected policy approaches

| Approach | Why rejected |
|---|---|
| **Pure RBAC** (roles → permissions only) | Cannot express *per-resource* constraints: residency, namespace ACL, per-model allowlist, budget. Would force one role per (model × region × namespace) — combinatorial role explosion. |
| **Pure ABAC** (attribute rules only) | Every decision evaluates the full rule set; no cheap short-circuit. Loses the auditable "this role grants this" story regulators want. Hot-path cost unbounded. |
| **OPA / Rego sidecar** | Out-of-process call (network hop + serialization) blows the ~1 ms `check()` budget on a path called 4× per request; adds an operational dependency the locked architecture does not want at the edge. We keep the engine **in-process** (in-process at launch is the locked inter-service posture) and reserve Rego-style externalization for the cross-encoder-style opt-in, out-of-band tooling, not the hot path. |
| **AWS-IAM-style allow-only with explicit-deny** | We adopt the *deny-overrides* idea but reject IAM's policy-document sprawl; our schema is typed and versioned (see below), not free-form JSON. |

### Decision algebra

- **Deny-overrides combiner.** The decision over a set of applicable rules is:
  if **any** rule yields `DENY`, the result is `DENY`. Else if **at least one**
  rule yields `PERMIT`, the result is `PERMIT`. Else (no applicable rule) the
  result is `DENY` — the **closed-world default**.
- **Fail-closed on error.** If policy evaluation *throws* — missing attribute,
  unparseable rule, store unavailable, ambiguous namespace (C2) — the result is
  `DENY`, the request gets a `403`, and an audit record is emitted. There is no
  code path from "evaluation failed" to "allowed."

```python
def combine(decisions: list[Decision]) -> Decision:
    if any(d.effect is Effect.DENY for d in decisions):
        return Decision.deny(reason="explicit_deny")
    if any(d.effect is Effect.PERMIT for d in decisions):
        return Decision.permit()
    return Decision.deny(reason="no_applicable_rule")  # closed-world
```

### The `Action` enum and `check()` contract

```python
from enum import Enum

class Action(str, Enum):
    READ       = "read"        # retrieve memory/context rows
    WRITE      = "write"       # write-back, consolidation output
    DELETE     = "delete"      # RTBF / tombstone (executes via Security, C11)
    ADMIN      = "admin"       # tenant config, policy edits, key rotation
    ROUTE      = "route"       # send context to a model backend (C10 authority)
    CACHE_READ = "cache_read"  # read a semantic/exact cache entry

@dataclass(frozen=True)
class Principal:
    tenant_id: str            # ULID; MUST equal the RLS tenant context
    subject_id: str           # user/agent/service ULID
    roles: frozenset[str]
    namespaces: frozenset[str]  # within-tenant namespaces granted (C2)
    attrs: Mapping[str, str]    # region, clearance, org_shared_grant, ...

@dataclass(frozen=True)
class Resource:
    type: str                 # "memory" | "cache" | "model" | "response_field"
    id: str | None
    tenant_id: str | None     # owning tenant of the resource instance
    namespace: str | None     # C2: project/agent/user; None => ambiguous => DENY
    attrs: Mapping[str, str]  # residency, sensitivity, model_id, region, ...

def check(principal: Principal, resource: Resource, action: Action) -> Decision:
    # Layer 0 — tenant gate (cheapest, hardest): cross-tenant is structurally impossible.
    if resource.tenant_id is not None and resource.tenant_id != principal.tenant_id:
        # Shared-org is the ONLY exception, and only via an explicit ABAC grant.
        if not _shared_org_permitted(principal, resource, action):
            return Decision.deny(reason="cross_tenant")

    # Layer 1 — RBAC: does any role grant (resource.type, action)?
    if not _rbac_grants(principal.roles, resource.type, action):
        return Decision.deny(reason="rbac_no_grant")

    # Layer 2 — ABAC: evaluate attribute rules, deny-overrides.
    return combine(_abac_rules(principal, resource, action))
```

`check()` is the **single authority**. No enforcement point hand-rolls a tenant
or namespace comparison; they all call `check()`. This is what makes the router
derive its allowlist from `check()` rather than a second policy store (C10).

### Policy language / schema

Policies are typed, versioned YAML compiled to an in-memory decision table at
load. No free-form expressions on the hot path — ABAC rules are a closed set of
**named predicates** evaluated by compiled Python, so a policy can never inject
arbitrary code and evaluation cost is bounded.

```yaml
# policy schema v1 — versioned; policy_version flows into the replay bundle & cache fingerprint
apiVersion: contextos.dev/policy/v1
kind: TenantPolicy
metadata:
  tenant_id: 01HZ...               # ULID
  policy_version: 7                # monotonic; part of ReplayResult & cache signature
roles:
  viewer:    { grants: [{ resource: memory, actions: [read, cache_read] }] }
  agent:     { grants: [{ resource: memory, actions: [read, write, cache_read] },
                         { resource: model,  actions: [route] }] }
  admin:     { grants: [{ resource: "*",    actions: [read, write, delete, admin, route, cache_read] }] }
abac_rules:
  - id: namespace-hard-filter        # C2: within-tenant namespace is a HARD filter
    when:  { action: [read, write, cache_read], resource_type: memory }
    require:
      - predicate: namespace_in_principal      # resource.namespace ∈ principal.namespaces
      - predicate: namespace_not_null          # missing/ambiguous => DENY
    on_fail: deny
  - id: shared-org-grant             # C2: opt-in shared-org namespace, RBAC-gated
    when:  { action: [read], resource_type: memory, resource_namespace_scope: org_shared }
    require:
      - predicate: principal_has_org_shared_grant
    on_fail: deny
  - id: residency                    # C9/C10: residency NEVER bypassed
    when:  { action: [route], resource_type: model }
    require:
      - predicate: model_region_in_allowed_residency   # resource.attrs.region ∈ tenant residency set
    on_fail: deny
  - id: model-allowlist              # C10: single authority for model allowlist
    when:  { action: [route], resource_type: model }
    require:
      - predicate: model_id_in_tenant_allowlist
    on_fail: deny
  - id: budget                       # hard-policy budget filter (C9: static, fail-closed)
    when:  { action: [route], resource_type: model }
    require:
      - predicate: tenant_budget_remaining_static       # evaluated on static ledger snapshot
    on_fail: deny
  - id: response-redaction
    when:  { action: [read], resource_type: response_field }
    require:
      - predicate: field_sensitivity_le_principal_clearance
    on_fail: deny
default_effect: deny                 # closed-world
```

`policy_version` is load-bearing beyond authorization: it is folded into the
**coarse cache fingerprint** (C6: `system_prompt_version + stable_fact_set_version
+ …`) so a policy change invalidates cached decisions, and it is recorded in the
**`ReplayResult`** (C7) so a replay reconstructs *the exact policy in force at
decision time*.

## 2.3.3 — The four enforcement points

The firewall is not a gate at the front door; it is enforced at **four** distinct
points along the pipeline, each guarding a different leak surface. They map onto
the pipeline ordering invariant:

```
auth/tenant ─▶ cache lookup ─▶ retrieve candidates ─▶ ACL/redaction ─▶ compression
   │ (sets       │ (EP2:           │ (EP1:               │ (EP4:           │
   │  RLS ctx)    │  cache key)     │  retriever)         │  redaction)     │
   ▼              ▼                 ▼                     ▼                 ▼
                                                       ... ─▶ assembly ─▶ routing (EP3: action='route') ─▶ adapter ─▶ stream
```

Note the placement: **EP4 redaction runs as part of ACL/redaction, *before*
compression** — compression always operates on already-redacted text, so a
compressor can never re-expose a field that redaction removed (pipeline
invariant).

### EP1 — Retriever (tenant_id + namespace, C2)

The repository boundary is where the **namespace hard filter** lives (C2). Two
independent predicates apply: `tenant_id` (RLS at the DB + the explicit
application predicate) and `namespace` (`check(..., READ)`'s ABAC
`namespace-hard-filter` rule). Missing or ambiguous namespace = **deny**,
fail-closed.

```python
async def retrieve_candidates(p: Principal, q: Query) -> list[Candidate]:
    if q.namespace is None or q.namespace not in p.namespaces:
        raise AccessDenied("namespace_hard_filter")            # C2 fail-closed
    res = Resource(type="memory", id=None, tenant_id=p.tenant_id,
                   namespace=q.namespace, attrs={})
    if not check(p, res, Action.READ).permitted:
        raise AccessDenied("retriever_acl")
    async with with_tenant(self.conn, p.tenant_id):            # RLS backstop
        # pgvector ANN probe + BM25 are BOTH namespace-filtered in SQL.
        # tenant_id is enforced by RLS; namespace is an explicit predicate.
        return await self._ann_and_bm25(q, namespace=q.namespace)  # <=512 candidates
```

Two things to note. First, the retriever returns **raw per-modality candidate
scores only** (C1) — it does *not* rank or pack; the Context Assembler is the sole
final-ranking + budget authority. The firewall does not change that division.
Second, the namespace filter is in the SQL `WHERE` *and* checked in Python; the
DB filter is the performance path, the Python check is the audited fail-closed
gate. The hard cap of **≤ 512 candidates** into assembly is preserved — the
firewall never widens the candidate set.

### EP2 — Cache key (tenant salt)

The two-tier cache (exact-hash in Redis < 1 ms p99; semantic-ANN in
pgvector/Qdrant 8–15 ms p95, C5) is **per-tenant namespaced**. Every cache key is
salted with `tenant_id`, so a hash collision or an embedding-bucket collision
*cannot* return another tenant's entry — the keyspaces are disjoint by
construction.

```python
def exact_cache_key(p: Principal, fingerprint: CoarseFingerprint) -> str:
    # tenant_id salt => keyspaces are disjoint across tenants.
    # Coarse fingerprint (C6): query-embedding bucket + model_id
    #   + system_prompt_version + stable_fact_set_version + policy_version.
    return f"cos:cache:{p.tenant_id}:{fingerprint.sha256()}"

async def cache_lookup(p: Principal, fp: CoarseFingerprint) -> CacheHit | None:
    if not check(p, Resource("cache", None, p.tenant_id, None, {}),
                 Action.CACHE_READ).permitted:
        return None                                            # fail-closed: treat as miss
    hit = await self.redis.get(exact_cache_key(p, fp))
    # Semantic tier ANN is ALSO tenant-namespaced (separate index partition / collection).
    return hit
```

Two correctness rules ride on this. **(a)** A semantic cache lookup that *misses*
pays the embed (~6 ms) + ANN (~4 ms) cost inside the 10 ms cache-stage budget;
the tenant salt applies to the ANN partition, not just the Redis key. **(b)**
**Memory-private-grounded responses are non-cacheable** (C6) — if a response was
grounded on a specific tenant's private memory rows, it is flagged
non-cacheable so it can never be served to *any* principal, including another
subject in the same tenant whose namespace differs. The cache-hit ratio target
(**25–45%**) already accounts for this exclusion.

### EP3 — Router (action='route' allowlist + residency, C10/C9)

The router is the egress point: it decides which model backend receives the
assembled context. It calls `check(principal, resource=model, action='route')`
as the **single authority** for both the model allowlist and residency (C10).
`RoutePolicy.allowed_backends` is *derived* from `check()` — there is no second
policy store the router consults.

```python
def resolve_route(p: Principal, candidates: list[Backend]) -> RoutePolicy:
    allowed = []
    for b in candidates:
        res = Resource(type="model", id=b.model_id, tenant_id=p.tenant_id,
                       namespace=None,
                       attrs={"region": b.region, "model_id": b.model_id})
        # action='route' enforces: allowlist + residency + static budget (hard filters).
        if check(p, res, Action.ROUTE).permitted:              # C10 single authority
            allowed.append(b)
    if not allowed:
        raise AccessDenied("route_no_allowed_backend")         # fail-closed 403, never default-out
    return RoutePolicy(allowed_backends=allowed)
```

The router's fail posture is split (C9), and the firewall sits on the
**hard-policy** half:

- **Hard-policy filters** — allowlist, residency, capability, budget — evaluate on
  the **static** policy snapshot and **fail-CLOSED independently of the
  health-store**. If the health/telemetry store is down, these still deny
  non-compliant backends. The **safe-default pool itself satisfies every hard
  filter** — residency is never bypassed to reach a fallback.
- **Optimization signals** — latency, queue depth, quality score — fail *open* to
  a static ranking. They are *not* part of the firewall; they never widen the
  allowlist, only reorder within it.

The router runs **after** assembly but selects the model **before** final packing
where the tokenizer matters (C3) — the firewall's job (which backends are
*eligible*) is independent of and prior to tokenizer-driven repacking.
GPU-aware routing remains a **telemetry reader**; the firewall never lets it
schedule GPUs or escalate a backend past a hard filter (scope-boundary
invariant).

### EP4 — Response redaction

The last enforcement point guards the *egress stream*. Even when retrieval and
routing were authorized, individual fields in a backend response (or in assembled
context echoed back) may exceed the principal's clearance — e.g., a tool result
carrying PII, or a memory citation a lower-clearance subject may not see.
Redaction runs in the **ACL/redaction** stage, **before compression**, so
compression never re-surfaces redacted content.

```python
def redact(p: Principal, fields: Iterable[ResponseField]) -> list[ResponseField]:
    out = []
    for f in fields:
        res = Resource(type="response_field", id=f.id, tenant_id=p.tenant_id,
                       namespace=f.namespace,
                       attrs={"sensitivity": f.sensitivity})
        if check(p, res, Action.READ).permitted:               # field_sensitivity <= clearance
            out.append(f)
        else:
            out.append(f.redacted())                            # replaced, not dropped silently
    return out
```

Redaction *replaces* rather than *drops* — a redacted field leaves a typed
placeholder so the omission is auditable and replay-reconstructable, rather than
a silent disappearance the caller cannot distinguish from "no data."

## 2.3.4 — Proof of zero cross-tenant leakage

The guarantee is **cross-tenant leakage = 0**. "Zero" is a strong word; we earn
it with four mutually-reinforcing mechanisms and a CI hard gate. No single one is
trusted.

### (1) RLS database backstop

Every multi-tenant relation runs `FORCE ROW LEVEL SECURITY` with a
`tenant_id = current_setting('app.tenant_id')` policy (§2.3.1). The application
role has no `BYPASSRLS` and is never superuser. **Even if every line of
application authorization code were deleted**, a query without the correct
transaction-scoped `app.tenant_id` returns zero rows (Invariant FC-1). RLS is the
floor under the firewall — it converts an application bug from a *leak* into a
*denial*.

### (2) Application RBAC firewall

The four enforcement points (§2.3.3) call the single `check()` authority at the
retriever, cache key, router, and redaction. Layer-0's tenant gate makes a
cross-tenant resource structurally `DENY` unless an explicit shared-org ABAC
grant applies. Deny-overrides + closed-world default mean the *absence* of a
permit is a denial. This is the primary control; RLS is the backstop. Defense in
depth: the firewall must agree with RLS, and CI proves it (below).

### (3) ≥10,000 hostile-probe property test (CI hard gate)

A property-based test (Hypothesis) generates **second-tenant hostile probes** and
asserts that none ever observes a foreign tenant's bytes. This is a **CI hard
gate: ≥ 10,000 hostile probes must pass with zero leaks** or the build is red.

```python
@given(probe=hostile_probes())          # strategy: forge tenant_id, namespace, cache key,
                                         # model residency, response field across tenant pairs
@settings(max_examples=10_000, deadline=None, derandomize=False)
def test_no_cross_tenant_leak(probe):
    victim, attacker = seed_two_tenants()          # distinct ULIDs, disjoint data
    attacker_principal = principal_for(attacker)

    # The attacker tries to reach the victim through every enforcement point.
    for surface in (RETRIEVER, CACHE, ROUTER, REDACTION):
        result = exercise(surface, attacker_principal, target=probe.against(victim))
        assert victim.secret_marker not in serialize(result)   # no bytes leak
        if probe.is_cross_tenant:
            assert result.denied or result.empty                # explicit deny or zero rows
        # RLS backstop check: even with the app check stubbed out, DB returns 0 rows.
        with app_authz_disabled():
            assert raw_db_query(surface, attacker_principal, victim).rowcount == 0
```

The strategy forges: mismatched `tenant_id`, foreign `namespace` (C2 ambiguity),
colliding cache keys (EP2 salt bypass attempt), cross-residency model routing
(EP3), and over-clearance response fields (EP4). The 10k figure is the *floor*,
not the target — the gate runs every CI build and on a nightly extended budget.

### (4) Mutation testing on fail-closed paths

Property tests prove the *current* code denies. **Mutation testing** (`mutmut` /
`cosmic-ray`) proves the tests would *catch a regression*. We deliberately mutate
the fail-closed branches and require the suite to **kill every mutant**:

| Mutated site | Injected mutation | Test that must kill it |
|---|---|---|
| `combine()` closed-world default | `return Decision.permit()` instead of deny | `test_no_applicable_rule_denies` |
| Deny-overrides | drop the `any(DENY)` short-circuit | `test_explicit_deny_wins` |
| RLS `FORCE` | downgrade `FORCE` → `ENABLE` in migration | `test_owner_role_cannot_bypass_rls` |
| `current_setting('app.tenant_id', true)` | flip `missing_ok` `true`→`false` (raise vs NULL) — and NULL-policy | `test_unset_tenant_returns_zero_rows` |
| EP1 namespace filter | `namespace not in` → `namespace in` | hostile-probe namespace cases |
| EP2 cache key | drop the `tenant_id` salt | `test_cache_key_disjoint_across_tenants` |
| EP3 residency | `region in allowed` → `True` | `test_residency_never_bypassed` |
| EP4 redaction | `permitted` → `True` (pass-through) | `test_overclearance_field_redacted` |

A surviving mutant on a fail-closed branch **fails CI**. This guarantees the
≥10k-probe gate has *teeth*: it is provably sensitive to exactly the regressions
that would cause a leak.

### Delete / RTBF (C11) — handled in Security

Cross-store deletion (right-to-be-forgotten) is **not** executed by the firewall;
the firewall only authorizes `Action.DELETE` via `check()`. The mechanism —
**tombstone + idempotent GC sweep**, with **embeddings inside crypto-shred scope**
(the vector payload and id are encrypted under the per-subject DEK; shredding the
DEK renders them unrecoverable) — is specified in the **Security** section
(C11). The firewall's only contribution is the authorization gate and the
`subject_dek_id` reference column carried on every row (§2.3.1).

## 2.3.5 — Summary of guarantees

| Guarantee | Mechanism | Canonical figure |
|---|---|---|
| Cross-tenant leakage = 0 | FORCE RLS backstop + app firewall + 10k hostile probes + mutation testing | 0 |
| CI isolation gate | hostile second-tenant property probes | ≥ 10,000 |
| `check()` hot-path cost | in-process RBAC short-circuit + bounded ABAC predicates | ~1 ms (within the 5 ms edge auth step, §9) |
| Fail-closed everywhere | deny-overrides + closed-world default + NULL-tenant RLS | denial, never leak |
| Single route authority | `check(p, model, route)`; `RoutePolicy` derived (C10) | one policy store |
| Residency never bypassed | hard-filter, static, fail-closed; safe pool satisfies it (C9) | — |

The firewall adds no latency line of its own to the §9 budget: EP1/EP2/EP3/EP4
`check()` calls are absorbed into the **5 ms edge auth/tenant step** and the
respective stage budgets (cache 10 ms, retrieval 40 ms, routing 5 ms). The
guarantee it buys — provable zero cross-tenant leakage — is the precondition for
every other claim ContextOS makes.
