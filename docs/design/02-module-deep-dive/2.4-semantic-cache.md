# 2.4 Semantic Cache Engine

The Semantic Cache Engine is ContextOS's first line of token-cost defense and its tightest latency component. It sits at position two in the pipeline — `auth/tenant -> [cache lookup] -> retrieve candidates -> ...` — and its job is to short-circuit the entire downstream pipeline (retrieval, ACL, compression, assembly, routing, inference) when a sufficiently-equivalent request has been served before. A hit here saves not just the model bill but the full `< 250 ms p95` control overhead plus the model round-trip.

This section specifies a **two-tier** cache (exact-hash + semantic-ANN), a **COARSE** correctness-preserving fingerprint, per-tenant namespacing with **zero cross-tenant hits**, and a **fail-open** posture so a cache outage degrades cost — never correctness or availability.

The cache is responsible for **15-30% of the 40-65% combined token-cost savings** envelope. It is *not* responsible for the other levers (routing downgrade 20-40%, compression 10-25%); those are accounted in their own sections and must not be double-counted against the cache.

---

## 2.4.1 Design goals and non-goals

| Goal | Target | Source of truth |
|---|---|---|
| Exact-tier lookup | `< 1 ms p99` (Redis) | Canonical facts |
| Semantic-tier lookup | `8-15 ms p95` (includes embed-on-miss) | Canonical facts |
| Cache hit-ratio | **25-45%** on realistic mixed workload | C6 (COARSE fingerprint) |
| Token savings attributable to cache | **15-30% of model spend** | Canonical token-cost table |
| Cross-tenant cache hits | **0**, fail-closed at namespace | C2, leakage invariant |
| Outage behavior | **Fail-open** (treat as miss, never block) | C12 best-effort posture |
| Correctness eval | Offline cache-correctness harness (roadmap) | C6 |

**Non-goals (scope-boundary invariants, C13):**

- The cache is **not** a vector database. It performs ANN over a per-tenant cache index that is a *bounded, TTL-evicted* set of prior responses, not a general retrieval index. It never owns the *memory* index (that is the Memory Engine, §2.2) and never exceeds the candidate-handling discipline.
- The cache **never serves memory-private-grounded responses**. Any response whose context was assembled from per-tenant private long-term/episodic memory is flagged `cacheable=false` and is structurally ineligible (see §2.4.5). This is the single most important correctness rule in this engine.
- The cache is **not** an LLM, an inference engine, or a router. It returns a previously-materialized response or it returns `Miss`.

---

## 2.4.2 Two-tier architecture (C5: one datastore choice)

We run **two tiers with two datastores**, and we are opinionated about which datastore owns which tier. Per **C5**, isolation and latency are consistent with the rest of the system: the exact tier lives in **Redis**, the semantic tier lives in **pgvector (Qdrant escape hatch at scale)** — the *same* vector substrate the rest of ContextOS uses. We do **not** introduce a third vector engine just for caching.

```
                         lookup(query, tenant)
                                  │
                  ┌───────────────▼───────────────┐
                  │  Tier 0: EXACT-HASH (Redis)    │   < 1 ms p99
                  │  key = coarse_sig (SHA-256)    │   per-tenant keyspace
                  └───────────────┬───────────────┘
                          hit?    │   no
                       yes│       ▼
                          │  ┌───────────────────────────────────┐
                          │  │ Tier 1: SEMANTIC-ANN (pgvector)   │  8-15 ms p95
                          │  │ HNSW over normalized query embed  │  (embed 6ms + ANN ~4ms
                          │  │ cosine >= calibrated threshold τ  │   + verify)
                          │  └───────────────┬───────────────────┘
                          │          hit?    │   no
                          │       yes│       ▼
                          ▼          ▼      Miss → downstream pipeline
                        Hit(response, tier, sig)
```

### Why two tiers and not one

**Rejected: semantic-only (single ANN tier).** A pure semantic cache pays the query-embedding cost (`~6 ms p95`) and an ANN probe on *every* request, even byte-identical repeats. For high-frequency identical traffic (retry storms, polling agents, idempotent re-sends) that is ~10 ms of waste per request and an unnecessary pgvector load. The exact tier answers these in `< 1 ms p99` with no embedding. **Rejected: exact-only.** Real traffic is paraphrastic; exact-only caps the hit-ratio in the low single digits. Two tiers is the only configuration that hits the **25-45%** target.

**Rejected: Redis for the semantic tier too (RediSearch/Redis-VSS).** It would unify the datastore but splits our vector story across two engines (Redis-VSS for cache, pgvector for memory) with two index types to tune, two isolation models, and two crypto-shred (C11) paths. pgvector co-located in Postgres already gives us **`p95 = 18 ms` HNSW ANN** at launch scale and inherits Postgres `FORCE ROW LEVEL SECURITY` — the cache's namespace guarantee then rides the *same* RLS firewall that protects every other row. Cache ANN sets are smaller and pre-filtered by `coarse_sig` equality, so the semantic-cache probe lands at the fast end of the pgvector envelope; we budget `~4 ms` for the ANN portion of the `8-15 ms` line (the rest is the `~6 ms` embed on a miss + verification), well inside the `18 ms` general-purpose HNSW figure.

> **Latency line (C4/§9 reference):** Section 9's table owns the authoritative budget. Cache lookup is **`10 ms p95`** there: exact tier `< 1 ms`; on a semantic miss, add the **query embedding `6 ms` + ANN `4 ms`**. The cache-miss embedding cost is *on the cache line*, not hidden. When the request continues to retrieval, that same embedding is **reused** (§2.4.6) — we never embed twice.

## 2.4.3 The COARSE fingerprint (C6)

The cache key is a **coarse correctness-preserving signature**. Per **C6**, it is precisely:

```
coarse_sig = SHA-256(
      normalized_query_embedding_bucket    # locality-sensitive bucket of the L2-normed embedding
    ‖ model_id                             # the routed/declared model identity
    ‖ system_prompt_version                # versioned system-prompt hash
    ‖ stable_fact_set_version              # version of the tenant's stable, cacheable fact set
)
```

Every component is mandatory; a missing component is a **hard miss** (fail-closed *into* the pipeline, not a degraded guess). The signature is the **exact-tier key** and also a **hard pre-filter** on the semantic tier (semantic ANN only considers entries sharing `model_id`, `system_prompt_version`, and `stable_fact_set_version`; only the embedding-bucket dimension is relaxed into similarity).

### Why each component exists, and what it rejects

| Component | Why | Rejected alternative |
|---|---|---|
| `normalized_query_embedding_bucket` | Captures *semantic* equivalence, not surface form. The bucket is an LSH-style hyperplane bucket of the **L2-normalized** 384-dim BGE embedding, so the exact tier can hash byte-identical-after-normalization queries while the semantic tier handles near-neighbors. | **Raw query string hash:** misses all paraphrase, kills hit-ratio. **Full-float embedding as key:** floats aren't hashable-stable across runs; forces semantic tier for *everything*. |
| `model_id` | A response cached for a strong model is **not** valid for a weak one (and vice versa). Caching across models silently corrupts quality and breaks routing economics. | **Model-agnostic key:** would serve a Haiku-tier answer to a request routed to a frontier model. Unacceptable. |
| `system_prompt_version` | The system prompt is part of the function being memoized. Change the prompt, change the output distribution. | **Ignore system prompt:** invalidates the cache's correctness contract the moment ops ships a prompt tweak. |
| `stable_fact_set_version` | The cache may only ground on **stable, tenant-shareable facts** (policy text, product catalogs, public docs) — versioned so a fact update invalidates en masse. **Private memory is excluded by construction** (§2.4.5). | **Whole-context hash:** would fold private memory and per-request retrieval into the key, making *every* request unique → 0% hit-ratio, the failure mode COARSE exists to avoid. |

### Why COARSE and not FINE

A FINE fingerprint (hashing the fully-assembled prompt, including retrieved candidates and memory) is *trivially correct* but yields a **near-zero hit-ratio** because assembled context is almost always unique per request. COARSE deliberately keys on the **stable, intent-bearing** inputs and **excludes** the volatile/private ones. This is the engineering decision that delivers the **re-derived 25-45% hit-ratio** instead of single digits. The price is that COARSE *can* be wrong (two coarsely-equal queries that should diverge), which is exactly why:

> **Roadmap dependency (C6):** An **offline cache-correctness eval harness** is required to bound COARSE's false-hit rate. It replays a labeled corpus through `lookup()`, compares served-vs-fresh responses with an NLI/semantic-equivalence judge, and reports a false-acceptance rate per threshold τ. The similarity threshold (§2.4.4) and the embedding-bucket width are **not** finalized until this harness gives a measured ROC curve. Until then we ship with the conservative defaults below and treat τ as provisional.

## 2.4.4 Embedding model & similarity-threshold strategy

### Embedding model

The cache uses the **same** self-hosted **`BAAI/bge-small-en-v1.5` (384-dim)** embedding as the rest of ContextOS, via the pluggable `EmbeddingProvider`. **`~6 ms p95`** in-process CPU.

**Rejected: a separate, larger cache-specific embedder** (e.g. bge-large/1024-dim). It would marginally sharpen semantic discrimination but (a) breaks embedding reuse with the retrieval path — we'd pay the embed cost twice, and (b) doubles the vector storage and ANN cost of the cache index. A single 384-dim embedder shared across cache + memory is the right call; if the cache-correctness harness shows τ cannot be set without unacceptable false-hits at 384-dim, the `EmbeddingProvider` seam lets us swap **the cache embedder only** without touching retrieval — but that is a harness-gated decision, not a default.

### Similarity threshold τ — a per-tenant, calibrated, conservative-by-default knob

Semantic hits require **cosine similarity `>= τ`** against the nearest cached entry sharing the `coarse_sig` pre-filter. τ trades hit-ratio against false-hit risk:

- **τ too low** → more hits, more wrong answers (semantically-different queries collapse).
- **τ too high** → fewer hits, cache underperforms its 15-30% savings mandate.

We ship **`τ_default = 0.92`** (cosine, on L2-normalized BGE-small embeddings) as a **conservative** default — biased toward correctness because a wrong cache hit is worse than a miss. τ is a **per-tenant, per-task-class** value, not a global constant, because tenants have different paraphrase distributions and different correctness tolerances.

```
# Calibration job (offline, scheduled; feeds the eval harness, C6)
# Owner: Platform/Caching team. Cadence: weekly + on-demand after prompt/model changes.

for each (tenant, task_class):
    pairs   = sample labeled (query_a, query_b, same_answer: bool) from replay log
    scores  = cosine(embed(query_a), embed(query_b))
    roc     = sweep τ over [0.80, 0.99], measure (hit_rate, false_accept_rate)
    τ*      = max τ such that false_accept_rate <= tenant.max_false_hit  # default 0.5%
    publish CacheThreshold(tenant, task_class, τ*, measured_hit_rate, false_accept_rate)
```

**Rejected: a single global τ.** It either over-caches a strict tenant (correctness incident) or under-caches a lax one (lost savings). **Rejected: per-request adaptive τ via online learning.** Non-deterministic τ makes the Context Replay Debugger's cache decisions irreproducible — a flagship-feature violation. τ must be a **versioned, published artifact** so replay can pin the exact τ used at decision time.

## 2.4.5 What is cacheable — and the non-cacheable rule

**C6, restated as a hard invariant:** **memory-private-grounded responses are NON-cacheable.** A response is eligible for write-back to the cache **only if** every input it was grounded on is stable and shareable within the cache's keyed scope.

```
def is_cacheable(decision: AssemblyDecision) -> bool:
    if decision.used_private_memory:        # any long-term/episodic/semantic per-tenant private fact
        return False                        # ← the rule. Non-negotiable.
    if decision.nondeterministic_tools:     # live tool calls, fresh retrieval of volatile data
        return False
    if decision.stable_fact_set_version is None:
        return False                        # cannot key it → cannot cache it
    if decision.pii_in_output:              # redaction-flagged output never enters a reusable cache
        return False
    return True
```

This is enforced **after** assembly (which is the authority on what grounded the answer) and is recorded in the `ReplayResult` so the debugger can show *why* a response was or wasn't cached. The flag is set by the Assembler/ACL layers, never inferred by the cache itself — the cache trusts the `cacheable` bit and refuses to write anything without it.

> **Pipeline ordering note:** the cache *lookup* runs early (position 2). The cache *write-back* runs last, off the hot path, in the **async write-back** stage, and only after `is_cacheable()` passes. Compression having run *after* ACL/redaction means any cached body is already redaction-clean.

## 2.4.6 `CacheEntry` schema and `lookup()`

```python
from __future__ import annotations
from dataclasses import dataclass
from typing import Literal
# IDs = ULID; timestamps = RFC-3339 UTC; tenant_id = non-null partition key (canonical facts)

@dataclass(frozen=True)
class CacheEntry:
    entry_id: str                  # ULID
    tenant_id: str                 # NON-NULL partition key; RLS + Redis keyspace namespace
    namespace: str                 # within-tenant project/agent/user scope (C2), part of isolation
    coarse_sig: bytes              # SHA-256 of the C6 fingerprint (exact-tier key + ANN pre-filter)
    query_embedding: list[float]   # 384-dim, L2-normalized BGE-small (semantic tier)
    model_id: str                  # fingerprint component (denormalized for audit)
    system_prompt_version: str     # fingerprint component
    stable_fact_set_version: str   # fingerprint component
    response_body: bytes           # materialized final response (streaming-replayable, C15)
    response_tokens: int           # for savings accounting
    finish_reason: str             # server terminal reason at time of caching (C8 — committed only)
    created_at: str                # RFC-3339 UTC
    expires_at: str                # RFC-3339 UTC (TTL; see §2.4.7)
    hit_count: int                 # popularity, for eviction + observability
    is_negative: bool              # negative-cache sentinel (§2.4.8)

@dataclass(frozen=True)
class Hit:
    entry: CacheEntry
    tier: Literal["exact", "semantic"]
    similarity: float | None       # None for exact tier; cosine for semantic tier
    coarse_sig: bytes              # echoed for replay determinism

@dataclass(frozen=True)
class Miss:
    coarse_sig: bytes
    query_embedding: list[float] | None   # reused downstream if computed during semantic probe
    reason: Literal["cold", "below_threshold", "namespace_denied", "outage_fail_open"]
```

```python
async def lookup(query: Query, tenant: TenantCtx) -> Hit | Miss:
    # ── Namespace is a HARD filter, evaluated at the repository boundary (C2) ──
    ns = resolve_namespace(query, tenant)          # missing/ambiguous → raise → deny (fail-closed)
    if ns is None:
        return Miss(coarse_sig=b"", query_embedding=None, reason="namespace_denied")

    # ── Tier 0: exact-hash (Redis), < 1 ms p99 ──
    # The embedding bucket needs the embedding; for the exact tier we use a normalization-stable
    # surface hash fast-path (no float embed) keyed under {tenant}:{ns}. Identical-after-normalize
    # queries resolve here with no embedding cost.
    try:
        sig0 = coarse_exact_sig(query, tenant, ns)         # surface-normalized, embedding-free
        e = await redis.get_entry(tenant.id, ns, sig0)     # keyspace = cache:{tenant}:{ns}:{sig0}
        if e and not _expired(e):
            if e.is_negative:
                return Hit(e, tier="exact", similarity=None, coarse_sig=sig0)  # negative hit
            return Hit(e, tier="exact", similarity=None, coarse_sig=sig0)
    except RedisUnavailable:
        return Miss(coarse_sig=b"", query_embedding=None, reason="outage_fail_open")  # FAIL-OPEN

    # ── Tier 1: semantic-ANN (pgvector), 8-15 ms p95 (embed 6ms + ANN ~4ms + verify) ──
    emb = await embed_query(query)                  # ~6 ms p95; REUSED by retrieval on a miss (§2.4.6)
    sig1 = coarse_semantic_sig(emb, query, tenant, ns)
    τ = await thresholds.get(tenant.id, query.task_class)   # published calibrated value (§2.4.4)
    try:
        # ANN pre-filtered by (tenant_id via RLS) AND namespace AND model/prompt/fact versions.
        cand = await pgvector.nearest(
            tenant_id=tenant.id, namespace=ns, embedding=emb,
            model_id=query.model_id, sys_v=query.system_prompt_version,
            fact_v=query.stable_fact_set_version, k=1)
        if cand and cand.similarity >= τ and not _expired(cand.entry):
            return Hit(cand.entry, tier="semantic", similarity=cand.similarity, coarse_sig=sig1)
        return Miss(coarse_sig=sig1, query_embedding=emb, reason="below_threshold")
    except PgVectorUnavailable:
        return Miss(coarse_sig=sig1, query_embedding=emb, reason="outage_fail_open")  # FAIL-OPEN
```

**Embedding reuse (the no-double-embed guarantee):** on a semantic miss, `Miss.query_embedding` carries the `~6 ms` embedding forward; the Memory Engine (§2.2) consumes it for `pgvector ANN || BM25 + RRF` rather than re-embedding. This is why the §9 retrieval line says "embed reused" — the cache pays the embed, retrieval inherits it. The net additional cost the cache imposes on a *miss* is therefore just the **ANN probe (~4 ms)** plus exact-tier `< 1 ms`, consistent with the `10 ms` cache line.

## 2.4.7 TTL, invalidation, and staleness

A cache that never invalidates is a correctness time-bomb. We use **TTL + version-keyed mass invalidation + explicit triggers**, layered:

| Mechanism | Granularity | Trigger | Effect |
|---|---|---|---|
| **TTL** | per-entry | `expires_at` (default **24 h** for stable-fact responses; **15 min** for negative entries) | Lazy expiry on read + active sweep; bounds worst-case staleness window |
| **Version-keyed invalidation** | en masse | `stable_fact_set_version` or `system_prompt_version` or `model_id` bumps | New version ⇒ new `coarse_sig` space ⇒ **all** prior entries become unreachable (no scan needed) |
| **Explicit purge** | namespace/tenant | RTBF (C11), admin action, incident | Tombstone + idempotent GC sweep; embeddings crypto-shredded |
| **Eviction** | per-tenant budget | memory/row-count pressure | LRU-by-`hit_count`+`created_at`; per-tenant quota so one tenant can't evict another's hot set |

**Version-keyed invalidation is the headline mechanism.** Because the fingerprint *contains* `stable_fact_set_version` / `system_prompt_version` / `model_id`, bumping any of them **instantly orphans** the old cache — no expensive scan-and-delete, no race window where a stale answer can be served under the new version. Old entries are then reaped lazily by TTL/eviction. This is far superior to:

- **Rejected: TTL-only.** Forces a staleness/cost tradeoff (short TTL = low hit-ratio; long TTL = stale answers after a fact update). Version-keying decouples the two.
- **Rejected: pub/sub active invalidation lists.** Maintaining and replaying per-fact invalidation messages across replicas is a consistency liability; a fact-set version bump is one atomic, replay-friendly fact.

**Staleness bound:** worst case = TTL window *for a given version*. A `stable_fact_set_version` bump on a fact update reduces the *effective* staleness to "until the next request re-populates," because old-version entries are immediately unreachable. We record `created_at` + version in every `CacheEntry` so the **replay debugger** can prove which version a served answer was grounded on (C7 — cache decisions are deterministic stages).

## 2.4.8 Negative caching

We cache **negatives** — a request that produced a deterministic non-answer (policy refusal under a stable policy version, validated empty-result for a stable query, hard 4xx-class deterministic rejection). `CacheEntry.is_negative=True`, short TTL (**15 min** default), keyed by the *same* `coarse_sig`.

**Why:** retry storms and polling agents re-issue the same refused/empty request; without negative caching each one pays the full pipeline + model call to re-derive the same "no." Negative caching converts those into `< 1 ms` exact-tier hits.

**Guardrails (so negatives don't become correctness bugs):**
- Only **deterministic** negatives are cached — never a transient backend error (those are excluded by `finish_reason` not being a committed terminal, per C8; a client-abort or 5xx never writes back).
- Short TTL so a fixed-upstream / policy change recovers fast.
- Negative entries are still namespace- and version-keyed, so a `system_prompt_version` bump (which may change a refusal) orphans them with everything else.

**Rejected: no negative caching.** Leaves the cheapest, highest-frequency repeat traffic (refusals/empties) paying full freight, eroding the 15-30% savings target precisely on the easy cases routing would otherwise downgrade.

## 2.4.9 Per-tenant namespacing — zero cross-tenant hits (C2)

The cache enforces the **same** isolation guarantee as the rest of ContextOS: **cross-tenant leakage = 0**. There is no "shared cache" at launch.

- **Redis exact tier:** every key is namespaced `cache:{tenant_id}:{namespace}:{coarse_sig}`. A lookup for tenant A *cannot construct* a key in tenant B's space; the `tenant_id` is bound from the authenticated `TenantCtx`, never from request content.
- **pgvector semantic tier:** every `CacheEntry` row carries non-null `tenant_id` and is protected by Postgres **`FORCE ROW LEVEL SECURITY`** with `SET LOCAL` per request. An ANN query physically cannot return another tenant's row — the same RLS firewall that guarantees the system-wide **0 cross-tenant leakage** invariant, exercised by the **>= 10,000 hostile second-tenant property probes** CI hard gate. The cache adds no new isolation surface; it reuses the proven one.
- **Within-tenant namespace (C2):** `project/agent/user` namespace is a **hard, fail-closed filter** evaluated at the repository boundary alongside `tenant_id`. **Missing or ambiguous namespace = deny** (`Miss(reason="namespace_denied")`), never a wildcard match. An opt-in **shared-org namespace** is gated by an `RBACPolicy` rule (`action='cache_read'`, C10) — but that is *within* a tenant's org, still inside the tenant boundary.

### Why secure cross-tenant sharing is deferred

A cross-tenant *shared* cache (tenant B benefiting from tenant A's cached answer for a public-knowledge query) is tantalizing for hit-ratio but is **explicitly deferred**, for concrete reasons:

1. **Leakage blast radius.** A single mis-scoped shared entry is a cross-tenant data incident — the exact thing the `0 leakage` invariant forbids. The COARSE fingerprint already excludes private memory, but *proving* a given cached answer contains no tenant-specific signal requires a verified content classifier we do not yet have.
2. **It depends on the correctness harness (C6).** We cannot certify shared entries as safe-to-share until the offline cache-correctness eval harness can measure leakage-via-cache empirically. Sharing is gated on that harness landing.
3. **The savings are already met without it.** Per-tenant caching delivers the **25-45% hit-ratio** and **15-30% token savings** targets. Cross-tenant sharing is upside, not a requirement — so it does not justify weakening the launch isolation guarantee.

Cross-tenant sharing therefore enters the roadmap behind: (a) the cache-correctness harness, (b) a verified no-tenant-signal content classifier, and (c) an explicit per-tenant opt-in with a residency/RBAC contract. Until all three exist, **every hit is within a single tenant.**

## 2.4.10 Fail-open posture (C12)

The cache is a **best-effort accelerator**, never a dependency on the critical path's *correctness*. Per **C12**, on any cache-tier outage (Redis down, pgvector probe timeout, embedding provider unavailable for the cache path):

```
cache outage  →  treat as Miss(reason="outage_fail_open")  →  proceed through full pipeline
```

- A Redis or pgvector failure **never** blocks a request; it costs a cache miss (more tokens, more latency) but the answer is still produced correctly downstream.
- Cache **lookups and traces are fail-open + sampled** (C12 best-effort), so a telemetry hiccup never wedges a request.
- The **write-back** is also best-effort: if the post-response cache write fails, the response was already streamed to the client (terminal-event committed, C8); we drop the cache write and log it. We never delay or fail a user response to persist a cache entry.
- **The one thing that is NOT fail-open: namespace/tenant scoping.** Isolation is fail-**closed**. A namespace that cannot be resolved denies the lookup (returns a miss into the pipeline) rather than risking a cross-namespace hit. Correctness and isolation are fail-closed; *performance* is fail-open. That asymmetry is deliberate and matches C9's posture for the router (hard filters fail-closed, optimization fails-open).

## 2.4.11 Observability & replay (C7)

Cache decisions are **deterministic ContextOS stages** and therefore fully captured in the single `ReplayResult` schema (C7). For every request the cache emits, into the replay bundle:

| Field | Purpose |
|---|---|
| `coarse_sig` | the exact fingerprint computed (reproducible) |
| `tier` / `similarity` / `τ` | which tier answered, the cosine score, the published threshold in force |
| `cacheable` + reason | why the *response* was/wasn't written back (private-memory flag, PII, version) |
| `namespace_decision` | resolved namespace or `namespace_denied` |
| `versions` | `model_id`, `system_prompt_version`, `stable_fact_set_version` at decision time |

Because τ is a **published, versioned artifact** (§2.4.4) and the fingerprint inputs are all recorded, the **Context Replay Debugger can byte-exactly reproduce the cache decision** — "this request hit the semantic tier at cosine 0.94 against entry `01J...` under τ=0.92, fact-set v17" — which is the whole point of the flagship feature. `backend.invoke` remains non-deterministic (C7), but the cache hit/miss decision is not.

**Metrics surfaced** (per tenant, per namespace): `hit_ratio` (target **25-45%**, alarms if it drifts below 20% — signals τ too high or fingerprint churn — or above 50% — signals possible over-caching / false hits), `exact_vs_semantic_hit_split`, `false_hit_rate` (from the eval harness), `tokens_saved` (toward the 15-30% mandate), `p99 exact / p95 semantic latency`, and `fail_open_rate`.

## 2.4.12 Summary of opinionated choices

| Decision | Chosen | Rejected | Why |
|---|---|---|---|
| Tiering | Exact (Redis) + Semantic (pgvector) | Semantic-only; exact-only; Redis-VSS for both | Two tiers hit 25-45%; reuse pgvector RLS, no third vector engine |
| Fingerprint | COARSE (C6) | FINE (full-prompt hash) | FINE → ~0% hits; COARSE keys on stable intent, excludes private memory |
| Embedder | shared BGE-small 384-dim | separate large cache embedder | embedding reuse with retrieval; no double-embed |
| Threshold τ | per-tenant calibrated, conservative default 0.92, versioned | global τ; online-adaptive τ | correctness-first + replay-deterministic |
| Invalidation | version-keyed + TTL + tombstone GC | TTL-only; pub/sub lists | instant mass-orphan on fact/prompt/model bump, replay-friendly |
| Private memory | non-cacheable, hard rule | cache everything | the core correctness guarantee |
| Cross-tenant | per-tenant only at launch | shared cache now | 0-leakage invariant; sharing gated on correctness harness |
| Outage | fail-open (perf); fail-closed (isolation) | fail-closed everywhere; fail-open everywhere | accelerator must not break correctness or availability |
```
